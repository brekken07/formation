Module 1 — Pourquoi Ansible ?

1.1 Le problème

Sans outil d'automatisation, gérer plusieurs serveurs Linux revient à se connecter manuellement en SSH sur chacun, taper les mêmes commandes, faire des erreurs différentes sur chaque machine, et ne plus savoir exactement ce qui est installé où.

 

Ansible résout ce problème : on décrit l'état désiré des machines dans des fichiers texte, et Ansible applique cet état partout, de façon identique et reproductible.

 

1.2 Le concept clé : l'idempotence

Idempotence : si tu lances le même playbook 10 fois, le résultat est identique à la 1re exécution.

Ansible vérifie l'état actuel avant d'agir. Si nginx est déjà installé, il ne réinstalle pas.

 

changed: 1  →  Ansible a effectué une modification réelle

changed: 0  →  Rien à faire, état déjà conforme

 

1.3 Comment ça fonctionne

Ansible fonctionne en mode push depuis une machine de contrôle. Il se connecte aux machines cibles via SSH sans installer d'agent. Il a uniquement besoin de Python 3 sur les cibles.

 

Machine de contrôle (10.116.0.195)

├── Inventory  →  liste des machines cibles

├── Playbook   →  ce qu'on veut faire

└── Ansible    →  se connecte via SSH

                       │

          ┌────────────┼────────────┐

          ▼            ▼            ▼

     machine2      machine3      ...

  (10.116.0.238) (10.116.0.239)

Question : Qu'est-ce que l'idempotence en Ansible ?

Réponse : Si on lance le même playbook plusieurs fois, le résultat est identique à la 1re exécution. Ansible vérifie l'état actuel avant d'agir et ne modifie que ce qui est nécessaire.

 

Question : Ansible a-t-il besoin d'un agent sur les machines cibles ?

Réponse : Non. Ansible utilise SSH + Python 3. Rien d'autre à installer sur les machines cibles.


Module 2 — Installation et premier contact

2.1 Installation sur la machine de contrôle

# Installation via pip (recommandée)

$ pip3 install ansible

 

# Ou via le dépôt Debian

$ sudo apt install ansible -y

 

# Vérification

$ ansible --version

 

2.2 Configuration de l'accès SSH par clé

Deux mécanismes distincts à bien comprendre :

  • La clé SSH      →  sert à SE CONNECTER à la machine

  • Le mot de passe sudo  →  sert à ÉLEVER LES PRIVILÈGES une fois connecté

Ce sont deux choses complètement différentes.

 

# 1. Générer une paire de clés ED25519 sur la machine de contrôle

$ ssh-keygen -t ed25519 -C "ansible" -f ~/.ssh/ansible_key

 

# 2. Copier la clé publique sur chaque machine cible

$ ssh-copy-id -i ~/.ssh/ansible_key.pub setupuser@10.116.0.238

$ ssh-copy-id -i ~/.ssh/ansible_key.pub setupuser@10.116.0.239

 

# 3. Tester la connexion sans mot de passe

$ ssh -i ~/.ssh/ansible_key setupuser@10.116.0.238

Résultat attendu : connexion directe sans saisie de mot de passe.


Module 3 — L'Inventory

3.1 Structure du projet

ansible-tp/                    ← Racine du projet

├── inventory/

│   ├── hosts.ini              ← Liste des machines et groupes

│   └── group_vars/

│       └── all.yml            ← Variables communes à TOUTES les machines

├── roles/

├── templates/

└── site.yml                   ← Playbook principal

 

3.2 Le fichier hosts.ini

[machines_cibles]

machine2   ansible_host=10.116.0.238

machine3   ansible_host=10.116.0.239

 

[machines_cibles:vars]

ansible_user=setupuser

ansible_ssh_private_key_file=~/.ssh/ansible_key

ansible_python_interpreter=/usr/bin/python3

 

3.3 Ce qu'on invente vs ce qu'Ansible impose

Élément

Qui décide ?

Exemple

Toi — libre

[machines_cibles]

Toi — libre

machine2

Valeurs des variables

Toi

setupuser, ~/.ssh/...

Noms ansible_XXX

Ansible — imposés

ansible_host, ansible_user

Ansible — imposée

[machines_cibles:vars]

 

3.4 Le dossier group_vars

Règle clé : le nom du fichier dans group_vars doit correspondre EXACTEMENT au nom du groupe.

 

[machines_cibles]  →  group_vars/machines_cibles.yml

[webservers]       →  group_vars/webservers.yml

[all]              →  group_vars/all.yml  (chargé pour TOUTES les machines)

 

Ansible charge automatiquement les bons fichiers — tu ne précises jamais lequel utiliser.

Une erreur de nommage est ignorée silencieusement, sans message d'erreur.

 

Question : Dans hosts.ini, 'machine2' est-il le hostname réel de la VM ?

Réponse : Non. C'est un alias libre. Le hostname réel est défini par ansible_host=10.116.0.238. L'alias sert uniquement à l'affichage dans les sorties Ansible.

 

Question : Si j'ai [webservers] et un fichier group_vars/serveurs_web.yml, Ansible fait-il le lien ?

Réponse : Non. Le nom du fichier doit être exactement identique au nom du groupe. group_vars/webservers.yml est le fichier correct.

Module 4 — Le Playbook

4.1 Règles YAML

• Indentation OBLIGATOIRE avec des espaces, jamais des tabulations.

• Convention : 2 espaces par niveau d'indentation.

• Le tiret - signifie 'élément d'une liste'.

• Toute valeur commençant par {{ doit être entre guillemets : "{{ variable }}".

• Le fichier commence toujours par ---

 

4.2 Anatomie complète

---                              # Marqueur de début YAML

 

- name: Nom du play             # Nom libre

  hosts: machines_cibles        # Groupe cible de l'inventory

  become: true                  # Utiliser sudo

 

  tasks:

 

    - name: Installer nginx

      ansible.builtin.apt:

        name: nginx

        state: present

 

  handlers:

    - name: Restart nginx

      ansible.builtin.service:

        name: nginx

        state: restarted

 

4.3 Commandes d'exécution

# Simulation sans modification — TOUJOURS avant un déploiement

$ ansible-playbook -i inventory/hosts.ini site.yml --check

 

# Exécution réelle

$ ansible-playbook -i inventory/hosts.ini site.yml

 

# Avec mot de passe sudo interactif

$ ansible-playbook -i inventory/hosts.ini site.yml --ask-become-pass

 

# Mode verbeux

$ ansible-playbook -i inventory/hosts.ini site.yml -v

 

# Limiter à une seule machine

$ ansible-playbook -i inventory/hosts.ini site.yml --limit machine2

 

4.4 Lire la sortie

Statut

Signification

ok

Tâche exécutée, rien modifié (idempotence)

changed

Ansible a effectué une modification réelle

skipped

Tâche ignorée (condition when non remplie)

unreachable

Machine non joignable

failed

La tâche a échoué

 

⚠  Le mode --check peut signaler des erreurs sur les tâches qui dépendent d'une tâche non encore exécutée (ex : démarrer un service non encore installé). Ce n'est pas forcément une erreur dans le playbook.

2. Lancer pour de vrai avec --ask-become-pass

Module 5 — Modules essentiels et boucles

5.1 Les modules les plus utilisés

ansible.builtin.apt — paquets Debian/Ubuntu

- name: Installer nginx

  ansible.builtin.apt:

    name: nginx

    state: present        # present / absent / latest

    update_cache: true

 

ansible.builtin.service — services système

- name: Démarrer nginx

  ansible.builtin.service:

    name: nginx

    state: started        # started / stopped / restarted / reloaded

    enabled: true         # démarrage automatique au boot

 

ansible.builtin.user — utilisateurs

- name: Créer un utilisateur

  ansible.builtin.user:

    name: deployer

    shell: /bin/bash

    create_home: true

    groups: sudo

    state: present

 

ansible.builtin.template — fichiers Jinja2

- name: Déployer la config nginx

  ansible.builtin.template:

    src: templates/nginx.conf.j2

    dest: /etc/nginx/nginx.conf

    mode: '0644'

  notify: Restart nginx

 

ansible.builtin.copy — fichiers statiques

- name: Copier un fichier

  ansible.builtin.copy:

    src: files/app.conf

    dest: /etc/app/app.conf

    owner: root

    mode: '0644'

 

ansible.builtin.file — fichiers, dossiers, liens

- name: Créer un répertoire

  ansible.builtin.file:

    path: /opt/myapp

    state: directory      # directory / file / absent / link

    owner: deployer

    mode: '0755'

 

ansible.builtin.lineinfile — modifier une ligne

- name: Désactiver PermitRootLogin

  ansible.builtin.lineinfile:

    path: /etc/ssh/sshd_config

    regexp: '^PermitRootLogin'

    line: 'PermitRootLogin no'

  notify: Restart sshd

 

ansible.builtin.debug — afficher des informations

- name: Afficher l'IP

  ansible.builtin.debug:

    msg: "IP : {{ ansible_default_ipv4.address }}"

 

5.2 Les boucles avec loop

# Boucle avec une liste directe

- name: Installer les outils

  ansible.builtin.apt:

    name: "{{ item }}"      # guillemets obligatoires

    state: present

  loop:

    - vim

    - curl

    - git

 

# Boucle avec une variable depuis group_vars/all.yml

- name: Installer les outils

  ansible.builtin.apt:

    name: "{{ item }}"

    state: present

  loop: "{{ outils }}"

 

⚠  Ne jamais écrire name: {{ item }} sans guillemets. En YAML, une valeur qui commence par {{ est ambiguë. Toujours écrire name: "{{ item }}".

Module 7 — Les Handlers

7.1 Pourquoi les handlers ?

Sans handler, on serait tenté d'ajouter une tâche 'restart nginx' après chaque modification de config. Le problème : cette tâche redémarrerait nginx à chaque exécution du playbook, même si rien n'a changé — provoquant des interruptions de service inutiles.

 

Un handler = une tâche spéciale qui ne se déclenche que si :

  1. Une tâche l'a notifié avec notify:

  2. ET cette tâche a produit un changed (pas un ok)

 

Il s'exécute UNE SEULE FOIS en fin de play, dans la section RUNNING HANDLERS,

même si plusieurs tâches l'ont notifié.

 

7.2 Syntaxe

  tasks:

 

    - name: Déployer la config nginx

      ansible.builtin.template:

        src: templates/nginx.conf.j2

        dest: /etc/nginx/nginx.conf

      notify: Redémarrer nginx      # ← même nom EXACT que le handler

 

  handlers:

 

    - name: Redémarrer nginx        # ← doit correspondre exactement au notify

      ansible.builtin.service:

        name: nginx

        state: restarted

 

7.3 Ce qu'on voit dans la sortie

TASK [Déployer la config nginx] ****

changed: [machine2]                  ← changed → handler sera déclenché

 

RUNNING HANDLERS *******************

changed: [machine2]                  ← handler exécuté

 

PLAY RECAP *************************

 

⚠  Le lien notify / handler se fait par correspondance STRICTE du nom. Un seul caractère différent (majuscule, accent, espace) et le handler est silencieusement ignoré, sans message d'erreur.

 

Question : Quand le handler ne se déclenche-t-il pas ?

Réponse : Quand la tâche qui le notifie retourne ok (pas changed). Si la config nginx n'a pas été modifiée, la tâche retourne ok et le handler n'est pas déclenché — nginx n'est pas redémarré inutilement.

 

Question : Ansible parcourt-il automatiquement toute l'arborescence du projet ?

Réponse : Non. Ansible suit des chemins conventionnels précis. Il charge group_vars/ automatiquement, et pour les roles il cherche tasks/main.yml, handlers/main.yml, templates/ à des emplacements fixes. Un fichier mal nommé ou mal placé est ignoré silencieusement.

Module 8 — Les Roles

8.1 Pourquoi les roles ?

Un playbook qui grossit devient vite illisible et impossible à réutiliser. Les roles découpent le code en briques indépendantes, chacune responsable d'une seule fonctionnalité. Un role peut être réutilisé sur n'importe quel projet.

 

Principe de responsabilité unique :

  • Role common  →  outils de base, utilisateurs

  • Role nginx   →  installation et configuration de nginx

  • Role web     →  déploiement de l'application web

 

Le site.yml principal devient un simple chef d'orchestre qui appelle les roles.

 

8.2 Structure d'un role

roles/

└── nginx/                    # Nom du role

    ├── tasks/

    │   └── main.yml          # Point d'entrée des tâches (obligatoire)

    ├── handlers/

    │   └── main.yml          # Handlers du role

    ├── templates/

    │   └── *.j2              # Templates Jinja2

    ├── files/                # Fichiers statiques

    ├── vars/

    │   └── main.yml          # Variables (priorité haute)

    └── defaults/

        └── main.yml          # Variables par défaut (priorité basse)

 

8.3 Contenu des fichiers main.yml d'un role

Un main.yml de role ne contient QUE la liste des tâches — sans l'entête du play (name, hosts, become, tasks:). On commence directement par le premier tiret.

 

# roles/common/tasks/main.yml

- name: Installer les outils de base

  ansible.builtin.apt:

    name: "{{ item }}"

    state: present

  loop: "{{ outils }}"

 

- name: Créer les utilisateurs

  ansible.builtin.user:

    name: "{{ item }}"

    shell: /bin/bash

    create_home: true

    state: present

  loop: "{{ stagiaires }}"

 

8.4 Templates dans un role

Dans un role, le chemin vers le template est relatif au dossier templates/ du role. On n'écrit pas templates/ dans le src — juste le nom du fichier.

 

# roles/web/tasks/main.yml

- name: Déployer la page web

  ansible.builtin.template:

    src: index.html.j2        # Relatif à roles/web/templates/

    dest: /var/www/html/index.html

  notify: Redémarrer nginx

 

8.5 Le site.yml avec des roles

---

 

- name: mon premier playbook

  hosts: machines_cibles

  become: true

 

  roles:

    - role: common

    - role: nginx

    - role: web

 

⚠  Ansible ne parcourt pas l'arborescence librement. Il suit des chemins conventionnels précis. Le fichier main.yml doit s'appeler exactement main.yml et se trouver dans le bon dossier.

 

Question : Dans un role, doit-on préciser templates/ dans le chemin src du module template ?

Réponse : Non. Dans un role, Ansible sait déjà que les templates se trouvent dans roles/nom_role/templates/. On écrit uniquement le nom du fichier : src: index.html.j2
