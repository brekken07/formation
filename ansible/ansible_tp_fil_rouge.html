Les fondations

Avant d'automatiser quoi que ce soit, il faut savoir ce qu'on gère et vérifier qu'Ansible peut joindre les machines. Un mauvais inventory, c'est un playbook qui ne s'exécute jamais sur les bonnes cibles.

1.1 Créer la structure du projet
🎯 Contexte

Ansible cherche ses fichiers dans des emplacements conventionnels. La structure que tu crées maintenant servira de base pour toute la journée.

📋 Ta mission

Crée l'arborescence suivante dans ton répertoire home :

devcorp-ansible/
├── inventory/
│   └── group_vars/
├── roles/
└── templates/

Vérifie avec : find ~/devcorp-ansible -type d

La commande mkdir avec l'option -p crée tous les dossiers intermédiaires en une seule fois.

Tu peux chaîner plusieurs mkdir -p sur une même ligne avec &&.

1.2 Créer l'inventory
🎯 Contexte

L'inventory est le cœur d'Ansible. Il dit QUI sont tes machines et COMMENT les joindre. On organise les machines en groupes — ici toutes les machines de formation appartiennent au groupe [formation].

📋 Ta mission

Crée inventory/hosts.ini avec :

  • Le groupe [formation] avec machine2 (10.116.0.238) et machine3 (10.116.0.239)
  • Le bloc [formation:vars] avec les 3 directives de connexion indispensables
  • Utilisateur : setupuser — Clé : ~/.ssh/ansible_key

Vérifie avec : ansible all -i inventory/hosts.ini --list-hosts

Les noms de groupes et les alias de machines sont libres — tu les inventes.

Les directives ansible_XXX sont des mots-clés imposés par Ansible — tu ne peux pas les renommer.

Pour la clé SSH, la directive s'appelle ansible_ssh_private_key_file.

Pour Python : ansible_python_interpreter.

[formation]
machine2   ansible_host=10.116.0.238
machine3   ansible_host=10.116.0.239

[formation:vars]
ansible_user=setupuser
ansible_ssh_private_key_file=~/.ssh/ansible_key
ansible_python_interpreter=/usr/bin/python3
1.3 Créer le fichier de variables globales
🎯 Contexte

Les variables externalisées dans group_vars/ permettent de changer les données sans toucher aux playbooks. Le fichier all.yml s'applique automatiquement à toutes les machines. Le nom du fichier doit correspondre exactement au nom du groupe dans l'inventory.

📋 Ta mission

Crée inventory/group_vars/all.yml avec :

  • formation_name : le nom de ta formation
  • promotion : le nom de la promo (ex: Promo Avril 2025)
  • stagiaires : liste de 3 comptes (stagiaire01, 02, 03)
  • outils_base : liste de paquets (vim, curl, git, htop, tree, jq)
  • partage_dir : chemin du répertoire partagé (/opt/formation)

Un fichier group_vars commence par ---.

Variable simple : ma_variable: "ma valeur"

Liste :

ma_liste:
  - element1
  - element2
---

formation_name: "Formation Ansible DevCorp"
promotion: "Promo Avril 2025"

stagiaires:
  - stagiaire01
  - stagiaire02
  - stagiaire03

outils_base:
  - vim
  - curl
  - git
  - htop
  - tree
  - jq

partage_dir: /opt/formation
1.4 Tester la connectivité et découvrir les commandes ad-hoc
🎯 Contexte

Les commandes ad-hoc exécutent un module directement en ligne de commande sans créer de fichier playbook. Utile pour tester et déboguer rapidement.

📋 Ta mission

Exécute ces commandes dans l'ordre et observe les résultats :

# Tester la connectivité SSH + Python
ansible all -i inventory/hosts.ini -m ping

# Vérifier l'uptime de toutes les machines
ansible all -i inventory/hosts.ini -m command -a "uptime"

# Afficher l'espace disque
ansible all -i inventory/hosts.ini -m command -a "df -h /"

# Voir les facts de machine2 (cherche ansible_default_ipv4)
ansible machine2 -i inventory/hosts.ini -m setup | grep -A5 ipv4
🔍 Valider
  • Le ping retourne SUCCESS / pong sur les deux machines
  • L'uptime affiche la durée de fonctionnement de chaque machine
  • Tu vois l'adresse IP de machine2 dans la sortie setup

Préparer les machines

Les machines sont joignables. Il faut maintenant les configurer : outils, comptes stagiaires, et accès sudo automatisé pour les playbooks suivants.

2.1 Créer le playbook principal site.yml
🎯 Contexte

Le site.yml est le point d'entrée de tout le projet. Pour l'instant il contiendra directement les tâches — on le refactorisera en roles à l'Acte 5.

📋 Ta mission

Crée ~/devcorp-ansible/site.yml avec un seul play :

  • Nom du play libre, hôtes : groupe formation, become: true
  • Une tâche debug qui affiche : Démarrage de la configuration de {{ ansible_hostname }}

Lance : ansible-playbook -i inventory/hosts.ini site.yml --ask-become-pass

---
- name: Nom du play
  hosts: mon_groupe
  become: true
  tasks:
    - name: Ma tâche
      ansible.builtin.debug:
        msg: "{{ ma_variable }}"
---

- name: Préparation de l'environnement DevCorp
  hosts: formation
  become: true

  tasks:

    - name: Annonce de démarrage
      ansible.builtin.debug:
        msg: "Démarrage de la configuration de {{ ansible_hostname }}"
2.2 Installer les outils de base
🎯 Contexte

Grâce à loop, une seule tâche suffit pour toute la liste. Si on ajoute un outil dans all.yml, il sera automatiquement installé au prochain lancement.

📋 Ta mission

Ajoute une tâche dans site.yml qui installe tous les outils de la variable outils_base.

  • Utiliser une boucle loop avec la variable outils_base
  • Module apt avec update_cache: true et state: present

Lance avec --check d'abord, puis sans.

Syntaxe de la boucle avec une variable : loop: "{{ ma_variable }}"

Dans le module, le paquet courant s'écrit : name: "{{ item }}"

N'oublie pas les guillemets autour de {{ }} quand la valeur commence par ces accolades.

    - name: Installation des outils de base
      ansible.builtin.apt:
        name: "{{ item }}"
        state: present
        update_cache: true
      loop: "{{ outils_base }}"
🔍 Valider

Relance le playbook une deuxième fois. Tous les paquets déjà installés affichent ok (changed: 0). C'est l'idempotence en action.

2.3 Créer les comptes stagiaires
🎯 Contexte

Chaque stagiaire a besoin d'un compte Linux avec home directory et accès sudo pour ses propres exercices.

📋 Ta mission

Ajoute une tâche qui crée les comptes de la variable stagiaires avec :

  • Shell : /bin/bash
  • Home créé automatiquement
  • Groupe secondaire : sudo
  • State : present

Vérifie : ansible machine2 -i inventory/hosts.ini -m command -a "cat /etc/passwd" | grep stagiaire

Le module à utiliser : ansible.builtin.user

Pour les groupes secondaires, le paramètre s'appelle groups (pas group).

Pour créer le home : create_home: true

    - name: Création des comptes stagiaires
      ansible.builtin.user:
        name: "{{ item }}"
        shell: /bin/bash
        create_home: true
        groups: sudo
        state: present
      loop: "{{ stagiaires }}"
2.4 Configurer sudo sans mot de passe pour Ansible
🎯 Contexte

En production, le compte Ansible doit pouvoir faire sudo sans mot de passe pour que les playbooks soient entièrement automatisables. On configure ça proprement via /etc/sudoers.d/ — plus sûr que modifier /etc/sudoers directement.

📋 Ta mission

Ajoute une tâche qui crée le fichier /etc/sudoers.d/setupuser avec le contenu :

setupuser ALL=(ALL) NOPASSWD: ALL
  • Utilise ansible.builtin.copy avec le paramètre content:
  • Mode du fichier : 0440 (lecture seule, root uniquement)
  • Propriétaire et groupe : root

Après ce lancement, tu pourras supprimer --ask-become-pass de toutes tes commandes.

ansible.builtin.copy avec content: au lieu de src: écrit du texte directement :

ansible.builtin.copy:
  content: "ma ligne de texte\n"
  dest: /chemin/du/fichier
  mode: '0440'
  owner: root
  group: root

Le \n à la fin est important — un fichier sudoers doit terminer par une newline.

Bonus : ajoute validate: /usr/sbin/visudo -cf %s pour valider la syntaxe avant d'écrire.

    - name: Configuration sudo sans mot de passe pour setupuser
      ansible.builtin.copy:
        content: "setupuser ALL=(ALL) NOPASSWD: ALL\n"
        dest: /etc/sudoers.d/setupuser
        owner: root
        group: root
        mode: '0440'
        validate: /usr/sbin/visudo -cf %s
🔍 Valider

Lance le playbook une dernière fois AVEC --ask-become-pass. Ensuite, lance-le sans ce paramètre — il doit fonctionner. Tu n'en auras plus besoin pour la suite.

Informer les stagiaires

Les comptes sont créés, les outils installés. Maintenant chaque stagiaire doit savoir où il est et ce qui est disponible. On déploie une page web d'accueil personnalisée par machine.

3.1 Installer et démarrer nginx
📋 Ta mission

Ajoute deux tâches dans site.yml :

  • Installer le paquet nginx avec apt (state: present)
  • Démarrer le service nginx et l'activer au boot (state: started, enabled: true)

Lance le playbook puis vérifie : http://10.116.0.238 et http://10.116.0.239

Tu dois voir la page par défaut de nginx.

Pour l'installation : ansible.builtin.apt avec update_cache: true

Pour le service : ansible.builtin.service

Les valeurs de state pour un service : started / stopped / restarted / reloaded

3.2 Créer le template de la page d'accueil
🎯 Contexte

Un template Jinja2 est un fichier texte avec des zones dynamiques. Ansible remplace les {{ variable }} et évalue les {% blocs %} avant de déposer le fichier sur la machine. Chaque machine reçoit un fichier différent, personnalisé avec ses propres valeurs.

Facts Ansible disponibles
FactContenu
ansible_hostnameNom d'hôte de la machine
ansible_default_ipv4.addressAdresse IP principale
ansible_os_familyFamille d'OS (Debian, RedHat…)
ansible_distributionDistribution Linux
ansible_distribution_versionVersion de la distribution
ansible_memtotal_mbRAM totale en Mo
ansible_processor_vcpusNombre de vCPU
ansible_date_time.dateDate courante
📋 Ta mission

Crée templates/accueil.html.j2 qui affiche :

  • Le nom de la formation et la promotion
  • Le hostname et l'IP de la machine (facts)
  • La distribution et sa version
  • La RAM disponible
  • La liste des stagiaires (boucle for)
  • La liste des outils installés (boucle for)
  • La date de déploiement

Syntaxe Jinja2 :

{{ ma_variable }}          # afficher une variable
{{ mon_fact.sous_cle }}    # afficher un fact imbriqué
{% for x in ma_liste %}    # boucle
  {{ x }}                  # élément courant
{% endfor %}               # fin de boucle
{% if condition %}         # conditionnel
{% endif %}

⚠ ansible_default_ipv4.address s'écrit avec deux d dans address.

<!DOCTYPE html>
<html lang="fr">
<head>
    <meta charset="UTF-8">
    <title>{{ formation_name }}</title>
    <style>
        body { font-family: Arial, sans-serif; margin: 40px; }
        h1   { color: #1F3864; border-bottom: 2px solid #2E75B6; }
        h2   { color: #2E75B6; }
        .info { background: #D6E4F0; padding: 15px; border-radius: 8px; }
    </style>
</head>
<body>
    <h1>{{ formation_name }}</h1>
    <p><em>{{ promotion }} — Déployé le {{ ansible_date_time.date }}</em></p>

    <div class="info">
        <h2>Informations machine</h2>
        <ul>
            <li>Hostname : {{ ansible_hostname }}</li>
            <li>IP : {{ ansible_default_ipv4.address }}</li>
            <li>OS : {{ ansible_distribution }} {{ ansible_distribution_version }}</li>
            <li>RAM : {{ ansible_memtotal_mb }} Mo</li>
        </ul>
    </div>

    <h2>Stagiaires</h2>
    <ul>
    {% for s in stagiaires %}
        <li>{{ s }}</li>
    {% endfor %}
    </ul>

    <h2>Outils disponibles</h2>
    <ul>
    {% for o in outils_base %}
        <li>{{ o }}</li>
    {% endfor %}
    </ul>
</body>
</html>
3.3 Déployer le template et configurer le handler
🎯 Contexte

Le handler permet de redémarrer nginx uniquement si la page a changé. Sans handler, nginx redémarrerait à chaque exécution — inutile et perturbant pour les utilisateurs connectés.

📋 Ta mission

Dans site.yml :

  • Ajoute une tâche qui déploie accueil.html.j2 vers /var/www/html/index.html avec mode 0644 et un notify
  • Ajoute un bloc handlers: (même niveau que tasks:) avec un handler qui redémarre nginx

Teste en deux temps : avec puis sans modification du template.

Module template : ansible.builtin.template avec src: accueil.html.j2

Le notify et le name du handler doivent être strictement identiques.

Le handler utilise ansible.builtin.service avec state: restarted

Le bloc handlers: se place au même niveau d'indentation que tasks:.

  # Dans tasks:
    - name: Déploiement de la page d'accueil
      ansible.builtin.template:
        src: accueil.html.j2
        dest: /var/www/html/index.html
        mode: '0644'
      notify: Redémarrer nginx

  # Bloc handlers: au même niveau que tasks:
  handlers:
    - name: Redémarrer nginx
      ansible.builtin.service:
        name: nginx
        state: restarted
🔍 Valider
  • http://10.116.0.238 affiche le hostname et l'IP de machine2
  • http://10.116.0.239 affiche les infos de machine3
  • Même structure, contenu différent sur chaque machine

Mettre des ressources à disposition

Les stagiaires ont leurs comptes et leur page d'accueil. Il faut maintenant leur préparer un répertoire partagé avec les ressources du TP : paquets téléchargés, README dynamique.

4.1 Créer le répertoire partagé
🎯 Contexte

Le répertoire /opt/formation sera le point de dépôt commun. Tous les stagiaires peuvent y lire. Seul setupuser (via Ansible) peut y écrire.

📋 Ta mission

Crée le répertoire principal {{ partage_dir }} (owner: root, group: sudo, mode: 0755), puis les sous-répertoires :

  • {{ partage_dir }}/paquets
  • {{ partage_dir }}/configs
  • {{ partage_dir }}/scripts

Essaie de créer les 3 sous-répertoires en une seule tâche avec une boucle.

Module pour créer des dossiers : ansible.builtin.file avec state: directory

Tu peux utiliser une boucle loop avec une liste de chemins complets :

loop:
  - /opt/formation/paquets
  - /opt/formation/configs

Ou mieux, utilise la variable : "{{ partage_dir }}/paquets"

    - name: Créer le répertoire de partage principal
      ansible.builtin.file:
        path: "{{ partage_dir }}"
        state: directory
        owner: root
        group: sudo
        mode: '0755'

    - name: Créer les sous-répertoires
      ansible.builtin.file:
        path: "{{ item }}"
        state: directory
        owner: root
        group: sudo
        mode: '0755'
      loop:
        - "{{ partage_dir }}/paquets"
        - "{{ partage_dir }}/configs"
        - "{{ partage_dir }}/scripts"
4.2 Télécharger des paquets sans les installer
🎯 Contexte

apt download télécharge les fichiers .deb sans les installer. Utile pour créer un dépôt local ou préparer une installation hors ligne. On utilise ansible.builtin.command car apt download n'a pas de module Ansible dédié.

📋 Ta mission

Télécharge les paquets nginx, curl, vim dans {{ partage_dir }}/paquets/.

  • La commande doit s'exécuter dans le bon répertoire (chdir)
  • La tâche doit être idempotente : ne pas re-télécharger si le .deb existe déjà
  • Utilise une boucle sur la liste des paquets

Module command avec chdir et creates :

ansible.builtin.command:
  cmd: apt download {{ item }}
  chdir: /chemin/du/repertoire
  creates: /chemin/{{ item }}_*.deb

Le paramètre creates rend la tâche idempotente : si le fichier/pattern existe déjà, la tâche est skipped.

    - name: Téléchargement des paquets de formation
      ansible.builtin.command:
        cmd: "apt download {{ item }}"
        chdir: "{{ partage_dir }}/paquets"
        creates: "{{ partage_dir }}/paquets/{{ item }}_*.deb"
      loop:
        - nginx
        - curl
        - vim
🔍 Valider

ansible machine2 -i inventory/hosts.ini -m command -a "ls -lh /opt/formation/paquets"

Lance le playbook une 2e fois — les tâches de téléchargement doivent être skipped.

4.3 Déployer un README dynamique
🎯 Contexte

Un fichier README dans /opt/formation/ guidera les stagiaires. On le génère avec un template pour qu'il contienne les vraies informations de la machine.

📋 Ta mission

Crée templates/README.j2 — un fichier texte (pas HTML) avec :

  • Titre de la formation et promotion
  • Date de déploiement et infos machine
  • Liste des outils installés
  • Liste des paquets disponibles dans /opt/formation/paquets/
  • Instructions de connexion SSH par stagiaire

Puis ajoute une tâche qui déploie ce README vers {{ partage_dir }}/README.txt

Un template peut générer n'importe quel type de fichier, pas seulement du HTML.

Pour un titre en texte brut, utilise des = ou des -.

La liste des outils vient de la variable outils_base dans all.yml.

Filtre Jinja2 utile : {{ ma_variable | upper }} pour mettre en majuscules.

# templates/README.j2
================================================
{{ formation_name | upper }}
{{ promotion }}
================================================

Déployé le : {{ ansible_date_time.date }}
Serveur    : {{ ansible_hostname }} ({{ ansible_default_ipv4.address }})

OUTILS INSTALLÉS
----------------
{% for o in outils_base %}
  - {{ o }}
{% endfor %}

PAQUETS DISPONIBLES
-------------------
  Répertoire : {{ partage_dir }}/paquets/

CONNEXION
---------
{% for s in stagiaires %}
  ssh {{ s }}@{{ ansible_hostname }}
{% endfor %}

Page d'accueil : http://{{ ansible_default_ipv4.address }}
# Tâche dans site.yml
    - name: Déploiement du README
      ansible.builtin.template:
        src: README.j2
        dest: "{{ partage_dir }}/README.txt"
        mode: '0644'

Industrialiser

Le projet fonctionne, mais tout est dans un seul playbook. Difficile à maintenir, impossible à réutiliser. On découpe en roles — une brique par responsabilité.

5.1 Planifier la découpe en roles
🎯 Contexte

Avant de créer les fichiers, réfléchis à la découpe. Un bon role a une seule responsabilité. Si le nom d'un role contient "et", c'est souvent le signe qu'il faut le couper en deux.

📋 Ta mission

Identifie quelles tâches de ton site.yml appartiennent à chaque role :

  • common → ce qui est commun à toute machine (outils, utilisateurs, sudo)
  • webserver → tout ce qui concerne nginx (install, service, template, handler)
  • partage → répertoires, téléchargements, README
  • securite → (nouveau) configurer SSH : PermitRootLogin no, PasswordAuthentication no

Note sur papier la répartition avant de créer les fichiers.

✅ Conseil

Commence toujours par la découpe sur papier. Créer les fichiers avant d'avoir clarifié les responsabilités mène souvent à tout réorganiser ensuite.

5.2 Créer la structure des roles
📋 Ta mission

Crée l'arborescence complète des 4 roles :

roles/
├── common/tasks/
├── webserver/tasks/
├── webserver/handlers/
├── webserver/templates/
├── partage/tasks/
├── partage/templates/
└── securite/tasks/
    securite/handlers/

Déplace ensuite accueil.html.j2 et README.j2 dans les bons dossiers templates/.

Vérifie : find roles/ -type d

mkdir -p crée tous les dossiers intermédiaires en une fois.

Pour déplacer : mv templates/accueil.html.j2 roles/webserver/templates/

Dans un role, les templates sont cherchés dans roles/nom_role/templates/ — tu n'écris que le nom du fichier dans src:, plus le chemin complet.

5.3 Créer les fichiers main.yml de chaque role
🎯 Contexte

Un main.yml de role ne contient que la liste des tâches — pas d'entête de play. On commence directement par le premier tiret. Les variables de group_vars/all.yml sont disponibles automatiquement.

📋 Ta mission

Crée les 4 fichiers roles/*/tasks/main.yml en répartissant les tâches selon la découpe de l'étape 5.1.

  • roles/webserver/handlers/main.yml — le handler nginx
  • roles/securite/tasks/main.yml — deux tâches lineinfile pour SSH
  • roles/securite/handlers/main.yml — handler restart sshd

Dans le role webserver, le src du template devient juste accueil.html.j2 sans chemin.

Pour le role securite, utilise ansible.builtin.lineinfile :

ansible.builtin.lineinfile:
  path: /etc/ssh/sshd_config
  regexp: '^#?PermitRootLogin'
  line: 'PermitRootLogin no'
notify: Redémarrer sshd

Le handler correspondant :

- name: Redémarrer sshd
  ansible.builtin.service:
    name: sshd
    state: restarted
# roles/common/tasks/main.yml
- name: Installation des outils de base
  ansible.builtin.apt:
    name: "{{ item }}"
    state: present
    update_cache: true
  loop: "{{ outils_base }}"

- name: Création des comptes stagiaires
  ansible.builtin.user:
    name: "{{ item }}"
    shell: /bin/bash
    create_home: true
    groups: sudo
    state: present
  loop: "{{ stagiaires }}"

- name: Configuration sudo sans mot de passe
  ansible.builtin.copy:
    content: "setupuser ALL=(ALL) NOPASSWD: ALL\n"
    dest: /etc/sudoers.d/setupuser
    owner: root
    group: root
    mode: '0440'
    validate: /usr/sbin/visudo -cf %s
# roles/webserver/tasks/main.yml
- name: Installation de nginx
  ansible.builtin.apt:
    name: nginx
    state: present

- name: Démarrage du service nginx
  ansible.builtin.service:
    name: nginx
    state: started
    enabled: true

- name: Déploiement de la page d'accueil
  ansible.builtin.template:
    src: accueil.html.j2
    dest: /var/www/html/index.html
    mode: '0644'
  notify: Redémarrer nginx

# roles/webserver/handlers/main.yml
- name: Redémarrer nginx
  ansible.builtin.service:
    name: nginx
    state: restarted
# roles/securite/tasks/main.yml
- name: Désactiver la connexion root SSH
  ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    regexp: '^#?PermitRootLogin'
    line: 'PermitRootLogin no'
  notify: Redémarrer sshd

- name: Désactiver l'auth par mot de passe SSH
  ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    regexp: '^#?PasswordAuthentication'
    line: 'PasswordAuthentication no'
  notify: Redémarrer sshd

# roles/securite/handlers/main.yml
- name: Redémarrer sshd
  ansible.builtin.service:
    name: sshd
    state: restarted
5.4 Réécrire le site.yml final
🎯 Contexte

Le site.yml ne doit plus contenir aucune tâche — il ne fait qu'appeler les roles dans le bon ordre. On ajoute des tags pour pouvoir exécuter seulement certains roles.

📋 Ta mission

Réécris site.yml pour appeler les 4 roles dans cet ordre : securite → common → webserver → partage.

Ajoute un tag sur chaque role (le nom du role comme tag). Syntaxe :

roles:
  - role: mon_role
    tags: mon_role

Teste les tags :

ansible-playbook -i inventory/hosts.ini site.yml --tags webserver
ansible-playbook -i inventory/hosts.ini site.yml --tags common,partage
---

- name: Déploiement environnement de formation DevCorp
  hosts: formation
  become: true

  roles:
    - role: securite
      tags: securite
    - role: common
      tags: common
    - role: webserver
      tags: webserver
    - role: partage
      tags: partage
🔍 Valider le projet complet
  • Lancer le playbook complet sans erreur
  • Tester les tags — seul le role ciblé s'exécute
  • Vérifier la page web sur les deux IPs
  • ansible all -i inventory/hosts.ini -m command -a "cat /opt/formation/README.txt"
  • ansible all -i inventory/hosts.ini -m command -a "grep PermitRootLogin /etc/ssh/sshd_config"
