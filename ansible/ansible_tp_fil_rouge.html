🏢 Scénario : Tu es administrateur système chez DevCorp. On te confie une mission : automatiser entièrement la préparation des environnements de formation pour les nouvelles promotions.

Les fondations

Avant d'automatiser quoi que ce soit, il faut savoir ce qu'on gère et vérifier qu'Ansible peut joindre les machines. Un mauvais inventory, c'est un playbook qui ne s'exécute jamais sur les bonnes cibles.

1.1Créer la structure du projet
🎯 Contexte

Ansible cherche ses fichiers dans des emplacements conventionnels. La structure que tu crées maintenant servira de base pour toute la journée.

📋 Ta mission

Crée l'arborescence suivante dans ton répertoire home :

devcorp-ansible/
├── inventory/
│   └── group_vars/
├── roles/
└── templates/

Vérifie avec : find ~/devcorp-ansible -type d

La commande mkdir -p crée tous les dossiers intermédiaires en une seule fois. Tu peux chaîner plusieurs commandes avec &&.

1.2Créer l'inventory
🎯 Contexte

L'inventory dit QUI sont tes machines et COMMENT les joindre. On les organise en groupes — toutes les machines de formation appartiennent au groupe [formation].

📋 Ta mission

Crée inventory/hosts.ini avec le groupe [formation], ses deux machines (avec alias et IP), et le bloc [formation:vars] avec les 3 directives de connexion. Utilisateur : setupuser — Clé : ~/.ssh/ansible_key.

Vérifie : ansible all -i inventory/hosts.ini --list-hosts

Noms de groupes et alias : libres. Directives ansible_XXX : imposées par Ansible.

Clé SSH : ansible_ssh_private_key_file — Python : ansible_python_interpreter

[formation]
machine2   ansible_host=10.116.0.238
machine3   ansible_host=10.116.0.239

[formation:vars]
ansible_user=setupuser
ansible_ssh_private_key_file=~/.ssh/ansible_key
ansible_python_interpreter=/usr/bin/python3
1.3Créer le fichier de variables globales
🎯 Contexte

Les variables dans group_vars/ permettent de changer les données sans toucher aux playbooks. Le fichier all.yml s'applique automatiquement à toutes les machines. Son nom doit correspondre exactement au nom du groupe.

📋 Ta mission

Crée inventory/group_vars/all.yml avec ces 5 variables : formation_name, promotion, stagiaires (liste de 3 comptes), outils_base (vim, curl, git, htop, tree, jq), partage_dir (/opt/formation).

Le fichier commence par ---. Variable simple : nom: "valeur"

ma_liste:
  - element1
  - element2
---

formation_name: "Formation Ansible DevCorp"
promotion: "Promo Avril 2025"

stagiaires:
  - stagiaire01
  - stagiaire02
  - stagiaire03

outils_base:
  - vim
  - curl
  - git
  - htop
  - tree
  - jq

partage_dir: /opt/formation
1.4Tester la connectivité et les commandes ad-hoc
🎯 Contexte

Les commandes ad-hoc exécutent un module directement en ligne de commande sans playbook. Utile pour tester et déboguer rapidement.

📋 Ta mission
# 1. Tester SSH + Python
ansible all -i inventory/hosts.ini -m ping

# 2. Vérifier l'uptime
ansible all -i inventory/hosts.ini -m command -a "uptime"

# 3. Espace disque
ansible all -i inventory/hosts.ini -m command -a "df -h /"

# 4. Facts de machine2
ansible machine2 -i inventory/hosts.ini -m setup | grep -A5 ipv4
🔍 Valider
  • Le ping retourne SUCCESS / pong sur les deux machines
  • Tu vois l'adresse IP de machine2 dans la sortie setup

Préparer les machines

Les machines sont joignables. Il faut maintenant les configurer : outils, comptes stagiaires, et sudo automatisé pour que les playbooks suivants n'aient plus besoin de mot de passe.

2.1Créer le playbook principal site.yml
🎯 Contexte

Le site.yml est le point d'entrée du projet. Pour l'instant il contiendra directement les tâches — on le refactorisera en roles à l'Acte 5.

📋 Ta mission

Crée ~/devcorp-ansible/site.yml avec un play ciblant le groupe formation, become: true, et une tâche debug affichant Démarrage de la configuration de {{ ansible_hostname }}.

Lance : ansible-playbook -i inventory/hosts.ini site.yml --ask-become-pass

---
- name: Nom du play
  hosts: mon_groupe
  become: true
  tasks:
    - name: Ma tâche
      ansible.builtin.debug:
        msg: "{{ ma_variable }}"
---

- name: Préparation de l'environnement DevCorp
  hosts: formation
  become: true

  tasks:

    - name: Annonce de démarrage
      ansible.builtin.debug:
        msg: "Démarrage de la configuration de {{ ansible_hostname }}"
2.2Installer les outils de base
🎯 Contexte

Grâce à loop, une seule tâche suffit pour toute la liste. Si on ajoute un outil dans all.yml, il sera automatiquement installé au prochain lancement.

📋 Ta mission

Ajoute une tâche qui installe tous les outils de outils_base avec le module apt, update_cache: true, et une boucle loop. Lance avec --check d'abord, puis sans.

Boucle avec une variable : loop: "{{ ma_variable }}" — élément courant : name: "{{ item }}" — guillemets obligatoires quand ça commence par {{.

    - name: Installation des outils de base
      ansible.builtin.apt:
        name: "{{ item }}"
        state: present
        update_cache: true
      loop: "{{ outils_base }}"
🔍 Valider

Relance le playbook une deuxième fois. Tous les paquets affichent ok (changed: 0) — l'idempotence en action.

2.3Créer les comptes stagiaires
📋 Ta mission

Ajoute une tâche qui crée les comptes de stagiaires : shell /bin/bash, home créé, groupe secondaire sudo, state present. Vérifie : ansible machine2 -i inventory/hosts.ini -m command -a "grep stagiaire /etc/passwd"

Module : ansible.builtin.user — Groupes secondaires : groups (pas group) — Home : create_home: true

    - name: Création des comptes stagiaires
      ansible.builtin.user:
        name: "{{ item }}"
        shell: /bin/bash
        create_home: true
        groups: sudo
        state: present
      loop: "{{ stagiaires }}"
2.4Configurer sudo sans mot de passe pour Ansible
🎯 Contexte

En production, le compte Ansible doit pouvoir faire sudo sans mot de passe pour l'automatisation complète. On configure ça via /etc/sudoers.d/ — plus sûr que modifier directement /etc/sudoers.

📋 Ta mission

Crée /etc/sudoers.d/setupuser avec le contenu setupuser ALL=(ALL) NOPASSWD: ALL. Utilise ansible.builtin.copy avec content:, mode 0440, owner et group root. Après ce lancement réussi, tu peux supprimer --ask-become-pass.

content: écrit du texte directement sans fichier source. Le \n final est important.

ansible.builtin.copy:
  content: "ma ligne\n"
  dest: /chemin/fichier
  mode: '0440'
  owner: root
  group: root

Bonus : validate: /usr/sbin/visudo -cf %s valide la syntaxe avant d'écrire.

    - name: Configuration sudo sans mot de passe pour setupuser
      ansible.builtin.copy:
        content: "setupuser ALL=(ALL) NOPASSWD: ALL\n"
        dest: /etc/sudoers.d/setupuser
        owner: root
        group: root
        mode: '0440'
        validate: /usr/sbin/visudo -cf %s
🔍 Valider

Lance une fois avec --ask-become-pass. Ensuite sans — il doit fonctionner. Tu n'en auras plus besoin pour la suite.

Informer les stagiaires

Les comptes sont créés, les outils installés. Maintenant chaque stagiaire doit savoir où il est et ce qui est disponible. On déploie une page web d'accueil personnalisée par machine.

3.1Installer et démarrer nginx
📋 Ta mission

Ajoute deux tâches : installer nginx et démarrer le service (state: started, enabled: true). Vérifie dans un navigateur : http://10.116.0.238 → page par défaut nginx.

Installation : ansible.builtin.apt — Service : ansible.builtin.service — State possibles : started / stopped / restarted / reloaded

3.2Créer le template de la page d'accueil
🎯 Contexte

Un template Jinja2 génère des fichiers dynamiques. Ansible remplace les {{ variable }} et évalue les {% blocs %} avant de déposer le fichier. Chaque machine reçoit un fichier personnalisé.

Facts Ansible disponibles
FactContenu
ansible_hostnameNom d'hôte
ansible_default_ipv4.addressAdresse IP principale
ansible_distributionDistribution Linux
ansible_distribution_versionVersion
ansible_memtotal_mbRAM totale en Mo
ansible_date_time.dateDate courante
📋 Ta mission

Crée templates/accueil.html.j2 affichant : nom de la formation, promotion, hostname, IP, distribution, RAM, liste des stagiaires (boucle for), liste des outils (boucle for), date de déploiement.

{{ ma_variable }}         {# afficher une variable #}
{{ fact.sous_cle }}       {# fact imbriqué #}
{% for x in ma_liste %}   {# boucle #}
  {{ x }}
{% endfor %}

⚠ ansible_default_ipv4.address — deux d dans address. Erreur très fréquente.

3.3Déployer le template et configurer le handler
🎯 Contexte

Le handler redémarre nginx uniquement si la page a changé. Sans handler, nginx redémarrerait à chaque exécution — inutile et perturbant pour les stagiaires connectés.

📋 Ta mission
  • Tâche : déployer accueil.html.j2 → /var/www/html/index.html, mode 0644, avec notify
  • Bloc handlers: (même niveau que tasks:) avec un handler qui redémarre nginx

Teste en deux temps : avec puis sans modification du template.

Module : ansible.builtin.template — Le notify et le name du handler doivent être strictement identiques.

Le bloc handlers: se place au même niveau d'indentation que tasks:.

  # Dans tasks:
    - name: Déploiement de la page d'accueil
      ansible.builtin.template:
        src: accueil.html.j2
        dest: /var/www/html/index.html
        mode: '0644'
      notify: Redémarrer nginx

  # Bloc handlers: au même niveau que tasks:
  handlers:
    - name: Redémarrer nginx
      ansible.builtin.service:
        name: nginx
        state: restarted
🔍 Valider
  • http://10.116.0.238 → hostname et IP de machine2
  • http://10.116.0.239 → hostname et IP de machine3

Mettre des ressources à disposition

Les stagiaires ont leurs comptes et leur page d'accueil. Il faut maintenant préparer un répertoire partagé : paquets téléchargés, README dynamique.

4.1Créer le répertoire partagé
🎯 Contexte

Le répertoire /opt/formation sera le dépôt commun. Tous les stagiaires peuvent y lire. Seul setupuser (via Ansible) peut y écrire.

📋 Ta mission

Crée {{ partage_dir }} (owner: root, group: sudo, mode: 0755), puis les sous-répertoires paquets, configs, scripts — idéalement en une seule tâche avec une boucle.

Module : ansible.builtin.file avec state: directory

loop:
  - "{{ partage_dir }}/paquets"
  - "{{ partage_dir }}/configs"
  - "{{ partage_dir }}/scripts"
    - name: Créer le répertoire principal
      ansible.builtin.file:
        path: "{{ partage_dir }}"
        state: directory
        owner: root
        group: sudo
        mode: '0755'

    - name: Créer les sous-répertoires
      ansible.builtin.file:
        path: "{{ item }}"
        state: directory
        owner: root
        group: sudo
        mode: '0755'
      loop:
        - "{{ partage_dir }}/paquets"
        - "{{ partage_dir }}/configs"
        - "{{ partage_dir }}/scripts"
4.2Télécharger des paquets sans les installer
🎯 Contexte

apt download télécharge les .deb sans les installer. On utilise ansible.builtin.command car apt download n'a pas de module dédié.

📋 Ta mission

Télécharge nginx, curl, vim dans {{ partage_dir }}/paquets/. Contraintes : utilise chdir pour exécuter dans le bon répertoire, et creates pour rendre la tâche idempotente.

ansible.builtin.command:
  cmd: apt download {{ item }}
  chdir: /chemin/repertoire
  creates: /chemin/{{ item }}_*.deb

creates : si le fichier/pattern existe déjà, la tâche est skipped → idempotence.

    - name: Téléchargement des paquets de formation
      ansible.builtin.command:
        cmd: "apt download {{ item }}"
        chdir: "{{ partage_dir }}/paquets"
        creates: "{{ partage_dir }}/paquets/{{ item }}_*.deb"
      loop:
        - nginx
        - curl
        - vim
🔍 Valider

ansible machine2 -i inventory/hosts.ini -m command -a "ls -lh /opt/formation/paquets" — Lance le playbook une 2e fois : les téléchargements doivent être skipped.

4.3Déployer un README dynamique
🎯 Contexte

Un template peut générer n'importe quel type de fichier — pas seulement du HTML. Un README texte guidera les stagiaires avec les vraies informations de leur machine.

📋 Ta mission

Crée templates/README.j2 (fichier texte) avec : titre, promotion, date, infos machine, liste des outils, instructions SSH par stagiaire. Déploie-le vers {{ partage_dir }}/README.txt.

Titres en texte brut : utilise des = ou des -. Filtre utile : {{ ma_variable | upper }}. Tu peux boucler sur stagiaires pour générer les lignes SSH.

# templates/README.j2
================================================
{{ formation_name | upper }}
{{ promotion }}
================================================

Déployé le : {{ ansible_date_time.date }}
Serveur    : {{ ansible_hostname }} ({{ ansible_default_ipv4.address }})

OUTILS INSTALLÉS
----------------
{% for o in outils_base %}
  - {{ o }}
{% endfor %}

CONNEXION
---------
{% for s in stagiaires %}
  ssh {{ s }}@{{ ansible_hostname }}
{% endfor %}

Page d'accueil : http://{{ ansible_default_ipv4.address }}
# Tâche dans site.yml
    - name: Déploiement du README
      ansible.builtin.template:
        src: README.j2
        dest: "{{ partage_dir }}/README.txt"
        mode: '0644'

Industrialiser

Le projet fonctionne, mais tout est dans un seul playbook. Difficile à maintenir, impossible à réutiliser. On découpe en roles — une brique par responsabilité.

5.1Planifier la découpe en roles
🎯 Contexte

Avant de créer les fichiers, réfléchis à la découpe. Un bon role a une seule responsabilité. Si son nom contient "et", c'est souvent le signe qu'il faut le couper en deux.

📋 Ta mission

Note sur papier la répartition des tâches dans ces 4 roles :

  • common — outils, utilisateurs, sudo
  • webserver — nginx install, service, template, handler
  • partage — répertoires, téléchargements, README
  • securite — SSH : PermitRootLogin no, PasswordAuthentication no
✅ Conseil

Commence toujours par la découpe sur papier. Créer les fichiers avant d'avoir clarifié les responsabilités mène souvent à tout réorganiser ensuite.

5.2Créer la structure des roles
📋 Ta mission

Crée l'arborescence complète des 4 roles et déplace les templates dans les bons dossiers :

roles/
├── common/tasks/
├── webserver/tasks/
├── webserver/handlers/
├── webserver/templates/       ← accueil.html.j2
├── partage/tasks/
├── partage/templates/         ← README.j2
├── securite/tasks/
└── securite/handlers/

Vérifie : find roles/ -type d

mkdir -p crée tous les dossiers en une fois.

Déplacer : mv templates/accueil.html.j2 roles/webserver/templates/

Dans un role, src: ne contient que le nom du fichier — Ansible cherche dans roles/nom_role/templates/ automatiquement.

5.3Créer les fichiers main.yml de chaque role
🎯 Contexte

Un main.yml de role ne contient que la liste des tâches — pas d'entête de play. On commence directement par le premier tiret. Les variables de group_vars/all.yml sont disponibles automatiquement.

📋 Ta mission

Crée les fichiers tasks/main.yml des 4 roles selon la découpe de l'étape 5.1. N'oublie pas :

  • roles/webserver/handlers/main.yml — handler nginx
  • roles/securite/tasks/main.yml — deux tâches lineinfile
  • roles/securite/handlers/main.yml — handler sshd
ansible.builtin.lineinfile:
  path: /etc/ssh/sshd_config
  regexp: '^#?PermitRootLogin'
  line: 'PermitRootLogin no'
notify: Redémarrer sshd

Le ^#? gère la ligne commentée ET non commentée en une seule regexp.

# roles/common/tasks/main.yml
- name: Installation des outils de base
  ansible.builtin.apt:
    name: "{{ item }}"
    state: present
    update_cache: true
  loop: "{{ outils_base }}"

- name: Création des comptes stagiaires
  ansible.builtin.user:
    name: "{{ item }}"
    shell: /bin/bash
    create_home: true
    groups: sudo
    state: present
  loop: "{{ stagiaires }}"

- name: Configuration sudo sans mot de passe
  ansible.builtin.copy:
    content: "setupuser ALL=(ALL) NOPASSWD: ALL\n"
    dest: /etc/sudoers.d/setupuser
    owner: root
    group: root
    mode: '0440'
    validate: /usr/sbin/visudo -cf %s
# roles/webserver/tasks/main.yml
- name: Installation de nginx
  ansible.builtin.apt:
    name: nginx
    state: present

- name: Démarrage du service nginx
  ansible.builtin.service:
    name: nginx
    state: started
    enabled: true

- name: Déploiement de la page d'accueil
  ansible.builtin.template:
    src: accueil.html.j2
    dest: /var/www/html/index.html
    mode: '0644'
  notify: Redémarrer nginx

# roles/webserver/handlers/main.yml
- name: Redémarrer nginx
  ansible.builtin.service:
    name: nginx
    state: restarted
# roles/securite/tasks/main.yml
- name: Désactiver la connexion root SSH
  ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    regexp: '^#?PermitRootLogin'
    line: 'PermitRootLogin no'
  notify: Redémarrer sshd

- name: Désactiver l'auth par mot de passe SSH
  ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    regexp: '^#?PasswordAuthentication'
    line: 'PasswordAuthentication no'
  notify: Redémarrer sshd

# roles/securite/handlers/main.yml
- name: Redémarrer sshd
  ansible.builtin.service:
    name: sshd
    state: restarted
5.4Réécrire le site.yml final
🎯 Contexte

Le site.yml ne doit plus contenir aucune tâche — il appelle les roles dans le bon ordre et leur associe des tags pour l'exécution sélective.

📋 Ta mission

Réécris site.yml pour appeler les 4 roles dans l'ordre : securite → common → webserver → partage, chacun avec son tag. Teste :

ansible-playbook -i inventory/hosts.ini site.yml --tags webserver
ansible-playbook -i inventory/hosts.ini site.yml --tags common,partage
---

- name: Déploiement environnement de formation DevCorp
  hosts: formation
  become: true

  roles:
    - role: securite
      tags: securite
    - role: common
      tags: common
    - role: webserver
      tags: webserver
    - role: partage
      tags: partage
🔍 Valider le projet complet
  • Lancer le playbook complet — aucune erreur
  • Tester les tags — seul le role ciblé s'exécute
  • Vérifier la page web sur les deux IPs
  • ansible all -i inventory/hosts.ini -m command -a "cat /opt/formation/README.txt"
  • ansible all -i inventory/hosts.ini -m command -a "grep PermitRootLogin /etc/ssh/sshd_config"
