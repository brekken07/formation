01

Pourquoi Ansible ?

Comprendre le problème qu'Ansible résout avant d'apprendre la syntaxe.

Le problème

Sans outil d'automatisation, gérer plusieurs serveurs Linux revient à se connecter manuellement en SSH sur chacun, taper les mêmes commandes, faire des erreurs différentes sur chaque machine, et ne plus savoir exactement ce qui est installé où.

Ansible résout ce problème : on décrit l'état désiré des machines dans des fichiers texte, et Ansible applique cet état partout, de façon identique et reproductible.

Concept clé — l'idempotence
ℹ Idempotence

Si tu lances le même playbook 10 fois, le résultat est identique à la 1re exécution. Ansible vérifie l'état actuel avant d'agir. Si nginx est déjà installé, il ne réinstalle pas.

  • changed: 1 → Ansible a effectué une modification réelle
  • changed: 0 → Rien à faire, état déjà conforme
Comment ça fonctionne
Machine de contrôle (10.116.0.195)
├── Inventory  →  liste des machines cibles
├── Playbook   →  ce qu'on veut faire
└── Ansible    →  se connecte via SSH
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
     machine2      machine3      ...
  (10.116.0.238) (10.116.0.239)

Questions / Réponses
Qu'est-ce que l'idempotence en Ansible ?
Si on lance le même playbook plusieurs fois, le résultat est identique à la 1re exécution. Ansible vérifie l'état actuel avant d'agir et ne modifie que ce qui est nécessaire.
Ansible a-t-il besoin d'un agent sur les machines cibles ?
Non. Ansible utilise SSH + Python 3. Rien d'autre à installer sur les machines cibles.
02

Installation et premier contact

Mettre en place l'environnement et configurer l'accès SSH par clé.

Installation sur la machine de contrôle
# Installation via pip (recommandée)
$ pip3 install ansible

# Ou via le dépôt Debian
$ sudo apt install ansible -y

# Vérification
$ ansible --version
SSH vs sudo — deux mécanismes distincts
⚠ Point clé
  • La clé SSH → sert à SE CONNECTER à la machine
  • Le mot de passe sudo → sert à ÉLEVER LES PRIVILÈGES une fois connecté

Avec become: true dans un playbook, Ansible a besoin du mot de passe sudo séparément (option --ask-become-pass).

Configuration de l'accès SSH par clé
# 1. Générer une paire de clés ED25519 sur la machine de contrôle
$ ssh-keygen -t ed25519 -C "ansible" -f ~/.ssh/ansible_key

# 2. Copier la clé publique sur chaque machine cible
$ ssh-copy-id -i ~/.ssh/ansible_key.pub setupuser@10.116.0.238
$ ssh-copy-id -i ~/.ssh/ansible_key.pub setupuser@10.116.0.239

# 3. Tester la connexion sans mot de passe
$ ssh -i ~/.ssh/ansible_key setupuser@10.116.0.238
EX 1Tester la connexion SSH par clé
  1. Vérifier que la clé ansible_key existe : ls ~/.ssh/
  2. Se connecter sans mot de passe : ssh -i ~/.ssh/ansible_key setupuser@10.116.0.238
  3. Faire de même sur machine3

Résultat attendu : connexion directe sans saisie de mot de passe.

03

L'Inventory

Définir quelles machines Ansible gère et comment les atteindre.

Structure du projet
ansible-tp/                    ← Racine du projet
├── inventory/
│   ├── hosts.ini              ← Liste des machines et groupes
│   └── group_vars/
│       └── all.yml            ← Variables communes à TOUTES les machines
├── roles/
├── templates/
└── site.yml                   ← Playbook principal
Le fichier hosts.ini
[machines_cibles]
machine2   ansible_host=10.116.0.238
machine3   ansible_host=10.116.0.239

[machines_cibles:vars]
ansible_user=setupuser
ansible_ssh_private_key_file=~/.ssh/ansible_key
ansible_python_interpreter=/usr/bin/python3
Ce qu'on invente vs ce qu'Ansible impose
ÉlémentQui décide ?Exemple
Nom du groupeToi — libre[machines_cibles]
Alias de la machineToi — libremachine2
Valeurs des variablesToisetupuser, ~/.ssh/...
Noms ansible_XXXAnsible — imposésansible_host, ansible_user
Syntaxe [groupe:vars]Ansible — imposée[machines_cibles:vars]
Règle group_vars
ℹ Règle clé

Le nom du fichier dans group_vars/ doit correspondre exactement au nom du groupe dans hosts.ini.

  • [machines_cibles] → group_vars/machines_cibles.yml
  • [all] → group_vars/all.yml (chargé pour TOUTES les machines)

Une erreur de nommage est ignorée silencieusement, sans message d'erreur.

Questions / Réponses
Dans hosts.ini, 'machine2' est-il le hostname réel de la VM ?
Non. C'est un alias libre. Le hostname réel est défini par ansible_host=10.116.0.238. L'alias sert uniquement à l'affichage dans les sorties Ansible.
Si j'ai [webservers] et un fichier group_vars/serveurs_web.yml, Ansible fait-il le lien ?
Non. Le nom du fichier doit être exactement identique au nom du groupe. group_vars/webservers.yml est le fichier correct.
EX 2Créer et vérifier l'inventory
  1. Créer la structure : mkdir -p ~/ansible-tp/inventory/group_vars
  2. Créer hosts.ini avec le groupe [machines_cibles] et ses deux machines
  3. Ajouter le bloc [machines_cibles:vars] avec les 3 directives de connexion
  4. Vérifier : ansible all -i inventory/hosts.ini --list-hosts
  5. Tester : ansible all -i inventory/hosts.ini -m ping

Résultat attendu : SUCCESS / pong sur machine2 et machine3.

04

Le Playbook

Décrire les actions à effectuer sur les machines cibles.

Règles YAML
✅ Règles indispensables
  • Indentation obligatoire avec des espaces, jamais des tabulations
  • Convention : 2 espaces par niveau d'indentation
  • Le tiret - signifie "élément d'une liste"
  • Toute valeur commençant par {{ doit être entre guillemets : "{{ variable }}"
  • Le fichier commence toujours par ---
Anatomie complète
---                              # Marqueur de début YAML

- name: Nom du play             # Nom libre
  hosts: machines_cibles        # Groupe cible de l'inventory
  become: true                  # Utiliser sudo

  tasks:

    - name: Installer nginx
      ansible.builtin.apt:
        name: nginx
        state: present          # present / absent / latest

    - name: Démarrer nginx
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: true           # Démarrage automatique au boot

  handlers:                     # Exécutés une seule fois en fin de play
    - name: Restart nginx
      ansible.builtin.service:
        name: nginx
        state: restarted
Commandes d'exécution
# Simulation sans modification — TOUJOURS avant un déploiement
$ ansible-playbook -i inventory/hosts.ini site.yml --check

# Exécution réelle
$ ansible-playbook -i inventory/hosts.ini site.yml

# Avec mot de passe sudo
$ ansible-playbook -i inventory/hosts.ini site.yml --ask-become-pass

# Mode verbeux
$ ansible-playbook -i inventory/hosts.ini site.yml -v

# Limiter à une machine
$ ansible-playbook -i inventory/hosts.ini site.yml --limit machine2
Lire la sortie
StatutSignification
okTâche exécutée, rien modifié (idempotence)
changedAnsible a effectué une modification réelle
skippedTâche ignorée (condition when non remplie)
unreachableMachine non joignable
failedLa tâche a échoué
⚠ Limitation du mode --check

Le mode --check peut signaler des erreurs sur les tâches qui dépendent d'une tâche non encore exécutée (ex : démarrer un service non encore installé). Ce n'est pas forcément une erreur dans le playbook.

EX 3Premier playbook — installer vim
  1. Créer site.yml avec un play qui installe vim
  2. Tester avec --check --ask-become-pass
  3. Lancer pour de vrai avec --ask-become-pass
  4. Relancer une 2e fois → observer changed: 0 (idempotence)
05

Modules essentiels et boucles

La boîte à outils Ansible pour gérer une infrastructure Linux.

ansible.builtin.apt — paquets Debian/Ubuntu
- name: Installer nginx
  ansible.builtin.apt:
    name: nginx
    state: present        # present / absent / latest
    update_cache: true    # équivaut à apt update
ansible.builtin.service — services système
- name: Démarrer nginx
  ansible.builtin.service:
    name: nginx
    state: started        # started / stopped / restarted / reloaded
    enabled: true         # démarrage automatique au boot
ansible.builtin.user — utilisateurs
- name: Créer un utilisateur
  ansible.builtin.user:
    name: deployer
    shell: /bin/bash
    create_home: true
    groups: sudo
    state: present
ansible.builtin.copy — fichiers statiques
- name: Copier un fichier de configuration
  ansible.builtin.copy:
    src: files/app.conf
    dest: /etc/app/app.conf
    owner: root
    mode: '0644'
ansible.builtin.file — fichiers, dossiers, liens
- name: Créer un répertoire
  ansible.builtin.file:
    path: /opt/myapp
    state: directory      # directory / file / absent / link
    owner: deployer
    mode: '0755'
ansible.builtin.lineinfile — modifier une ligne
- name: Désactiver PermitRootLogin
  ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    regexp: '^PermitRootLogin'
    line: 'PermitRootLogin no'
  notify: Restart sshd
ansible.builtin.debug — afficher des informations
- name: Afficher l'IP
  ansible.builtin.debug:
    msg: "IP : {{ ansible_default_ipv4.address }}"
Les boucles avec loop

Sans boucle, installer 5 paquets demande 5 tâches identiques. Avec loop, on écrit la tâche une seule fois et on lui passe une liste. {{ item }} est remplacé automatiquement par chaque élément.

# Boucle avec liste directe
- name: Installer les outils
  ansible.builtin.apt:
    name: "{{ item }}"      # guillemets obligatoires
    state: present
  loop:
    - vim
    - curl
    - git

# Boucle avec une variable depuis group_vars/all.yml
- name: Installer les outils
  ansible.builtin.apt:
    name: "{{ item }}"
    state: present
  loop: "{{ outils_base }}"
⚠ Piège fréquent

Ne jamais écrire name: {{ item }} sans guillemets. En YAML, une valeur qui commence par {{ est ambiguë. Toujours écrire name: "{{ item }}".

Pourquoi préférer loop: "{{ outils }}" à une liste directe dans le playbook ?
Séparation des données et de la logique. La liste est dans group_vars/all.yml. Pour changer les outils, on modifie le fichier de variables sans toucher au playbook.
EX 4Boucles et utilisateurs
  1. Dans group_vars/all.yml : créer les variables outils et stagiaires
  2. Dans site.yml : utiliser loop: "{{ outils }}" pour l'installation
  3. Ajouter une tâche qui crée les utilisateurs en boucle sur stagiaires
  4. Vérifier : cat /etc/passwd | grep stagiaire

# templates/index.html.j2

<html>
<head><title>{{ formation_name }}</title></head>
<body>
    <h1>{{ ansible_hostname }}</h1>
    <p>IP : {{ ansible_default_ipv4.address }}</p>

ℹ Syntaxe Jinja2
  • {{ variable }} → afficher une variable ou un fact
  • {% if condition %} → bloc conditionnel
  • {% for x in liste %} → boucle sur une liste
  • {% endif %} / {% endfor %} → fermer le bloc
⚠ Piège fréquent

ansible_default_ipv4.address s'écrit avec deux d dans address. Erreur très fréquente qui provoque une erreur au rendu du template.

Quelle est la différence entre ansible.builtin.copy et ansible.builtin.template ?
copy dépose un fichier statique tel quel. template interprète les variables Jinja2 avant de déposer le fichier — chaque machine reçoit un fichier personnalisé avec ses propres valeurs.

Résultat attendu : page personnalisée avec le hostname et l'IP de chaque machine.

07

Les Handlers

Déclencher des actions uniquement quand quelque chose a vraiment changé.

Pourquoi les handlers ?

Sans handler, on serait tenté d'ajouter une tâche "restart nginx" après chaque modification de config. Le problème : cette tâche redémarrerait nginx à chaque exécution du playbook, même si rien n'a changé — provoquant des interruptions de service inutiles.

ℹ Un handler s'exécute si :
  1. Une tâche l'a notifié avec notify:
  2. ET cette tâche a produit un changed (pas un ok)

Il s'exécute une seule fois en fin de play, même si plusieurs tâches l'ont notifié.

Syntaxe
  tasks:

    - name: Déployer la config nginx
      ansible.builtin.template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: Redémarrer nginx      # ← même nom EXACT que le handler

  handlers:

    - name: Redémarrer nginx        # ← doit correspondre exactement
      ansible.builtin.service:
        name: nginx
        state: restarted
Ce qu'on voit dans la sortie
TASK [Déployer la config nginx] *****
changed: [machine2]                  ← changed → handler sera déclenché

RUNNING HANDLERS ********************
changed: [machine2]                  ← handler exécuté en fin de play

PLAY RECAP **************************
⚠ Piège fréquent

Le lien notify / handler se fait par correspondance stricte du nom. Un seul caractère différent (majuscule, accent, espace) et le handler est silencieusement ignoré, sans message d'erreur.

Quand le handler ne se déclenche-t-il pas ?
Quand la tâche qui le notifie retourne ok (pas changed). Si la config nginx n'a pas été modifiée, la tâche retourne ok et le handler n'est pas déclenché — nginx n'est pas redémarré inutilement.
EX 6Ajouter un handler au playbook
  1. Dans site.yml, ajouter un bloc handlers: après tasks:
  2. Créer un handler "Redémarrer nginx" avec state: restarted
  3. Ajouter notify: Redémarrer nginx sur la tâche de déploiement du template
  4. Lancer sans modifier le template → handler NON déclenché
  5. Modifier une ligne dans le template, relancer → RUNNING HANDLERS apparaît
08

Les Roles

Organiser le code en briques réutilisables.

Pourquoi les roles ?

Un playbook qui grossit devient vite illisible. Les roles découpent le code en modules indépendants, chacun responsable d'une seule fonctionnalité. Un role peut être réutilisé sur n'importe quel projet.

Structure d'un role
roles/
└── nginx/                    # Nom du role
    ├── tasks/
    │   └── main.yml          # Point d'entrée des tâches (obligatoire)
    ├── handlers/
    │   └── main.yml          # Handlers du role
    ├── templates/
    │   └── *.j2              # Templates Jinja2
    ├── files/                # Fichiers statiques
    ├── vars/
    │   └── main.yml          # Variables (priorité haute)
    └── defaults/
        └── main.yml          # Variables par défaut (priorité basse)
ℹ Règle importante

Un main.yml de role ne contient que la liste des tâches — pas d'entête de play. On commence directement par le premier tiret.

Dans un role, le src: d'un template ne contient que le nom du fichier. Ansible cherche automatiquement dans roles/nom_role/templates/.

Contenu d'un main.yml de role
# roles/nginx/tasks/main.yml
- name: Installation de nginx
  ansible.builtin.apt:
    name: nginx
    state: present

- name: Démarrage du service nginx
  ansible.builtin.service:
    name: nginx
    state: started
    enabled: true

- name: Déploiement de la page d'accueil
  ansible.builtin.template:
    src: index.html.j2        # Relatif à roles/nginx/templates/
    dest: /var/www/html/index.html
  notify: Redémarrer nginx
Utiliser les roles dans site.yml
---

- name: Configurer les serveurs web
  hosts: machines_cibles
  become: true

  roles:
    - role: common
      tags: common
    - role: nginx
      tags: nginx
    - role: docker
      when: install_docker | default(false)
Dans un role, doit-on préciser le chemin complet dans src du module template ?
Non. Dans un role, Ansible cherche automatiquement dans roles/nom_role/templates/. On écrit uniquement le nom du fichier : src: index.html.j2
Ansible parcourt-il automatiquement toute l'arborescence du projet ?
Non. Ansible suit des chemins conventionnels précis. Il charge group_vars/ automatiquement, et pour les roles il cherche tasks/main.yml, handlers/main.yml, templates/ à des emplacements fixes. Un fichier mal nommé ou mal placé est ignoré silencieusement.
EX 7Découper le playbook en roles
  1. Créer la structure : mkdir -p roles/common/tasks roles/nginx/{tasks,handlers,templates} roles/web/{tasks,templates}
  2. Créer roles/common/tasks/main.yml avec les tâches outils et utilisateurs
  3. Créer roles/nginx/tasks/main.yml avec installation et démarrage nginx
  4. Créer roles/nginx/handlers/main.yml avec le handler de redémarrage
  5. Déplacer index.html.j2 dans roles/web/templates/
  6. Créer roles/web/tasks/main.yml avec la tâche de déploiement du template
  7. Réécrire site.yml pour appeler les 3 roles
  8. Lancer le playbook et vérifier que tout fonctionne
