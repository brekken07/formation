01 ansible vs ansible-playbook — quelle différence ? ▼
CommandeUsage
ansibleCommandes ad-hoc : une tâche unique, un module, sur la ligne de commande
ansible-playbookExécuter un fichier YAML complet avec plusieurs plays et tâches
# Ad-hoc : tester la connectivité
ansible all -m ping
# Ad-hoc : installer nginx sans playbook
ansible webservers -m apt -a "name=nginx state=present" -b
# Playbook complet
ansible-playbook site.yml
02 --check et --diff — le dry-run Ansible ▼
FlagEffet
--checkSimule l'exécution sans modifier l'hôte (dry-run)
--diffAffiche les différences de fichiers (avant/après)
--check --diffCombo : simulation + aperçu des diffs — le plus utile
Piège
--check peut échouer sur des modules qui dépendent de ressources créées dans la même exécution. Si une tâche crée un dossier et une tâche suivante copie dedans, la 2e peut échouer en mode check (le dossier n'a pas été réellement créé).
03 -l, -t, --skip-tags, -e — les flags essentiels ▼
FlagRôle
-l webserversLimite l'exécution à un groupe ou hôte
-t install,configExécute seulement les tâches ayant ces tags
--skip-tags debugIgnore les tâches avec ce tag
-e "var=valeur"Extra vars — priorité maximale sur toutes les variables
-e @vars.ymlExtra vars depuis un fichier
-f 10Nombre de forks (hôtes en parallèle, défaut: 5)
--stepConfirmation manuelle avant chaque tâche
--start-at-task "nom"Reprend depuis une tâche spécifique
Piège
-e (extra-vars) a la priorité la plus haute — il écrase tout, y compris les variables définies dans un rôle ou un playbook. Utile en CI/CD, dangereux si mal utilisé.

06 Inventory INI vs YAML — formats et groupes ▼ 
# Format INI :
[webservers]
web1.example.com
web2.example.com  ansible_port=2222

[dbservers]
db1.example.com

[prod:children]   # groupe de groupes
webservers

[webservers:vars]  # variables du groupe

Piège
[groupe:children] déclare un groupe de groupes. [groupe:vars] déclare des variables de groupe. Ces deux syntaxes INI sont souvent confondues.
Patterns de sélection d'hôtes
all · webservers · web1 · web* · webservers:dbservers (union) · webservers:!db1 (exclusion) · webservers:&staging (intersection)

08 host_vars/ et group_vars/ — organisation des variables ▼ 
# Structure recommandée :
inventory/
├── hosts.ini
├── group_vars/
│   ├── all.yml          # variables pour TOUS les hôtes
│   ├── webservers.yml   # variables du groupe webservers
│   └── webservers/      # ou en dossier (plusieurs fichiers)
│       ├── vars.yml

Un dossier group_vars/webservers/ est équivalent au fichier group_vars/webservers.yml. Les deux formats sont valides. Le dossier permet de séparer les vars normales des vars vault.

09 Structure d'un play — les directives clés ▼ 
---
- name: Deploy web application
  hosts: webservers
  gather_facts: true
  become: true
  serial: 2              # rolling update
  any_errors_fatal: false
  vars:
    app_port: 8080
  vars_files:
    - vars/common.yml
  pre_tasks:   # avant les rôles
    - ...
  roles:
    - nginx
  tasks:
    - ...
  post_tasks:  # après les rôles
    - ...
  handlers:
    - ...
Ordre d'exécution
pre_tasks → roles → tasks → post_tasks → handlers (toujours à la fin)
10 when — conditions sur les tâches ▼ 
# Condition sur OS :
- ansible.builtin.apt:
    name: nginx
  when: ansible_os_family == "Debian"

# Conditions multiples (ET logique) :
  when:
    - ansible_os_family == "RedHat"
    - ansible_distribution_major_version | int >= 8

# OU logique :
  when: ansible_os_family == "Debian" or
        ansible_os_family == "Ubuntu"

# Sur le résultat d'une tâche précédente :
  when: result.rc != 0
Piège
Dans un when, les variables Jinja2 ne nécessitent pas les accolades {{ }}. Écrire directement when: variable == "valeur", pas when: "{{ variable }}" == "valeur".
11 loop vs with_items — le bon choix ▼
loopwith_items (ancien)
SyntaxeModerne (Ansible 2.5+)Déprécié
Variable d'itérationitemitem
Aplatissement autoNonOui (listes dans listes)
# Avec loop (recommandé) :
- ansible.builtin.user:
    name: "{{ item.name }}"
    groups: "{{ item.groups }}"
  loop:
    - { name: alice, groups: sudo }
    - { name: bob,   groups: docker }
Mémo
Dans la tâche : item est la variable d'itération. Pour un dictionnaire : item.clé. loop_var permet de renommer item pour éviter les conflits dans les boucles imbriquées.
12 register et les attributs du résultat ▼ 
# Capturer le résultat d'une tâche :
- ansible.builtin.command:
    cmd: nginx -v
  register: nginx_result
  changed_when: false

- ansible.builtin.debug:
    msg: "{{ nginx_result.stdout }}"
  when: nginx_result.rc == 0
AttributContenu
.rcCode de retour (0 = succès)
.stdoutSortie standard
.stderrSortie d'erreur
.stdout_linesstdout sous forme de liste
.changedtrue si la tâche a modifié quelque chose
.failedtrue si la tâche a échoué
13 block / rescue / always — gestion d'erreurs ▼ 
- block:
    - ansible.builtin.command:
        cmd: /opt/deploy.sh
  rescue:           # exécuté si le block échoue
    - ansible.builtin.debug:
        msg: "Déploiement échoué, rollback"
  always:           # exécuté dans tous les cas
    - ansible.builtin.debug:
        msg: "Nettoyage terminé"
Mémo — analogie Python
block = try · rescue = except · always = finally
Piège
Si rescue s'exécute et réussit, le play continue normalement — l'hôte n'est pas marqué comme failed. C'est différent de ignore_errors: true qui ne gère pas l'erreur, il l'ignore juste.
14 Handlers — comportement et flush ▼
À retenir
Les handlers s'exécutent une seule fois à la fin du play, même s'ils sont notifiés plusieurs fois. Ils ne s'exécutent que si la tâche qui les notifie a produit un changed. 
# Tâche qui notifie :
- ansible.builtin.template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  notify: restart nginx

# Handler :
handlers:
  - name: restart nginx
    ansible.builtin.service:
      name: nginx
      state: restarted

# Forcer l'exécution immédiate des handlers :
- ansible.builtin.meta:
    flush_handlers
Piège
Si le play s'arrête avec une erreur avant la fin, les handlers en attente ne s'exécutent pas. Utiliser ansible.builtin.meta: flush_handlers pour les forcer avant un point critique.
15 changed_when et failed_when — contrôler l'idempotence ▼ 
# changed_when : contrôler quand une tâche est "changed"
- ansible.builtin.command:
    cmd: nginx -t
  changed_when: false     # jamais "changed" (commande de check)

# failed_when : définir la condition d'échec
- ansible.builtin.command:
    cmd: /opt/script.sh
  register: result
  failed_when: result.rc not in [0, 1]  # 0 et 1 = OK
Piège
Les modules command et shell sont toujours changed car Ansible ne peut pas savoir si une commande shell a modifié quelque chose. Toujours utiliser changed_when avec eux pour maintenir l'idempotence.
16 import_tasks vs include_tasks — statique vs dynamique ▼
import_tasks (statique)include_tasks (dynamique)
RésolutionAu chargement du playbookÀ l'exécution
Tags héritésOuiNon
Nom de fichier dynamiqueNonOui (variable)
Visible par --list-tasksOuiNon
# Statique — le fichier est connu à l'avance :
- ansible.builtin.import_tasks:
    file: install.yml

# Dynamique — le fichier dépend d'une variable :
- ansible.builtin.include_tasks:
    file: "{{ ansible_os_family }}.yml"
Règle simple
Structure fixe → import. Fichier à déterminer à l'exécution → include.
17 serial — rolling update et contrôle du déploiement ▼
À retenir
serial contrôle combien d'hôtes sont traités à la fois dans un play. Utile pour les rolling updates sans downtime total. 
# Traiter 2 hôtes à la fois :
serial: 2

# Ou en pourcentage :
serial: "30%"

# Progression par étapes :
serial: [1, 5, "100%"]  # 1 puis 5 puis le reste
Piège
serial s'applique au niveau du play, pas de la tâche. Avec serial: 1, le play complet s'exécute sur l'hôte 1 avant de passer à l'hôte 2.

20 lineinfile vs blockinfile — modifier un fichier existant ▼
ModuleUsage
lineinfileAssure qu'une ligne est présente/absente dans un fichier
blockinfileAssure qu'un bloc de lignes est présent (avec marqueurs)
replaceRemplace toutes les occurrences d'un pattern regex
# lineinfile — désactiver auth password SSH :
- ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    regexp: '^PasswordAuthentication'
    line: 'PasswordAuthentication no'
    state: present
Piège
regexp dans lineinfile sert à trouver la ligne à remplacer. Si aucune ligne ne correspond, la nouvelle ligne est ajoutée à la fin. Si plusieurs lignes correspondent, seule la dernière est remplacée.

27 Structure d'un rôle — dossiers et rôles de chacun ▼ 
roles/nginx/
├── tasks/main.yml       # point d'entrée des tâches
├── handlers/main.yml    # handlers du rôle
├── templates/           # fichiers .j2 (Jinja2)
├── files/               # fichiers statiques (copy)
├── vars/main.yml        # vars haute priorité (non surchargeable facilement)
├── defaults/main.yml    # vars basse priorité (surchargeable)
├── meta/main.yml        # dépendances du rôle
└── README.md
Piège — defaults vs vars
defaults/main.yml = valeurs par défaut facilement surchargées → mettre les valeurs configurables.
vars/main.yml = variables internes au rôle difficiles à surcharger → mettre les constantes du rôle.
Créer un rôle vide
ansible-galaxy role init mon_role

Ansible Vault
30 Commandes Vault — chiffrer, déchiffrer, éditer ▼ 
ansible-vault create secrets.yml          # créer un fichier chiffré
ansible-vault encrypt vars/db.yml          # chiffrer un fichier existant
ansible-vault decrypt vars/db.yml          # déchiffrer ⚠ permanent
ansible-vault edit secrets.yml             # éditer sans déchiffrer sur disque
ansible-vault view secrets.yml             # afficher le contenu
ansible-vault rekey secrets.yml            # changer le mot de passe
ansible-vault encrypt_string 'motdepasse' --name db_pass
                                            # chiffrer une seule valeur inline
Piège
ansible-vault decrypt déchiffre le fichier de façon permanente sur disque. Pour juste lire le contenu, utiliser view. Pour modifier, utiliser edit.

33 Lookups — accéder à des données externes ▼ 
# Lire un fichier local :
key: "{{ lookup('file', '~/.ssh/id_rsa.pub') }}"

# Variable d'environnement :
path: "{{ lookup('env', 'HOME') }}"

# Générer un mot de passe aléatoire :
pass: "{{ lookup('password', '/dev/null length=16 chars=ascii_letters,digits') }}"

# Requête DNS :

Mémo
Les lookups s'exécutent sur le control node (pas sur l'hôte distant). Pour lire un fichier sur l'hôte distant, utiliser slurp ou fetch.

À retenir
Les modules Ansible sont idempotents par conception — sauf command, shell et raw qui sont toujours marqués changed.

37 ignore_errors vs failed_when vs block/rescue ▼
MécanismeComportement
ignore_errors: trueIgnore l'erreur, l'hôte n'est pas marqué failed, la tâche continue
failed_when: conditionDéfinit finement quand la tâche échoue
block / rescueGère l'erreur proprement, peut effectuer un rollback
Piège
ignore_errors: true n'est pas une bonne pratique générale — il masque les vrais problèmes. Préférer failed_when ou block/rescue pour une gestion d'erreur explicite.
38 delegate_to et run_once ▼ 
# Exécuter une tâche sur un autre hôte :
- ansible.builtin.command:
    cmd: haproxy -f /etc/haproxy.cfg -c
  delegate_to: loadbalancer.example.com

# Exécuter sur le control node :
- ansible.builtin.command:
    cmd: curl -s http://api/deploy
  delegate_to: localhost

# N'exécuter qu'une seule fois (sur le 1er hôte du play) :
- ansible.builtin.command:
    cmd: python manage.py migrate
  run_once: true
Mémo
delegate_to: localhost est l'équivalent du provisioner local-exec de Terraform. Utile pour les appels d'API ou les vérifications depuis le control node.

40 Pièges YAML à éviter dans les playbooks ▼ 
# ❌ YAML interprète yes/no comme booléen :
state: yes       # → true (booléen)
state: "yes"    # ✅ string "yes"

# ❌ Oublier les guillemets sur les nombres en string :
version: 1.10    # → float 1.1 (perd le zéro final)
version: "1.10"  # ✅ string "1.10"

# ❌ Double interpolation Jinja2 :
msg: "{{ "{{ var }}" }}"   # erreur
msg: "{{ var }}"          # ✅

# ❌ when avec double accolades :
when: "{{ var == 'value' }}"  # peut marcher mais mauvaise pratique
when: var == 'value'          # ✅ sans accolades dans when
Piège YAML classique
En YAML, yes, no, true, false, on, off sont des booléens. Toujours mettre entre guillemets si on veut une chaîne de caractères.
