Orchestrer des applications multi-conteneurs avec un seul fichier déclaratif.

// 01

Pourquoi Docker Compose ?

Dans le TP précédent, tu as lancé la stack DevCorp avec un script Bash de 80 lignes. C'est fonctionnel — mais pas déclaratif, pas lisible par un collègue, et difficile à maintenir. Docker Compose résout ça.

AVANT (scripts Bash)                  APRÈS (Docker Compose)
──────────────────────────────────    ──────────────────────────────────
docker network create devcorp-net     docker compose up -d
docker volume create devcorp-db
docker run -d --name db ...
docker run -d --name backend ...
docker run -d --name frontend ...
(80 lignes de script)

Compose lit un fichier YAML déclaratif (docker-compose.yml) qui décrit l'état désiré de ton application : quels services, sur quels réseaux, avec quels volumes, dans quel ordre. C'est l'Infrastructure as Code appliquée au niveau conteneur.

💡
Compose V2 — la syntaxe moderne

Depuis Docker Desktop 3.4+ et Docker Engine 20.10+, la commande est docker compose (sans tiret). L'ancienne commande docker-compose (avec tiret, version Python) est dépréciée. On utilise exclusivement la V2 dans ce cours.

// 02

Structure du fichier docker-compose.yml

yaml — anatomie d'un docker-compose.yml
# docker-compose.yml — structure générale

# Pas de "version:" depuis Compose V2 — il est ignoré

services:        # les conteneurs à lancer
  database:
    # configuration du service...

  backend:
    # configuration du service...

  frontend:
    # configuration du service...

networks:        # réseaux à créer
  devcorp-net:
    driver: bridge

volumes:         # volumes à créer
  db-data:
  backend-logs:
ℹ️
YAML — les pièges classiques

YAML est sensible à l'indentation (2 espaces, jamais de tabulations). Les listes commencent par -. Les chaînes avec des caractères spéciaux (:, #) doivent être entre guillemets. En cas de doute, valide avec docker compose config.

// 03

Services — configuration complète

yaml — docker-compose.yml DevCorp
services:

  # ── Base de données ─────────────────────────────────────────
  database:
    image: postgres:15-alpine          # image depuis le registry
    container_name: devcorp-db         # nom fixe du conteneur
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${DB_USER}        # depuis le .env
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    volumes:
      - db-data:/var/lib/postgresql/data
      - ./database/init.sql:/docker-entrypoint-initdb.d/init.sql  # init au démarrage
    networks:
      - devcorp-net
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER} -d ${DB_NAME}"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

  # ── Backend ─────────────────────────────────────────────────
  backend:
    build:
      context: ./backend              # dossier contenant le Dockerfile
      dockerfile: Dockerfile
      args:
        APP_VERSION: ${APP_VERSION:-1.0.0}
    image: devcorp-backend:${APP_VERSION:-1.0.0}
    container_name: devcorp-backend
    restart: unless-stopped
    environment:
      APP_ENV: ${APP_ENV:-production}
      DB_HOST: database                 # nom du service = nom DNS interne
      DB_PORT: "5432"
      DB_USER: ${DB_USER}
      DB_PASSWORD: ${DB_PASSWORD}
      DB_NAME: ${DB_NAME}
    ports:
      - "127.0.0.1:8000:8000"          # bind sur localhost seulement (sécurité)
    depends_on:
      database:
        condition: service_healthy    # attend que la DB soit prête
    networks:
      - devcorp-net
    volumes:
      - backend-logs:/app/logs
    healthcheck:
      test: ["CMD-SHELL", "curl -sf http://localhost:8000/health || exit 1"]
      interval: 15s
      timeout: 5s
      retries: 3

  # ── Frontend ────────────────────────────────────────────────
  frontend:
    build:
      context: ./frontend
    image: devcorp-frontend:${APP_VERSION:-1.0.0}
    container_name: devcorp-frontend
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    depends_on:
      backend:
        condition: service_healthy
    networks:
      - devcorp-net

networks:
  devcorp-net:
    driver: bridge

volumes:
  db-data:
  backend-logs:
// 04

Réseaux & volumes dans Compose

Compose crée automatiquement un réseau par défaut pour ton projet (nommé <projet>_default). Il est recommandé de définir explicitement tes réseaux pour plus de contrôle et de clarté.

yaml — réseaux avancés
networks:
  # Réseau interne — pas d'accès à internet
  backend-net:
    internal: true          # les conteneurs ne peuvent pas sortir sur internet

  # Réseau externe existant (créé en dehors de Compose)
  monitoring-net:
    external: true           # Compose ne le crée pas, il l'utilise

  # Réseau avec configuration IP fixe
  devcorp-net:
    driver: bridge
    ipam:
      config:
        - subnet: 172.20.0.0/16
yaml — volumes avancés
volumes:
  # Volume nommé simple (géré par Docker)
  db-data:

  # Volume avec driver spécifique (NFS, cloud...)
  shared-data:
    driver: local
    driver_opts:
      type: nfs
      o: addr=192.168.1.100,rw
      device: :/exports/data

  # Volume externe (créé en dehors de Compose)
  existing-data:
    external: true
// 05

Variables d'environnement & fichiers .env

Ne jamais mettre de valeurs sensibles en dur dans le docker-compose.yml. Compose charge automatiquement un fichier .env à la racine du projet.

bash — .env (ne pas versionner)
# .env — chargé automatiquement par docker compose
# Ne JAMAIS committer ce fichier

APP_VERSION=1.0.0
APP_ENV=production

DB_USER=devcorp
DB_PASSWORD=changeme_in_prod_please
DB_NAME=devcorp

POSTGRES_HOST_AUTH_METHOD=scram-sha-256
bash — .env.example (à versionner)
# .env.example — template sans valeurs sensibles
# Copier en .env et remplir les valeurs

APP_VERSION=1.0.0
APP_ENV=production

DB_USER=devcorp
DB_PASSWORD=           # à remplir
DB_NAME=devcorp
yaml — syntaxe des variables dans Compose
services:
  backend:
    environment:
      # Passer la variable depuis l'environnement hôte ou .env
      DB_PASSWORD: ${DB_PASSWORD}

      # Avec valeur par défaut si non définie
      APP_ENV: ${APP_ENV:-production}

      # Valeur fixe (ok pour les non-sensibles)
      PYTHONUNBUFFERED: "1"

    # Ou charger depuis un fichier .env spécifique
    env_file:
      - .env
      - .env.local          # surcharge locale (prioritaire)
⚠️
Secrets en production

Les fichiers .env sont pratiques en dev mais insuffisants en production — le fichier peut être lu par quiconque a accès au serveur. En production, on utilisera HashiCorp Vault ou les secrets Kubernetes. C'est l'objet de la Phase 6.

// 06

Healthchecks & depends_on

Le problème classique : le backend démarre avant que la base de données soit prête et plante immédiatement. Sans healthcheck, depends_on attend seulement que le conteneur soit démarré — pas qu'il soit prêt.

SANS healthcheck                      AVEC healthcheck
──────────────────────────────────    ──────────────────────────────────
DB démarre (process lancé)            DB démarre
Backend démarre immédiatement         DB → healthcheck toutes les 10s
Backend tente de connecter à DB       DB → healthy (pg_isready OK)
DB pas encore prête → ERREUR          Backend démarre (condition: healthy)
Backend crash (restart loop)          Backend → connexion DB réussie ✓
yaml — healthchecks complets
services:
  database:
    image: postgres:15-alpine
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER}"]
      interval: 10s      # fréquence des vérifications
      timeout: 5s        # délai max pour une vérification
      retries: 5         # nombre d'échecs avant "unhealthy"
      start_period: 30s  # délai avant le premier check (démarrage lent)

  backend:
    depends_on:
      database:
        condition: service_healthy   # attend healthy, pas juste started
    healthcheck:
      test: ["CMD-SHELL", "curl -sf http://localhost:8000/health"]
      interval: 15s
      timeout: 3s
      retries: 3

  frontend:
    depends_on:
      backend:
        condition: service_healthy
// 07

Profils & surcharge de fichiers

En développement, on veut des outils supplémentaires (Adminer pour voir la DB, logs en mode debug). En production, on veut des réglages plus stricts. Compose gère ça avec les profils et la surcharge de fichiers.

Surcharge avec docker-compose.override.yml

Compose charge automatiquement docker-compose.yml puis docker-compose.override.yml et fusionne les deux. C'est le mécanisme standard pour les différences dev/prod.

yaml — docker-compose.override.yml (dev uniquement)
# docker-compose.override.yml — chargé automatiquement en dev
# Ne pas committer si les valeurs sont sensibles

services:
  backend:
    build:
      target: development           # build stage dev du Dockerfile multi-stage
    volumes:
      - ./backend/app:/app            # bind mount pour hot-reload
    environment:
      APP_ENV: development
    command: uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload

  # Outil d'administration DB — seulement en dev
  adminer:
    image: adminer:latest
    ports:
      - "8080:8080"
    networks:
      - devcorp-net
bash — utilisation selon l'environnement
# Développement (charge compose.yml + override.yml automatiquement)
docker compose up -d

# Production (uniquement le fichier principal)
docker compose -f docker-compose.yml up -d

# Combinaison explicite
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d

Profils — services optionnels

yaml
services:
  backend:
    # pas de profil = toujours démarré

  adminer:
    image: adminer
    profiles: ["debug"]       # démarré seulement avec --profile debug

  mailhog:
    image: mailhog/mailhog
    profiles: ["debug", "mail"]
bash
# Stack normale
docker compose up -d

# Avec les outils de debug
docker compose --profile debug up -d
// 08

Commandes essentielles du quotidien

bash — référence des commandes Compose
# ── Démarrage ──────────────────────────────────────────────
docker compose up -d              # démarrer en arrière-plan
docker compose up -d --build      # forcer la reconstruction des images
docker compose up backend         # démarrer un seul service

# ── Arrêt ──────────────────────────────────────────────────
docker compose stop               # arrêter sans supprimer
docker compose down               # arrêter et supprimer conteneurs + réseau
docker compose down -v            # + supprimer les volumes (DANGER en prod)

# ── Surveillance ───────────────────────────────────────────
docker compose ps                 # état des services
docker compose logs               # logs de tous les services
docker compose logs -f backend    # logs d'un service en follow
docker compose logs --tail=50     # 50 dernières lignes
docker compose top                # processus dans les conteneurs

# ── Exécution ──────────────────────────────────────────────
docker compose exec backend bash  # shell dans un conteneur qui tourne
docker compose run --rm backend pytest  # lancer une commande one-shot

# ── Build ──────────────────────────────────────────────────
docker compose build              # construire toutes les images
docker compose build backend      # construire une seule image
docker compose build --no-cache   # sans cache

# ── Validation ─────────────────────────────────────────────
docker compose config             # valider et afficher la config fusionnée
docker compose config --services  # lister les services définis

# ── Mise à jour d'un service sans tout redémarrer ──────────
docker compose up -d --no-deps --build backend  # rebuild + restart backend seulement
