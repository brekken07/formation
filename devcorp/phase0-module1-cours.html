Bash Scripting

Écrire des scripts robustes, lisibles et prêts pour la production.

📅 Semaine 1 ⏱ ~6h de lecture + pratique 🎯 Phase 0 · Module 1 📦 Projet : DevCorp

// Objectifs du module

  • Maîtriser la structure et les bonnes pratiques d'un script de production
  • Manipuler variables, paramètres, conditions et boucles avec confiance
  • Structurer son code en fonctions réutilisables et bien loggées
  • Gérer les erreurs proprement avec trap et des codes de retour explicites
  • Produire du Bash lisible par n'importe quel ingénieur de ton équipe
// 01

Pourquoi le Bash avant tout le reste ?

Avant même de toucher à Docker ou Ansible, tu vas écrire des scripts. Pas parce que c'est un prérequis théorique — parce que dès la Phase 1, tu vas lire, modifier et déboguer des scripts dans des Dockerfiles, des pipelines CI/CD, des playbooks Ansible. Si le Bash t'est opaque, tu subis les outils au lieu de les maîtriser.

L'objectif de ce module n'est pas de faire de toi un expert en Bash. C'est de te rendre autonome : capable d'écrire un script de 50 lignes propre, robuste, et lisible par quelqu'un d'autre.

💡
Posture formateur

Pour tes futurs apprenants : ancre le Bash dans le quotidien système. "Vous avez déjà tapé des commandes dans un terminal ? Un script, c'est juste une liste de ces commandes que vous sauvegardez pour ne pas les retaper."

// 02

Structure d'un script professionnel

Tout script de prod commence par le même squelette. Pas par habitude — par nécessité.

bash 
#!/bin/bash
# ══════════════════════════════════════════════════════
# Nom        : deploy.sh
# Description: Script de déploiement DevCorp
# Auteur     : Titi
# Date       : 2024-01-01
# Usage      : ./deploy.sh <environnement> [--dry-run]
# ══════════════════════════════════════════════════════
set -euo pipefail

Le shebang #!/bin/bash

Il indique au système quel interpréteur utiliser quand on exécute le fichier directement (./script.sh). Sans lui, le système utilise le shell courant — ce qui peut varier d'une machine à l'autre et créer des comportements imprévisibles.

Le set -euo pipefail — ta ligne la plus importante

Option Comportement sans Comportement avec
-e Le script continue malgré les erreurs Arrêt immédiat à la première erreur
-u Variable non définie = chaîne vide silencieuse Erreur explicite si variable non définie
-o pipefail Un pipe retourne le code de la dernière commande Un pipe échoue si n'importe quelle commande échoue
⚠️
Piège classique

Sans set -e : rm -rf /app/config échoue silencieusement, systemctl restart app s'exécute quand même avec une config vide → prod cassée. Avec set -e : arrêt immédiat, dégât limité.

// 03

Variables

bash 
# ── Variables simples ──────────────────────────────────
APP_NAME="devcorp"       # pas d'espaces autour du = !
PORT=8080              # les nombres sans guillemets
readonly LOG_DIR="/var/log/${APP_NAME}"  # constante

# ── Utilisation ────────────────────────────────────────
echo "App : $APP_NAME"
echo "Logs : ${LOG_DIR}"    # {} recommandés

# Cas où les accolades sont OBLIGATOIRES
filename="rapport"
echo "${filename}_final.pdf"  # ✓ rapport_final.pdf
echo "$filename_final.pdf"  # ✗ bash cherche $filename_final

# ── Variables spéciales ────────────────────────────────
echo $0   # nom du script
echo $1   # premier argument
echo $#   # nombre d'arguments
echo $@   # tous les arguments
echo $$   # PID du script
echo $?   # code retour de la dernière commande

# ── Substitution de commande ───────────────────────────
current_date=$(date +%Y-%m-%d)
disk_usage=$(df -h / | awk 'NR==2 {print $5}')

# ── Valeurs par défaut ─────────────────────────────────
ENV="${1:-production}"   # défaut si $1 absent
DB_PASS="${DB_PASS:?DB_PASS doit être définie}"  # erreur si vide
// 04

Paramètres & validation

Un script robuste vérifie ce qu'on lui donne avant de faire quoi que ce soit. C'est ce qui distingue un script amateur d'un script de production.

bash 
#!/bin/bash
set -euo pipefail

usage() {
    echo "Usage: $0 <env> <version>"
    echo "  env     : dev | staging | production"
    echo "  version : format X.Y.Z"
    exit 1
}

# Vérifier le nombre d'arguments
if [[ $# -ne 2 ]]; then
    echo "Erreur : $# argument(s) reçu(s), 2 attendus" >&2
    usage
fi

ENV="$1"
VERSION="$2"

# Valider l'environnement
case "$ENV" in
    dev|staging|production)
        echo "Environnement : $ENV" ;;
    *)
        echo "Erreur : '$ENV' invalide" >&2
        usage ;;
esac

# Valider le format de version (regex)
if [[ ! "$VERSION" =~ ^[0-9]+\.[0-9]+\.[0-9]+$ ]]; then
    echo "Erreur : '$VERSION' invalide (format : X.Y.Z)" >&2
    exit 1
fi
💡
Bonne pratique

Toujours rediriger les messages d'erreur vers stderr avec >&2. Ça permet aux outils appelants de séparer la sortie normale des erreurs.

// 05

Conditions & tests

bash 
# ── Tests sur les fichiers ─────────────────────────────
[[ -f "$FILE" ]]   # fichier régulier
[[ -d "$DIR"  ]]   # répertoire
[[ -r "$FILE" ]]   # lisible
[[ -w "$FILE" ]]   # modifiable
[[ -x "$FILE" ]]   # exécutable
[[ -e "$FILE" ]]   # existe
[[ -s "$FILE" ]]   # existe et non vide

# ── Tests sur les chaînes ──────────────────────────────
[[ -z "$STR"      ]]  # vide (zero length)
[[ -n "$STR"      ]]  # non vide
[[ "$STR" == "x" ]]  # égalité
[[ "$STR" =~ ^hel ]]  # regex

# ── Tests numériques ───────────────────────────────────
[[ $N -eq 42 ]]  # égal      (equal)
[[ $N -ne  0 ]]  # différent (not equal)
[[ $N -gt 10 ]]  # supérieur (greater than)
[[ $N -lt 100 ]] # inférieur (less than)
[[ $N -ge 42 ]]  # supérieur ou égal

# ── Opérateurs logiques ────────────────────────────────
[[ -f "$F" && -r "$F" ]]    # ET
[[ "$E" == "dev" || "$E" == "staging" ]]  # OU
[[ ! -d "$DIR" ]]              # NON
ℹ️
[ ] vs [[ ]]

Toujours utiliser [[ ]] (double crochets) en Bash. Plus robuste, supporte les regex avec =~, gère mieux les espaces dans les variables. [ ] est l'ancienne syntaxe POSIX sh — à éviter dans des scripts Bash modernes.

// 06

Boucles

bash 
# ── for sur tableau (pattern DevOps très courant) ──────
SERVICES=("nginx" "postgresql" "redis")

for service in "${SERVICES[@]}"; do
    if systemctl is-active --quiet "$service"; then
        echo "✓ $service actif"
    else
        echo "✗ $service inactif"
    fi
done

# ── Lire un fichier ligne par ligne ────────────────────
while IFS= read -r line; do
    [[ "$line" =~ ^# ]] && continue   # ignorer commentaires
    [[ -z "$line"     ]] && continue   # ignorer lignes vides
    echo "Traitement : $line"
done < "/etc/hosts"

# ── Pattern retry (attendre qu'un service démarre) ─────
MAX=10; RETRY=0
until curl -sf "http://localhost:8080/health" >/dev/null 2>&1; do
    RETRY=$((RETRY + 1))
    if [[ $RETRY -ge $MAX ]]; then
        echo "Erreur : service indisponible" >&2
        exit 1
    fi
    echo "Attente... ($RETRY/$MAX)"
    sleep 5
done
// 07

Fonctions

Les fonctions transforment un script de 100 lignes illisible en un code structuré. Règle absolue : toute variable déclarée dans une fonction doit être local. Sans ça, tu pollues l'espace global et tu crées des bugs impossibles à déboguer.

bash 
# ── Fonctions de logging (réutilise dans tous tes scripts) ─
RED='\033[0;31m'; GREEN='\033[0;32m'
YELLOW='\033[1;33m'; BLUE='\033[0;34m'
NC='\033[0m'

log_info()    { echo -e "${GREEN}[$(date '+%H:%M:%S')] INFO${NC}    $*"; }
log_warning() { echo -e "${YELLOW}[$(date '+%H:%M:%S')] WARNING${NC} $*"; }
log_error()   { echo -e "${RED}[$(date '+%H:%M:%S')] ERROR${NC}   $*" >&2; }
log_step()    { echo -e "${BLUE}[$(date '+%H:%M:%S')] ───▶${NC}    $*"; }

# ── Fonction avec paramètres ───────────────────────────────
check_service() {
    local service_name="$1"   # TOUJOURS local

    if systemctl is-active --quiet "$service_name" 2>/dev/null; then
        return 0
    else
        return 1
    fi
}

# ── Retourner une valeur textuelle ─────────────────────────
get_disk_usage() {
    local mount="${1:-/}"
    local usage
    usage=$(df "$mount" | awk 'NR==2 {gsub(/%/,""); print $5}')
    echo "$usage"   # "retourner" une valeur → via echo + $(...)
}

# ── Utilisation ────────────────────────────────────────────
if check_service "ssh"; then
    log_info "SSH actif"
else
    log_error "SSH inactif !"
fi

DISK=$(get_disk_usage "/")
log_info "Disque : ${DISK}%"
// 08

Gestion des erreurs & trap

Le trap garantit qu'un script ne laisse jamais le système dans un état sale, qu'il se termine normalement, sur une erreur, ou sur un CTRL+C.

bash 
#!/bin/bash
set -euo pipefail

TEMP_DIR=""

cleanup() {
    local exit_code=$?
    if [[ -n "$TEMP_DIR" && -d "$TEMP_DIR" ]]; then
        rm -rf "$TEMP_DIR"
        echo "Nettoyage effectué"
    fi
    if [[ $exit_code -ne 0 ]]; then
        echo "Sortie avec erreur : $exit_code" >&2
    fi
    exit $exit_code
}

# Enregistrer AVANT toute opération risquée
trap cleanup EXIT
trap cleanup INT TERM    # CTRL+C et kill

# Même si ça échoue, cleanup() sera appelé
TEMP_DIR=$(mktemp -d)
cp /etc/hosts "$TEMP_DIR/"
# ... traitement ...
echo "Terminé proprement."
⚠️
Règle de sécurité

Toujours déclarer TEMP_DIR="" avant le trap. Si la variable n'est pas initialisée et que le script échoue avant le mktemp, la fonction cleanup tente de supprimer un chemin vide — ce qui sur certains systèmes peut tenter rm -rf "" avec des conséquences désastreuses.

// 09

Récapitulatif

Concept À retenir Piège classique
set -euo pipefail Toujours en tête de script, sans exception L'oublier et subir des erreurs silencieuses
Variables Pas d'espaces autour du =, utiliser ${} $filename_ext cherche $filename_ext, pas $filename
Paramètres Toujours valider avant d'agir Ne pas vérifier $# avant d'utiliser $1
Tests Utiliser [[ ]] pas [ ] Comparer des nombres avec == au lieu de -eq
Fonctions Toujours local pour les variables internes Polluer l'espace global sans local
trap Déclarer avant toute opération risquée Initialiser la variable à "" avant le trap
