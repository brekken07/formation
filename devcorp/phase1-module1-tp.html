Cahier de TP — Fondamentaux Docker

4 exercices progressifs pour conteneuriser les composants du projet DevCorp.

⚡ 4 exercices ⏱ ~5h 📦 Dépôt : devcorp/

// Contexte DevCorp

Tu vas conteneuriser les trois composants de l'application DevCorp : un backend Python (FastAPI), un frontend statique Nginx, et une base de données PostgreSQL. À la fin de ce module, chaque composant tourne dans son propre conteneur et peut communiquer avec les autres.

Les fichiers applicatifs sont fournis ci-dessous dans chaque TP — tu n'as pas besoin de coder l'application, seulement de la conteneuriser.

TP 3.1
Installation Docker & exploration
Installation, premiers conteneurs, commandes de base
Débutant

Objectif

Installer Docker sur le VPS, se familiariser avec les commandes de base, et comprendre le cycle de vie d'un conteneur par la pratique.

Étapes

  1. Installer Docker sur le VPS avec le script officiel
  2. Ajouter ton utilisateur au groupe docker et vérifier sans sudo
  3. Lancer docker run hello-world et analyser la sortie — comprendre ce qui s'est passé
  4. Lancer un conteneur Ubuntu interactif, explorer le système de fichiers, puis sortir
  5. Lancer Nginx en arrière-plan sur le port 8080 de l'hôte :
    bash
    docker run -d --name test-nginx -p 8080:80 nginx
curl http://localhost:8080
  6. Consulter les logs, entrer dans le conteneur, puis le supprimer
  7. Lancer un conteneur PostgreSQL :
    bash
    docker run -d \
  --name test-postgres \
  -e POSTGRES_USER=devcorp \
  -e POSTGRES_PASSWORD=secret \
  -e POSTGRES_DB=devcorp \
  -p 5432:5432 \
  postgres:15-alpine

# Tester la connexion
docker exec -it test-postgres psql -U devcorp -c "\l"
  8. Faire le ménage : supprimer tous les conteneurs de test et les images inutiles

Questions de réflexion

  • Que se passe-t-il si tu supprimes le conteneur PostgreSQL et le recrées ? Les données sont-elles là ?
  • Quelle est la différence entre docker stop et docker kill ?
  • Pourquoi EXPOSE 80 dans un Dockerfile ne suffit pas à rendre le port accessible depuis l'hôte ?
✓ Critères de validation
  • docker run hello-world fonctionne sans sudo
  • curl http://localhost:8080 retourne la page Nginx
  • Connexion PostgreSQL réussie avec psql
  • Tu peux expliquer ce que fait -d -p -e -v --name
  • docker ps -a ne montre plus les conteneurs de test
TP 3.2
Dockerfile — Backend DevCorp
Écrire un Dockerfile optimisé pour une app Python FastAPI
Intermédiaire

Contexte

Tu vas créer le Dockerfile du backend DevCorp. L'application est une API Python avec FastAPI. Crée la structure suivante dans le dossier backend/ de ton dépôt :

bash
mkdir -p backend/app
cd backend

Fichiers applicatifs à créer

python — backend/app/main.py
from fastapi import FastAPI
import os

app = FastAPI(title="DevCorp API", version="1.0.0")

@app.get("/")
def root():
    return {"message": "DevCorp Backend", "version": "1.0.0"}

@app.get("/health")
def health():
    return {"status": "ok", "env": os.getenv("APP_ENV", "local")}

@app.get("/info")
def info():
    return {
        "app": "DevCorp",
        "db_host": os.getenv("DB_HOST", "not configured"),
        "environment": os.getenv("APP_ENV", "local")
    }
text — backend/requirements.txt
fastapi==0.110.0
uvicorn[standard]==0.27.1

Ton travail — le Dockerfile

Crée backend/Dockerfile qui respecte ces contraintes :

  • Image de base : python:3.11-slim
  • Utilisateur non-root pour lancer l'application
  • Ordre des instructions optimisé pour le cache (dépendances avant le code)
  • Variables d'environnement : PYTHONUNBUFFERED=1, APP_ENV=production
  • WORKDIR dans /app
  • Le serveur démarre avec uvicorn app.main:app --host 0.0.0.0 --port 8000
  • .dockerignore à créer également

Build et tests

bash
# Construire depuis le dossier backend/
docker build -t devcorp-backend:1.0.0 .

# Lancer et tester
docker run -d \
  --name devcorp-backend \
  -p 8000:8000 \
  -e APP_ENV=development \
  devcorp-backend:1.0.0

# Vérifier
curl http://localhost:8000/
curl http://localhost:8000/health
curl http://localhost:8000/info

# Vérifier que l'utilisateur n'est pas root
docker exec devcorp-backend whoami

Sortie attendue

// curl http://localhost:8000/health
{"status":"ok","env":"development"}
✓ Critères de validation
  • L'image se construit sans erreur
  • Les 3 endpoints répondent correctement
  • docker exec devcorp-backend whoami ne retourne pas root
  • Modifier requirements.txt invalide le bon layer (pas le layer FROM)
  • Modifier app/main.py n'invalide pas le layer des dépendances
  • docker image history devcorp-backend:1.0.0 montre les layers attendus
// Commit attendu
bash
git add backend/Dockerfile backend/.dockerignore backend/requirements.txt backend/app/
git commit -m "feat(docker): add optimized Dockerfile for backend service"
TP 3.3
Réseaux & volumes — PostgreSQL persistant
Réseau custom, volumes nommés, communication inter-conteneurs
Intermédiaire

Objectif

Faire communiquer le backend DevCorp avec une base de données PostgreSQL dans un réseau Docker isolé, avec persistance des données via un volume nommé.

Étapes

  1. Créer un réseau Docker dédié : devcorp-net
  2. Créer un volume nommé : devcorp-db-data
  3. Lancer PostgreSQL sur le réseau avec le volume :
    bash
    docker run -d \
  --name devcorp-db \
  --network devcorp-net \
  -e POSTGRES_USER=devcorp \
  -e POSTGRES_PASSWORD=secret \
  -e POSTGRES_DB=devcorp \
  -v devcorp-db-data:/var/lib/postgresql/data \
  --restart unless-stopped \
  postgres:15-alpine
  4. Relancer le backend sur le même réseau :
    bash
    docker run -d \
  --name devcorp-backend \
  --network devcorp-net \
  -e APP_ENV=production \
  -e DB_HOST=devcorp-db \
  -e DB_PORT=5432 \
  -p 8000:8000 \
  devcorp-backend:1.0.0
  5. Vérifier que le backend atteint la DB par son nom :
    bash
    docker exec devcorp-backend ping -c 2 devcorp-db
curl http://localhost:8000/info
  6. Tester la persistance du volume :
    • Créer une table dans PostgreSQL
    • Supprimer le conteneur devcorp-db
    • Recréer le conteneur avec le même volume
    • Vérifier que la table est toujours là
  7. Inspecter le réseau et le volume :
    bash
    docker network inspect devcorp-net
docker volume inspect devcorp-db-data
⚠️
Ne jamais exposer PostgreSQL publiquement
Dans ce TP, on n'expose pas le port 5432 sur l'hôte (pas de -p 5432:5432 pour la DB). Elle est accessible uniquement depuis le réseau Docker interne. C'est la bonne pratique de sécurité.
✓ Critères de validation
  • Le ping depuis backend vers devcorp-db fonctionne
  • curl http://localhost:8000/info affiche "db_host":"devcorp-db"
  • La table créée dans PostgreSQL survit à la suppression/recréation du conteneur
  • Le port 5432 n'est pas exposé sur l'hôte (docker ps ne montre pas 5432)
  • docker network inspect devcorp-net montre les deux conteneurs connectés
TP 3.4
Stack complète avec frontend — TP de synthèse
Frontend Nginx, multi-conteneurs, script de démarrage
Avancé

Objectif

Ajouter le frontend DevCorp (Nginx servant une page HTML statique) et scripter le démarrage de la stack complète. C'est le précurseur de Docker Compose — tu vas faire à la main ce que Compose automatisera.

Fichiers frontend à créer

html — frontend/index.html
<!DOCTYPE html>
<html lang="fr">
<head>
  <meta charset="UTF-8">
  <title>DevCorp</title>
  <style>
    body { font-family: monospace; background: #0d1117; color: #e6edf3;
           display: flex; align-items: center; justify-content: center;
           height: 100vh; margin: 0; }
    .card { text-align: center; border: 1px solid #30363d;
            padding: 40px; border-radius: 12px; }
    h1 { color: #3fb950; }
    #status { color: #8b949e; margin-top: 20px; font-size: 14px; }
  </style>
</head>
<body>
  <div class="card">
    <h1>DevCorp</h1>
    <p>Application 3-tiers conteneurisée</p>
    <div id="status">Connexion au backend...</div>
  </div>
  <script>
    fetch('/api/health')
      .then(r => r.json())
      .then(d => document.getElementById('status').textContent =
            'Backend : ' + d.status + ' | Env : ' + d.env)
      .catch(() => document.getElementById('status').textContent = 'Backend inaccessible')
  </script>
</body>
</html>
nginx — frontend/nginx.conf
server {
    listen 80;
    root /usr/share/nginx/html;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://devcorp-backend:8000/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
dockerfile — frontend/Dockerfile
FROM nginx:1.25-alpine
COPY nginx.conf /etc/nginx/conf.d/default.conf
COPY index.html /usr/share/nginx/html/index.html
EXPOSE 80

Ton travail — script de stack

Crée un script scripts/stack.sh avec les commandes start, stop et status :

  • ./stack.sh start — crée le réseau et le volume si inexistants, build les images, lance les 3 conteneurs dans le bon ordre (db → backend → frontend)
  • ./stack.sh stop — arrête et supprime les conteneurs (mais garde le volume)
  • ./stack.sh status — affiche l'état des 3 conteneurs et teste les endpoints

Le script doit utiliser les fonctions log_info / log_error et attendre que la DB soit prête avant de lancer le backend (pattern retry du cours Bash).

Test final

bash
./scripts/stack.sh start
# Ouvrir http://<IP_VPS> dans un navigateur
# La page doit afficher "Backend : ok | Env : production"

./scripts/stack.sh status
./scripts/stack.sh stop
💡
Avant-goût de Docker Compose
Tu viens de recréer à la main ce que Docker Compose fait automatiquement. Le module suivant va te montrer comment réduire tout ce script à un fichier docker-compose.yml de 40 lignes.
✓ Critères de validation
  • ./stack.sh start démarre les 3 conteneurs sans erreur
  • La page web est accessible sur le port 80 du VPS
  • Le frontend affiche le statut du backend en temps réel
  • ./stack.sh stop arrête tout proprement
  • ./stack.sh start après un stop repart sans erreur (volume préservé)
  • Les images sont taguées et versionnées
