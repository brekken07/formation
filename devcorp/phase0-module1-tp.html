TP 1.1
Rapport d'informations système
Variables, paramètres, substitution de commande
Débutant

Objectif

Créer un script infos_systeme.sh qui affiche un rapport d'informations système formaté. Le script prend un nom d'environnement en paramètre optionnel (défaut : local).

Sortie attendue

// exemple de sortie
══════════════════════════════════════
  Rapport système — production
  2024-01-15 10:32:07
══════════════════════════════════════

Hostname     : vps-devcorp
Utilisateur  : titi
OS           : Debian GNU/Linux 12 (bookworm)
Kernel       : 6.1.0-17-amd64
Uptime       : up 3 days, 4 hours, 12 minutes
IP publique  : 51.75.XXX.XXX

Contraintes

  • set -euo pipefail obligatoire
  • Toutes les valeurs dynamiques dans des variables
  • Si plus d'un paramètre est fourni → afficher l'usage et quitter avec code 1
  • Tester les 3 cas : sans arg, avec un arg valide, avec deux args (doit échouer)

Commandes à découvrir

bash
hostname                          # nom de la machine
whoami                            # utilisateur courant
uname -r                          # version du kernel
lsb_release -d | cut -f2         # nom de l'OS
uptime -p                         # durée depuis le démarrage
date '+%Y-%m-%d %H:%M:%S'        # date formatée
curl -s ifconfig.me               # IP publique
💡
Astuce
Pour l'IP publique, ajoute un timeout : curl -s --max-time 3 ifconfig.me pour éviter que le script reste bloqué si le réseau est lent.
✓ Critères de validation
  • bash -n infos_systeme.sh sans erreur
  • ./infos_systeme.sh affiche le rapport avec "local"
  • ./infos_systeme.sh production affiche le rapport avec "production"
  • ./infos_systeme.sh a b affiche l'usage et retourne code 1
  • Toutes les valeurs sont correctes (vérifier manuellement)
// Commit attendu
bash
git add infos_systeme.sh
git commit -m "feat(bash): add system info report script"
TP 1.2
Vérification de santé du VPS
Fonctions, conditions, logging coloré
Intermédiaire

Objectif

Créer un script check_vps.sh qui vérifie l'état de santé du VPS et produit un rapport structuré. Ce script sera lancé au démarrage de chaque session de travail.

Ce que le script doit vérifier

  1. Espace disque → alerte si > 80%, erreur si > 95%
  2. RAM disponible → alerte si < 500 Mo libres
  3. Charge CPU (load average) → alerte si load > nombre de CPUs
  4. Services : ssh, cron, ufw
  5. Connectivité réseau → ping vers 8.8.8.8

Sortie attendue

// exemple de sortie
[10:32:15] ───▶  Vérification VPS — production
[10:32:15] INFO    Disque  : 23% utilisé ✓
[10:32:15] INFO    RAM     : 6.2 Go disponibles ✓
[10:32:15] INFO    CPU     : load 0.42 / 4 CPUs ✓
[10:32:15] INFO    ssh     : actif ✓
[10:32:15] INFO    cron    : actif ✓
[10:32:15] WARNING ufw     : inactif ✗
[10:32:15] INFO    Réseau  : connecté ✓
────────────────────────────────────────
[10:32:16] WARNING 1 avertissement(s) — vérifier les points signalés

Commandes utiles

bash
df / | awk 'NR==2 {gsub(/%/,""); print $5}'   # % disque utilisé
free -m | awk '/^Mem/ {print $7}'             # RAM dispo en Mo
cat /proc/loadavg | awk '{print $1}'           # load average 1 min
nproc                                           # nombre de CPUs
ping -c 1 -W 2 8.8.8.8 >/dev/null 2>&1         # test réseau
systemctl is-active --quiet ssh                 # service actif ?

Contraintes

  • Utiliser les fonctions log_info, log_warning, log_error du cours
  • Créer une fonction check_service() réutilisable
  • Compter les warnings et les erreurs, afficher le bilan en fin
  • Code de sortie : 0 = tout OK, 1 = erreurs, 2 = seulement warnings
⚠️
Point d'attention
La comparaison du load average nécessite de convertir le flottant (ex: 0.42) en entier pour la comparer avec nproc. Utilise LC_ALL=C awk '{print int($1+0.5)}' pour arrondir proprement.
✓ Critères de validation
  • Le script retourne 0 quand tout va bien
  • Le script retourne 2 si un service est inactif (warning)
  • Les couleurs s'affichent correctement dans le terminal
  • La fonction check_service est réutilisable (testée avec 3 services différents)
  • Le comptage des warnings/erreurs est correct
// Commit attendu
bash
git add check_vps.sh
git commit -m "feat(bash): add VPS health check script"
TP 1.3
Script de sauvegarde
Boucles, manipulation de fichiers, trap
Intermédiaire

Objectif

Créer un script backup.sh qui sauvegarde un ou plusieurs répertoires en archives compressées horodatées, avec rotation automatique.

Usage : ./backup.sh <source1> [source2] [source3]

Ce que le script doit faire

  1. Vérifier que chaque répertoire source existe
  2. Créer /var/backups/devcorp/ si inexistant
  3. Pour chaque source → archive <nom>_YYYYMMDD_HHMMSS.tar.gz
  4. Vérifier que chaque archive n'est pas vide (-s)
  5. Afficher la taille de chaque archive créée
  6. Supprimer les archives de plus de 7 jours
  7. Nettoyer les archives partielles si interruption (trap)

Sortie attendue

// exemple de sortie
[11:05:22] ───▶  Sauvegarde de 2 répertoire(s)
[11:05:22] INFO    Sauvegarde de /etc/ssh...
[11:05:22] INFO    Archive : ssh_20240115_110522.tar.gz (4.2K)
[11:05:22] INFO    Sauvegarde de /home/titi/tp-bash...
[11:05:23] INFO    Archive : tp-bash_20240115_110522.tar.gz (18K)
[11:05:23] INFO    Rotation : 0 archive(s) supprimée(s) (> 7 jours)
[11:05:23] INFO    Sauvegarde terminée avec succès

Commandes utiles

bash
tar -czf archive.tar.gz /path/to/dir   # créer archive
du -sh archive.tar.gz                  # taille lisible
find /dir -name "*.tar.gz" -mtime +7  # fichiers > 7 jours
find /dir -name "*.tar.gz" -mtime +7 -delete  # supprimer
basename /path/to/dir                  # nom du dossier seul
mktemp -d                              # dossier temporaire unique
$EUID                                  # UID de l'utilisateur courant (0 = root)

Contraintes

  • Le script ne doit pas s'exécuter en root (vérifier $EUID)
  • Utiliser mktemp pour les fichiers temporaires intermédiaires
  • trap cleanup EXIT INT TERM pour nettoyer en cas d'interruption
  • Ne pas écraser une archive existante (timestamp garantit l'unicité)
✓ Critères de validation
  • ./backup.sh /etc/ssh /home/titi crée 2 archives correctement
  • ./backup.sh /dossier/inexistant échoue avec un message clair
  • Lancer le script en root → refus avec message explicite
  • CTRL+C pendant l'exécution → pas d'archive partielle laissée
  • Les archives sont bien dans /var/backups/devcorp/
// Commit attendu
bash
git add backup.sh
git commit -m "feat(bash): add backup script with rotation and trap"
TP 1.4
Déploiement simulé — TP de synthèse
Assemblage de tous les concepts · Projet DevCorp
Avancé

Contexte

Tu simules le déploiement de l'application DevCorp. Pas encore de vraie app — les étapes sont simulées avec des echo et des sleep. Ce script est la base de ce qui deviendra ton vrai pipeline de déploiement dans les phases suivantes.

Usage : ./deploy.sh <environnement> <version>

Étapes du déploiement

  1. Validation des paramètres — env (dev/staging/production) + version (X.Y.Z)
  2. Vérifications pré-déploiement — appel de check_vps.sh, présence de Git, réseau
  3. Simulation du déploiement — 4 étapes avec sleep et messages de log
  4. Health check — simulé (succès ou échec aléatoire avec $RANDOM)
  5. Rollback — si health check échoue, simuler un rollback et exit 1
  6. Rapport final — durée, env, version, statut + log dans un fichier

Sortie attendue (cas succès)

// exemple de sortie

[14:22:10] INFO    Pré-vérifications...
[14:22:11] INFO    VPS OK

Points techniques clés

bash
# Mesurer la durée
START_TIME=$SECONDS
# ... traitements ...
DURATION=$((SECONDS - START_TIME))

# Health check aléatoire (pour simuler succès/échec)
if [[ $((RANDOM % 5)) -eq 0 ]]; then
    # 1 chance sur 5 d'échouer
    log_error "Health check échoué — rollback..."
    exit 1
fi

# Logger dans un fichier ET afficher en même temps
LOG_FILE="/tmp/deploy_${ENV}_$(date +%Y%m%d).log"
log_info "Message" | tee -a "$LOG_FILE"

# Vérifier qu'une commande est disponible
if ! command -v git >/dev/null 2>&1; then
    log_error "Git non installé"
    exit 1
fi
ℹ️
Conseil d'architecture
Organise ton script en sections claires avec des fonctions : validate_params(), pre_checks(), deploy(), health_check(), rollback(), final_report(). Le main() appelle chaque fonction dans l'ordre. C'est le pattern qu'on retrouve dans tous les scripts de prod sérieux.
✓ Critères de validation
  • ./deploy.sh production 1.0.0 — déploiement complet avec succès
  • ./deploy.sh dev — erreur propre (paramètre manquant)
  • ./deploy.sh production 1.0 — erreur propre (format version invalide)
  • ./deploy.sh invalid 1.0.0 — erreur propre (env invalide)
  • Le fichier de log est créé et contient les mêmes lignes que la sortie terminal
  • La durée affichée est correcte
  • Le rollback est simulé si le health check échoue (tester plusieurs fois)
