// 01

Pourquoi Docker ? Le problème qu'il résout

Avant Docker, déployer une application ressemblait à ça : "ça marche sur ma machine" — et personne ne sait pourquoi ça ne marche pas en prod. Les différences de versions de Python, de bibliothèques système, de variables d'environnement entre le poste du développeur et le serveur de production étaient une source permanente de bugs.

SANS DOCKER                          AVEC DOCKER
──────────────────────────────────   ────────────────────────────────────
Dev machine  : Python 3.9            Dev machine  : Docker Engine
              numpy 1.21                           ┌─────────────────┐
              Ubuntu 20.04                         │   Container     │
                  ↓                                │   Python 3.11   │
              "ça marche !"                        │   numpy 1.24    │
                                                   │   Ubuntu 22.04  │
Prod server  : Python 3.7            Prod server  : Docker Engine
              numpy 1.18                           └─────────────────┘
              CentOS 7                             même container = même résultat
                  ↓
              "ça plante !"

Docker résout ce problème en empaquetant l'application avec tout son environnement d'exécution dans un conteneur. Le conteneur est identique en dev, en staging et en prod.

ℹ️
Conteneur ≠ Machine virtuelle

Une VM embarque un OS complet (plusieurs Go). Un conteneur partage le kernel de l'hôte et n'embarque que ce dont l'application a besoin (quelques dizaines de Mo). Démarrage en millisecondes contre minutes pour une VM.

💡
Posture formateur

L'analogie qui fonctionne le mieux : un conteneur c'est comme un container maritime — peu importe ce qu'il y a dedans, le bateau (Docker Engine) sait comment le transporter. La standardisation du container permet de le mettre sur n'importe quel bateau.

Installation & premiers pas

bash — installation sur Debian/Ubuntu
# Méthode officielle recommandée
curl -fsSL https://get.docker.com | sh

# Ajouter son user au groupe docker (évite le sudo)
sudo usermod -aG docker $USER
newgrp docker          # activer sans déconnexion

# Vérifier
docker --version
docker info

# Premier conteneur
docker run hello-world
bash — commandes de base
# Lancer un conteneur interactif Ubuntu
docker run -it ubuntu bash
# -i : garder stdin ouvert (interactif)
# -t : allouer un pseudo-terminal

# Lancer en arrière-plan (detached)
docker run -d --name mon-nginx nginx

# Voir les conteneurs en cours
docker ps

# Voir tous les conteneurs (arrêtés inclus)
docker ps -a

# Logs d'un conteneur
docker logs mon-nginx
docker logs -f mon-nginx      # follow (comme tail -f)

# Exécuter une commande dans un conteneur qui tourne
docker exec -it mon-nginx bash

# Arrêter, démarrer, supprimer
docker stop mon-nginx
docker start mon-nginx
docker rm mon-nginx           # le conteneur doit être arrêté
docker rm -f mon-nginx        # force la suppression
// 04

Images & layers

Une image Docker est une pile de couches (layers) en lecture seule. Chaque instruction du Dockerfile crée une nouvelle couche. Quand tu lances un conteneur, Docker ajoute une couche en lecture/écriture au-dessus.

┌─────────────────────────────────┐  ← Layer 5 : RW (conteneur en cours)
├─────────────────────────────────┤  ← Layer 4 : COPY app/ /app/
├─────────────────────────────────┤  ← Layer 3 : RUN pip install -r requirements.txt
├─────────────────────────────────┤  ← Layer 2 : RUN apt-get update && apt-get install...
├─────────────────────────────────┤  ← Layer 1 : FROM python:3.11-slim
└─────────────────────────────────┘  (Image de base)

L'intérêt des layers : ils sont mis en cache. Si tu modifies uniquement ton code applicatif (Layer 4), Docker ne re-exécute pas l'installation des dépendances (Layer 3) — il réutilise le cache. C'est pourquoi l'ordre des instructions dans le Dockerfile a une importance critique sur les temps de build.

bash — gestion des images
# Lister les images locales
docker images
docker image ls

# Télécharger une image
docker pull python:3.11-slim
docker pull nginx:1.25-alpine

# Inspecter les layers d'une image
docker image history python:3.11-slim

# Supprimer une image
docker rmi nginx:1.25-alpine

# Nettoyage global (images, conteneurs arrêtés, cache)
docker system prune -af
// 05

Le Dockerfile

Le Dockerfile est la recette de ton image. Chaque ligne est une instruction qui produit un layer. L'ordre des instructions détermine l'efficacité du cache.

Instructions essentielles

dockerfile — référence complète
# Image de base — toujours la première instruction
FROM python:3.11-slim

# Métadonnées
LABEL maintainer="titi@capgemini.com"
LABEL version="1.0.0"

# Variables de build (disponibles uniquement pendant le build)
ARG APP_VERSION="1.0.0"

# Variables d'environnement (disponibles dans le conteneur)
ENV APP_HOME=/app \
    PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1

# Répertoire de travail (créé automatiquement s'il n'existe pas)
WORKDIR /app

# Exécuter des commandes shell pendant le build
RUN apt-get update && apt-get install -y --no-install-recommends \
        curl \
        gcc \
    && rm -rf /var/lib/apt/lists/*

# Copier des fichiers depuis le contexte de build
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copier le code applicatif (après les dépendances !)
COPY app/ .

# Utilisateur non-root (bonne pratique de sécurité)
RUN useradd --no-create-home appuser
USER appuser

# Port exposé (documentation uniquement — ne publie pas)
EXPOSE 8000

# Volume pour les données persistantes
VOLUME /app/data

# Commande par défaut au lancement du conteneur
CMD ["python", "app.py"]

# Ou ENTRYPOINT pour un binaire fixe + CMD pour les arguments
# ENTRYPOINT ["python"]
# CMD ["app.py"]

L'ordre des instructions — optimisation du cache

dockerfile — mauvais vs bon ordre
# ✗ MAUVAIS — le code change souvent, invalide le cache des dépendances
FROM python:3.11-slim
COPY . /app                          # moindre modification du code → cache invalidé
RUN pip install -r requirements.txt   # réinstallation complète à chaque fois !

# ✓ BON — dépendances d'abord, code ensuite
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .              # ne change que rarement
RUN pip install -r requirements.txt   # mis en cache jusqu'à ce que requirements.txt change
COPY . .                             # le code change souvent → cache invalidé ici seulement

Construire et tagger une image

bash
# Construire depuis le répertoire courant
docker build -t devcorp-backend:1.0.0 .
# -t : tag (nom:version)
# .  : contexte de build (répertoire courant)

# Construire avec un Dockerfile spécifique
docker build -f Dockerfile.prod -t devcorp-backend:prod .

# Voir les étapes de build avec cache info
docker build --progress=plain -t devcorp-backend:1.0.0 .

# Forcer la reconstruction sans cache
docker build --no-cache -t devcorp-backend:1.0.0 .
⚠️
Le .dockerignore — indispensable

Comme le .gitignore, le .dockerignore évite d'envoyer des fichiers inutiles au démon Docker pendant le build. Sans lui, node_modules, .git, et les logs sont copiés dans l'image — taille et temps de build multipliés.

.dockerignore
.git
.gitignore
node_modules
__pycache__
*.pyc
.env
*.log
dist/
build/
.vscode/
README.md
*.md
// 06

Conteneurs — cycle de vie complet

docker create  →  Created
docker start   →  Running  →  docker stop  →  Stopped  →  docker rm  →  [supprimé]
                      ↑                           |
               docker restart              docker start
                      |
               docker pause  →  Paused  →  docker unpause
bash — cycle de vie complet
# Lancer avec options complètes
docker run \
  -d \                              # detached
  --name devcorp-backend \          # nom du conteneur
  --restart unless-stopped \        # redémarrage automatique
  -p 8000:8000 \                    # port hôte:port conteneur
  -e DATABASE_URL=postgresql://... \ # variable d'environnement
  -e APP_ENV=production \
  -v devcorp-data:/app/data \       # volume nommé
  --memory 512m \                   # limite mémoire
  --cpus 1.0 \                      # limite CPU
  devcorp-backend:1.0.0

# Inspecter un conteneur (configuration complète en JSON)
docker inspect devcorp-backend

# Stats en temps réel
docker stats devcorp-backend

# Copier un fichier depuis/vers un conteneur
docker cp devcorp-backend:/app/config.json ./config-backup.json
docker cp ./config.json devcorp-backend:/app/config.json
💡
--restart unless-stopped

Cette option redémarre le conteneur automatiquement si le serveur reboot ou si l'application plante — sans intervention manuelle. C'est le comportement attendu en production. always redémarre même si tu l'as arrêté manuellement. unless-stopped respecte un arrêt manuel.

// 07

Réseaux Docker

Par défaut, les conteneurs sont isolés du réseau hôte. Docker crée des réseaux virtuels pour permettre la communication entre conteneurs et avec l'extérieur.

Internet  →  Port mapping (-p)  →  Hôte  →  Bridge network  →  Conteneurs

  ┌─────────────────────────────────────────────┐
  │              Réseau bridge (devcorp-net)     │
  │                                             │
  │  ┌──────────┐    ┌──────────┐  ┌─────────┐  │
  │  │ frontend │ →  │ backend  │→ │database │  │
  │  │ :3000    │    │  :8000   │  │  :5432  │  │
  │  └──────────┘    └──────────┘  └─────────┘  │
  │       ↑                                     │
  └───────┼─────────────────────────────────────┘
          │ -p 3000:3000
        Hôte → Internet
bash
# Créer un réseau personnalisé
docker network create devcorp-net

# Lancer des conteneurs sur ce réseau
docker run -d --name backend --network devcorp-net devcorp-backend:1.0.0
docker run -d --name database --network devcorp-net postgres:15

# Sur un réseau personnalisé, les conteneurs se trouvent par leur NOM
# depuis le conteneur backend, on accède à la DB avec : database:5432
# (pas besoin d'IP — Docker gère le DNS interne)

# Lister les réseaux
docker network ls

# Inspecter un réseau (voir les conteneurs connectés)
docker network inspect devcorp-net

# Connecter un conteneur existant à un réseau
docker network connect devcorp-net mon-conteneur
ℹ️
Réseau bridge vs host vs none

bridge (défaut) : réseau privé isolé, conteneurs communiquent entre eux. host : le conteneur partage le réseau de l'hôte directement (performances max, isolation nulle). none : aucune interface réseau (cas très spécifiques).

// 08

Volumes & persistance des données

Un conteneur est éphémère : quand tu le supprimes, toutes ses données disparaissent. Les volumes permettent de persister les données en dehors du cycle de vie du conteneur.

Trois types de stockage :

1. Volume nommé (recommandé)
   docker run -v mon-volume:/app/data ...
   → Géré par Docker, stocké dans /var/lib/docker/volumes/

2. Bind mount (dev seulement)
   docker run -v /chemin/hote:/app/data ...
   → Dossier hôte monté directement, modifs visibles en temps réel

3. tmpfs mount (données temporaires)
   docker run --tmpfs /app/cache ...
   → En mémoire uniquement, jamais persisté
bash
# ── Volumes nommés (production) ───────────────────────────
docker volume create devcorp-db-data

docker run -d \
  --name postgres \
  -e POSTGRES_PASSWORD=secret \
  -v devcorp-db-data:/var/lib/postgresql/data \
  postgres:15

# Les données survivent à la suppression du conteneur
docker rm -f postgres
docker run -d --name postgres -v devcorp-db-data:/var/lib/postgresql/data postgres:15
# → les données sont toujours là

# ── Bind mount (développement) ────────────────────────────
docker run -d \
  --name backend-dev \
  -v $(pwd)/app:/app \    # le code local est monté dans le conteneur
  -p 8000:8000 \
  devcorp-backend:dev
# Modifier app/*.py en local → changement immédiatement visible dans le conteneur

# ── Gestion des volumes ────────────────────────────────────
docker volume ls
docker volume inspect devcorp-db-data
docker volume rm devcorp-db-data    # ATTENTION : supprime les données !
// 09

Registry & partage d'images

Un registry est un dépôt d'images Docker. Docker Hub est le registry public par défaut. Dans un contexte DevOps, on utilise souvent le registry intégré de GitLab (GHCR, GitLab Registry) pour les images privées.

bash — Docker Hub
# Se connecter
docker login

# Tagger une image pour Docker Hub (format: user/image:tag)
docker tag devcorp-backend:1.0.0 tonuser/devcorp-backend:1.0.0
docker tag devcorp-backend:1.0.0 tonuser/devcorp-backend:latest

# Pousser
docker push tonuser/devcorp-backend:1.0.0
docker push tonuser/devcorp-backend:latest

# Récupérer depuis n'importe quelle machine
docker pull tonuser/devcorp-backend:1.0.0
bash — GitHub Container Registry (GHCR)
# Authentification avec un Personal Access Token GitHub
echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin

# Tagger (format: ghcr.io/user/image:tag)
docker tag devcorp-backend:1.0.0 ghcr.io/tonuser/devcorp-backend:1.0.0

# Pousser
docker push ghcr.io/tonuser/devcorp-backend:1.0.0
