Cahier de TP — Git Avancé

4 exercices progressifs pour maîtriser Git dans un contexte d'équipe et de CI/CD.

⚡ 4 exercices ⏱ ~3h 📦 Dépôt : devcorp

// Avant de commencer

Tous les TP se font sur le dépôt devcorp déjà créé. Tu travailles sur ta branche develop comme base, et tu crées des branches de feature pour chaque exercice.

Garde git lg ouvert dans un second terminal pour visualiser l'évolution du graphe en temps réel.

TP 2.1
Configuration & initialisation du dépôt
Config globale, aliases, structure de branches
Débutant

Objectif

Configurer Git de façon professionnelle et mettre en place la structure de branches du projet DevCorp.

Étapes

  1. Appliquer toute la configuration globale du cours (user, aliases, pull.rebase, etc.)
  2. Vérifier la configuration avec git config --global --list
  3. Dans le dépôt devcorp, créer la branche develop si elle n'existe pas
  4. Créer un fichier README.md structuré avec : présentation du projet, stack technique (React, Python/Node, PostgreSQL), et les phases du programme
  5. Committer sur develop avec le bon format de message
  6. Pousser main et develop sur GitHub
  7. Configurer la protection de branche sur GitHub (Settings → Branches) : interdire les push directs sur main

Résultat attendu

// git lg
* a1b2c3d (HEAD -> develop, origin/develop) docs: add project README
* e4f5g6h (origin/main, main) chore: initial project structure
✓ Critères de validation
  • git lg affiche le graphe correctement
  • git config --global alias.lg retourne la commande configurée
  • Les branches main et develop existent sur GitHub
  • Le README est lisible et structuré
  • La protection de branche est active sur main
TP 2.2
Workflow de branches GitFlow
Feature branches, merge --no-ff, historique propre
Débutant

Objectif

Simuler le travail de deux développeurs en parallèle sur deux features différentes, puis les intégrer dans develop.

Étapes

  1. Depuis develop, créer la branche feature/project-structure
  2. Y créer l'arborescence du projet DevCorp :
    bash
    mkdir -p frontend backend database scripts
touch frontend/.gitkeep backend/.gitkeep
touch database/.gitkeep scripts/.gitkeep
touch docker-compose.yml .env.example
  3. Faire 2 commits distincts (structure + fichiers de config)
  4. Depuis develop (sans merger feature/project-structure), créer feature/gitignore
  5. Y créer un .gitignore complet (celui du cours)
  6. Committer sur feature/gitignore
  7. Merger feature/gitignore dans develop avec --no-ff
  8. Merger feature/project-structure dans develop avec --no-ff
  9. Observer le graphe avec git lg — les deux features doivent apparaître comme des "bulles" distinctes
  10. Supprimer les deux branches de feature (local + remote)

Résultat attendu dans git lg

// git lg après les deux merges
*   c3d4e5f (HEAD -> develop) merge: project structure for DevCorp
|\
| * b2c3d4e feat(project): add docker-compose and env template
| * a1b2c3d feat(project): add directory structure
|/
*   f6g7h8i merge: gitignore for DevOps project
|\
| * e5f6g7h chore: add comprehensive .gitignore
|/
* d4e5f6g docs: add project README
✓ Critères de validation
  • git lg montre deux "bulles" de merge distinctes
  • Les branches de feature sont supprimées localement et sur GitHub
  • Le .gitignore contient bien les entrées .env et *.key
  • Le fichier .env.example est bien versionné (contrairement à .env)
TP 2.3
Rebase interactif & résolution de conflits
rebase -i, squash, résolution de conflits, reflog
Intermédiaire

Objectif

Pratiquer le rebase interactif pour nettoyer un historique "brouillon", résoudre un conflit provoqué volontairement, et récupérer un commit après un reset.

Partie A — Rebase interactif

  1. Créer la branche feature/scripts-bash depuis develop
  2. Y déplacer les scripts Bash du Module 1 dans le dossier scripts/
  3. Faire volontairement 4 commits "sales" :
    bash
    git commit -m "add check script"          # message trop court
git commit -m "fix typo"                  # trivial
git commit -m "WIP"                       # brouillon
git commit -m "add all bash scripts"      # ok
  4. Nettoyer avec git rebase -i HEAD~4 :
    • Squasher les 3 premiers dans le dernier
    • Réécrire le message final au format conventional commit
  5. Vérifier que git lg ne montre plus qu'un seul commit propre

Partie B — Conflit volontaire

  1. Sur develop, modifier la première ligne du README.md et committer
  2. Sur feature/scripts-bash, modifier la même ligne différemment et committer
  3. Tenter de rebaser feature/scripts-bash sur develop → conflit
  4. Résoudre le conflit manuellement, garder les deux contenus fusionnés
  5. Continuer le rebase

Partie C — Récupération avec reflog

  1. Faire un commit quelconque sur feature/scripts-bash
  2. Faire un git reset --hard HEAD~1 (le commit est "perdu")
  3. Retrouver le hash du commit perdu avec git reflog
  4. Récupérer le commit et le réappliquer
⚠️
Important
Ne jamais faire de rebase interactif sur des commits déjà poussés sur une branche partagée. Dans ce TP, on travaille exclusivement sur une branche locale avant de pousser.
✓ Critères de validation
  • Les 4 commits sales ont été réduits à 1 commit au format conventional
  • Le conflit a été résolu sans perdre de contenu
  • Le commit "perdu" a été retrouvé et récupéré via reflog
  • git lg montre un historique propre
// Merger quand tout est propre
bash
git checkout develop
git merge --no-ff feature/scripts-bash -m "merge: bash scripts into project"
git branch -d feature/scripts-bash
TP 2.4
Hooks Git & SSH multi-remote — TP de synthèse
pre-commit, commit-msg, SSH GitHub + GitLab
Intermédiaire

Partie A — Hooks Git

  1. Créer le hook pre-commit du cours dans le dépôt devcorp
  2. Tester qu'il bloque bien un fichier .env stagé :
    bash
    echo "DB_PASS=secret" > .env
git add .env
git commit -m "test"   # doit être bloqué par le hook
  3. Créer le hook commit-msg du cours
  4. Tester qu'il bloque un message non conforme :
    bash
    git commit -m "ajout fichier"                           # bloqué
git commit -m "feat(hooks): add pre-commit and commit-msg"  # OK
  5. Créer le dossier .githooks/ à la racine, y déplacer les hooks, configurer core.hooksPath
  6. Versionner le dossier .githooks/ dans Git

Partie B — SSH multi-remote

  1. Générer deux paires de clés ED25519 (une pour GitHub, une pour GitLab)
  2. Configurer ~/.ssh/config comme dans le cours
  3. Ajouter les clés publiques dans GitHub et GitLab (Settings → SSH Keys)
  4. Tester les deux connexions avec ssh -T
  5. Ajouter le dépôt GitLab comme second remote :
    bash
    git remote add gitlab git@gitlab.com:<user>/devcorp.git
git push gitlab develop
  6. Vérifier que le dépôt est bien mirrored sur GitLab
✓ Critères de validation
  • Le hook pre-commit bloque les fichiers .env
  • Le hook commit-msg bloque les messages non conformes
  • Les hooks sont versionnés dans .githooks/
  • ssh -T git@github.com retourne une authentification réussie
  • ssh -T git@gitlab.com retourne une authentification réussie
  • Le dépôt devcorp est visible sur GitLab avec l'historique complet
// Commits finaux
bash
git add .githooks/
git commit -m "feat(git): add shared pre-commit and commit-msg hooks"
git push origin develop
git push gitlab develop
