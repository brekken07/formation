Cahier de TP — Docker Compose

4 exercices pour passer des commandes manuelles à un fichier déclaratif — et clore la Phase 1.

⚡ 4 exercices ⏱ ~4h 📦 Dépôt : devcorp/

// Contexte

Tu as la stack DevCorp qui tourne avec des commandes docker run manuelles. L'objectif de ces TP est de tout migrer vers Compose, d'ajouter les healthchecks, de séparer les environnements dev et prod, puis de conclure la Phase 1 avec une stack 100 % reproductible en une commande.

Point de départ : le dossier devcorp/ avec backend/, frontend/, database/ et scripts/ déjà en place depuis les TP précédents.

TP 4.1
Découverte de Compose — stack minimale
Premier docker-compose.yml, commandes de base
Débutant

Objectif

Écrire un premier docker-compose.yml simple avec deux services (backend + base de données), sans healthcheck ni configuration avancée. L'objectif est de comprendre la syntaxe de base et les commandes fondamentales.

Étapes

  1. Arrêter tous les conteneurs lancés manuellement dans les TP précédents
  2. À la racine du projet, créer docker-compose.yml avec deux services : database (postgres:15-alpine) et backend (build depuis ./backend)
  3. Configurer les variables d'environnement nécessaires directement dans le fichier (pas de .env encore)
  4. Définir un réseau devcorp-net et un volume db-data
  5. Tester les commandes essentielles :
    bash
    docker compose config          # valider le YAML
docker compose up -d           # démarrer
docker compose ps              # état des services
docker compose logs -f backend # logs en temps réel
docker compose exec backend sh # entrer dans le conteneur
docker compose down            # arrêter et nettoyer
  6. Observer la différence entre down (volume conservé) et down -v (volume supprimé)

Résultat attendu

// docker compose ps
NAME               IMAGE                    COMMAND                  SERVICE    CREATED         STATUS
devcorp-backend    devcorp-backend:1.0.0    "uvicorn app.main..."    backend    2 minutes ago   Up 2 minutes
devcorp-db         postgres:15-alpine       "docker-entrypoint..."   database   2 minutes ago   Up 2 minutes
✓ Critères de validation
  • docker compose config ne retourne aucune erreur
  • Les deux services démarrent avec up -d
  • curl http://localhost:8000/health répond {"status":"ok"}
  • Le backend atteint la DB par son nom de service (ping database depuis le conteneur)
  • Le volume db-data survit à un down mais pas à down -v
TP 4.2
Stack DevCorp complète avec healthchecks
3 services, depends_on conditionnel, fichier .env
Intermédiaire

Objectif

Écrire le fichier Compose de production de DevCorp : les 3 services complets (database, backend, frontend), avec healthchecks sur chacun, dépendances conditionnelles, et toutes les variables sensibles dans un fichier .env.

Étapes

  1. Créer le fichier .env avec toutes les variables du projet (voir cours) et ajouter .env au .gitignore
  2. Créer .env.example avec les clés mais sans valeurs sensibles, et le versionner
  3. Réécrire docker-compose.yml avec :
    • Les 3 services : database, backend, frontend
    • Un healthcheck sur chaque service (pg_isready pour la DB, curl /health pour le backend, curl / pour le frontend)
    • Des depends_on avec condition: service_healthy dans la chaîne
    • Toutes les variables d'environnement depuis ${VAR}
    • Le port 80 exposé uniquement pour le frontend
    • Le backend lié sur 127.0.0.1:8000 (pas accessible de l'extérieur)
  4. Lancer et observer le démarrage ordonné :
    bash
    docker compose up           # sans -d pour voir les logs
# Observer : database healthy → backend démarre → backend healthy → frontend démarre
  5. Vérifier les health statuts :
    bash
    docker compose ps
# STATUS doit afficher "Up X minutes (healthy)" pour chaque service
docker inspect devcorp-backend --format '{{.State.Health.Status}}'

Résultat attendu

// docker compose ps après démarrage complet
NAME                IMAGE                      STATUS
devcorp-db          postgres:15-alpine         Up 3 minutes (healthy)
devcorp-backend     devcorp-backend:1.0.0      Up 2 minutes (healthy)
devcorp-frontend    devcorp-frontend:1.0.0     Up 1 minute  (healthy)
⚠️
start_period pour PostgreSQL
PostgreSQL peut mettre 10 à 30 secondes à accepter des connexions au premier démarrage (initialisation du cluster). Mets start_period: 30s pour éviter que le healthcheck déclare la DB "unhealthy" prématurément.
✓ Critères de validation
  • Les 3 services affichent (healthy) dans docker compose ps
  • La page frontend est accessible sur le port 80
  • Le port 8000 n'est PAS accessible depuis l'extérieur (seulement 127.0.0.1)
  • Aucune valeur sensible n'est en dur dans docker-compose.yml
  • .env est dans le .gitignore, .env.example est versionné
  • Tuer la DB (docker compose stop database) puis la relancer sans recréer backend/frontend
// Commits attendus
bash
git add docker-compose.yml .env.example
git commit -m "feat(compose): add full 3-tier stack with healthchecks"
git push origin develop
TP 4.3
Environnements dev vs prod
Override, bind mount hot-reload, profils, Adminer
Intermédiaire

Objectif

Mettre en place un workflow dev confortable : hot-reload du code sans rebuild d'image, outils de debug additionnels (Adminer pour la DB), et profils Compose pour activer/désactiver des services.

Partie A — Surcharge dev

  1. Créer docker-compose.override.yml qui :
    • Monte le code source ./backend/app dans le conteneur (bind mount)
    • Lance uvicorn avec --reload
    • Passe APP_ENV=development
    • Ajoute le service adminer (image adminer:latest, port 8080) sur le réseau devcorp-net
  2. Tester le hot-reload :
    bash
    docker compose up -d
# Modifier backend/app/main.py — ajouter un endpoint /version
# Observer les logs uvicorn — rechargement automatique
curl http://localhost:8000/version
# Doit répondre sans avoir fait de rebuild
  3. Vérifier Adminer sur http://localhost:8080 — se connecter à la DB DevCorp

Partie B — Lancement production

  1. Lancer sans le fichier override (comme en prod) :
    bash
    docker compose down
docker compose -f docker-compose.yml up -d
# Adminer ne doit pas être présent
# Hot-reload désactivé
docker compose ps   # seulement 3 services, pas adminer
  2. Valider que docker compose config diffère selon le mode (avec ou sans override)

Partie C — Profils

  1. Ajouter un service mailhog (image mailhog/mailhog) avec le profil debug dans l'override
  2. Tester :
    bash
    docker compose up -d                    # mailhog absent
docker compose --profile debug up -d   # mailhog présent
💡
Posture formateur — ce pattern est universel
Le couple docker-compose.yml + docker-compose.override.yml est le standard que tu retrouveras dans tous les projets professionnels. C'est aussi exactement ce que fait GitLab CI quand il lance des services additionnels pendant les pipelines.
✓ Critères de validation
  • Modifier main.py déclenche le hot-reload sans rebuild
  • Adminer est accessible sur :8080 en mode dev, absent en mode prod
  • Le mode prod ne charge pas l'override (vérifier avec docker compose config)
  • Le profil debug démarre mailhog, le mode normal non
// Commit
bash
git add docker-compose.override.yml
git commit -m "feat(compose): add dev override with hot-reload and Adminer"
TP 4.4
TP de synthèse Phase 1 — DevCorp containerisé
Consolidation complète, script de déploiement Compose, vérification finale
Avancé

Objectif

Consolider tout le travail de la Phase 1 en un projet propre, documenté et reproductible. À la fin de ce TP, n'importe qui avec Docker installé doit pouvoir cloner le dépôt et lancer la stack en moins de 5 minutes.

Partie A — Nettoyage et consolidation

  1. Vérifier que tous les Dockerfiles ont un .dockerignore associé
  2. Vérifier que toutes les images utilisent un utilisateur non-root
  3. Vérifier que .env est dans le .gitignore et que .env.example est complet
  4. Tagger les images avec la version 1.0.0 :
    bash
    docker compose build
docker images | grep devcorp  # vérifier les tags

Partie B — Script de déploiement Compose

Créer scripts/deploy.sh qui remplace l'ancien stack.sh du TP Docker. Ce script doit :

  • Vérifier que docker et docker compose sont installés
  • Vérifier que le fichier .env existe (sinon afficher un message d'aide)
  • Faire un docker compose pull si les images ne sont pas à construire localement
  • Lancer docker compose up -d --build
  • Attendre que tous les services soient healthy (boucle de vérification)
  • Afficher un résumé : URLs accessibles, état des services
bash — structure du script
#!/bin/bash
set -euo pipefail

# Fonctions log
log_info()  { echo -e "\033[0;32m[INFO]\033[0m  $*"; }
log_error() { echo -e "\033[0;31m[ERROR]\033[0m $*" >&2; }
log_warn()  { echo -e "\033[0;33m[WARN]\033[0m  $*"; }

# Vérifications prérequis
check_requirements() { ... }

# Attendre que tous les services soient healthy
wait_for_healthy() {
    local max_wait=120
    local waited=0
    while [[ $waited -lt $max_wait ]]; do
        # docker inspect --format ... pour vérifier le health status
        ...
    done
}

# Déploiement
deploy() { ... }

# Résumé
show_summary() { ... }

main() {
    check_requirements
    deploy
    wait_for_healthy
    show_summary
}

main "$@"

Partie C — Documentation

  1. Mettre à jour le README.md avec une section "Démarrage rapide" :
    • Prérequis (Docker Engine 20.10+)
    • Cloner le dépôt
    • Copier .env.example en .env et remplir les valeurs
    • Lancer ./scripts/deploy.sh
    • URLs des services
  2. Ajouter une section "Architecture" avec un schéma ASCII de la stack

Partie D — Test de reproductibilité

Le test final : simuler un déploiement depuis zéro.

bash
# 1. Tout nettoyer (images, conteneurs, volumes)
docker compose down -v
docker rmi devcorp-backend:1.0.0 devcorp-frontend:1.0.0 2>/dev/null || true
docker system prune -f

# 2. Repartir de zéro
./scripts/deploy.sh

# 3. Vérifier que tout fonctionne
curl http://localhost/           # frontend
curl http://localhost:8000/health # backend
docker compose ps                 # tous healthy
ℹ️
Phase 1 terminée — ce que tu as construit
Tu as une application 3-tiers entièrement conteneurisée, avec des Dockerfiles optimisés, des healthchecks, une séparation dev/prod, et un déploiement reproductible en une commande. C'est exactement ce que la Phase 2 (CI/CD GitLab) va automatiser sur un pipeline.
✓ Critères de validation finale Phase 1
  • La stack démarre depuis zéro avec ./scripts/deploy.sh sans intervention manuelle
  • Les 3 services sont healthy après démarrage
  • La page frontend est accessible et affiche le statut du backend
  • Le README permet à quelqu'un de démarrer sans te demander d'aide
  • Aucun secret n'est committé dans Git (git log -p | grep PASSWORD ne retourne rien)
  • Les images utilisent des utilisateurs non-root
  • L'historique Git est propre (conventional commits, branches de feature)
