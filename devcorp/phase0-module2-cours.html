Git Avancé

Maîtriser Git comme un ingénieur DevOps — branches, rebase, hooks et workflows d'équipe.

📅 Semaine 2 ⏱ ~5h 🎯 Phase 0 · Module 2 📦 Projet : DevCorp

// Objectifs du module

  • Configurer Git de façon professionnelle avec aliases et paramètres globaux
  • Maîtriser les workflows de branches : GitFlow et trunk-based development
  • Utiliser le rebase interactif pour maintenir un historique propre
  • Récupérer des erreurs Git sans paniquer grâce au reflog
  • Automatiser des vérifications avec les hooks pre-commit
  • Configurer SSH pour GitHub et GitLab simultanément
// 01

Pourquoi aller plus loin que les bases ?

Tu connais git init, git commit, git push. C'est suffisant pour travailler seul. Mais dans un contexte DevOps, Git est le déclencheur de tout : un push sur main lance un pipeline, déploie en production, notifie une équipe. Un mauvais usage de Git peut casser une prod, effacer le travail d'une équipe, ou exposer un secret en clair dans l'historique public.

Ce module t'apprend à utiliser Git comme un outil de collaboration et d'automatisation — pas seulement comme un système de sauvegarde.

💡
Posture formateur

L'analogie qui fonctionne le mieux pour les débutants : Git c'est comme un système de sauvegarde de jeu vidéo, mais avec des sauvegardes parallèles (branches) qu'on peut fusionner. Le rebase, c'est "rejouer ses actions depuis un autre point de sauvegarde".

// 02

Configuration globale professionnelle

Avant tout, une configuration Git soignée. Ces réglages s'appliquent à tous tes dépôts et t'évitent des erreurs récurrentes.

bash
# ── Identité ───────────────────────────────────────────────
git config --global user.name  "Titi"
git config --global user.email "titi@capgemini.com"

# ── Comportements par défaut ───────────────────────────────
git config --global init.defaultBranch main      # main au lieu de master
git config --global pull.rebase true              # rebase au lieu de merge
git config --global push.autoSetupRemote true     # push sans -u à chaque fois
git config --global core.editor nano              # éditeur de ton choix
git config --global core.autocrlf input           # gestion des fins de ligne

# ── Aliases indispensables ─────────────────────────────────
git config --global alias.lg  "log --oneline --graph --decorate --all"
git config --global alias.st  "status -sb"
git config --global alias.undo "reset HEAD~1 --mixed"
git config --global alias.unstage "reset HEAD --"
git config --global alias.last "log -1 HEAD --stat"

# Vérifier la configuration complète
git config --global --list
ℹ️
git lg — ton meilleur ami

La commande git lg affiche l'historique sous forme de graphe coloré. C'est souvent le moment où les débutants "voient" enfin ce que font les branches. Lance-la régulièrement pendant les TP.

// 03

Branches & GitFlow

Une branche Git est un pointeur léger vers un commit. Créer une branche ne coûte presque rien — c'est l'un des grands avantages de Git sur les anciens systèmes de versioning.

Le modèle GitFlow

GitFlow est le workflow standard en entreprise pour des projets avec des releases planifiées. Il structure le travail en branches avec des rôles définis.

main        ●────────────────────────────────●────  production stable
             \                              /
develop       ●──────────────────────────●─────── intégration continue
               \          \             /
feature/login   ●─●─●      |           |
                            \          |
feature/api                  ●─●─●─●──●
                                         \
hotfix/crash                              ●─●── patch urgent → main

Les branches ont des rôles précis :

  • main — code en production. On ne pousse jamais directement dessus.
  • develop — branche d'intégration. Toutes les features y convergent.
  • feature/* — une branche par fonctionnalité, créée depuis develop.
  • hotfix/* — correction urgente créée depuis main, mergée dans main ET develop.
  • release/* — préparation d'une version (tests finaux, bump de version).

Workflow complet d'une feature

bash
# 1. Toujours partir d'un develop à jour
git checkout develop
git pull origin develop

# 2. Créer la branche de feature
git checkout -b feature/docker-setup

# 3. Travailler et committer régulièrement
#    Convention : feat / fix / docs / chore / refactor / test
git add Dockerfile
git commit -m "feat(docker): add Dockerfile for backend service"
git commit -m "feat(docker): add docker-compose for local dev"
git commit -m "docs: add Docker setup instructions to README"

# 4. Mettre à jour sa branche si develop a avancé
git fetch origin
git rebase origin/develop

# 5. Merger dans develop (--no-ff = garder la trace de la feature)
git checkout develop
git merge --no-ff feature/docker-setup -m "merge: docker setup for DevCorp"
git push origin develop

# 6. Supprimer la branche (local + remote)
git branch -d feature/docker-setup
git push origin --delete feature/docker-setup

Résolution de conflits

Un conflit survient quand deux branches ont modifié la même partie d'un fichier. Git ne choisit pas — il te demande de décider.

bash
# Pendant un merge ou rebase, si conflit :
git status         # voir les fichiers en conflit (both modified)

# Dans le fichier en conflit, Git insère des marqueurs :
# <<<<<<< HEAD          ← ta version actuelle
# ton contenu
# =======
# leur contenu
# >>>>>>> feature/api   ← version entrante

# Éditer le fichier, supprimer les marqueurs, garder ce qu'il faut
nano fichier_en_conflit.txt

# Marquer comme résolu
git add fichier_en_conflit.txt
git rebase --continue    # si tu étais en rebase
# ou
git merge --continue     # si tu étais en merge

# Abandonner en cas de doute
git rebase --abort
git merge --abort
⚠️
Erreur classique

git pull sans configuration fait un merge automatique et crée des commits de merge parasites dans l'historique. Avec pull.rebase true configuré plus haut, git pull fera automatiquement un rebase — historique propre garanti.

// 04

Rebase & historique propre

Le rebase est l'outil qui distingue un historique Git lisible d'un historique chaotique. Il "rejoue" tes commits sur une base différente — comme si tu avais commencé à travailler depuis un point plus récent.

Rebase simple vs rebase interactif

bash
# Rebase simple — mettre à jour sa branche
git rebase origin/develop    # rejoue tes commits sur le dernier develop

# Rebase interactif — réécrire les N derniers commits
git rebase -i HEAD~4         # ouvre un éditeur avec les 4 derniers commits

L'éditeur du rebase interactif ressemble à ça :

pick a1b2c3d feat: add Dockerfile
pick e4f5g6h fix: typo in Dockerfile
pick i7j8k9l feat: add docker-compose
pick m1n2o3p WIP: still working on volumes

# Commandes disponibles :
# pick   = garder tel quel
# reword = garder mais changer le message
# squash = fusionner avec le commit précédent
# fixup  = fusionner (ignorer le message)
# drop   = supprimer ce commit

Exemple concret — nettoyer avant de pousser :

bash
# Avant : 4 commits dont un WIP et un fix de typo
pick a1b2c3d feat: add Dockerfile
squash e4f5g6h fix: typo in Dockerfile   ← fusionné dans le précédent
pick i7j8k9l feat: add docker-compose
drop m1n2o3p WIP: still working          ← supprimé

# Après : 2 commits propres
# a1b2c3d feat: add Dockerfile
# i7j8k9l feat: add docker-compose
⚠️
Règle d'or du rebase

Ne jamais rebaser des commits déjà poussés sur une branche partagée (main, develop). Le rebase réécrit l'historique — sur ta branche locale c'est parfait, sur une branche partagée c'est le chaos pour tes collègues.

// 05

Stash & récupération d'erreurs

Le stash — mettre son travail en pause

bash
# Tu travailles sur quelque chose, une urgence arrive
git stash save "WIP: docker volumes configuration"

# Traiter l'urgence sur une autre branche...

# Lister les stashs
git stash list
# stash@{0}: On feature/docker: WIP: docker volumes configuration

# Reprendre le stash (et le supprimer de la liste)
git stash pop

# Reprendre un stash spécifique sans le supprimer
git stash apply stash@{1}

# Supprimer un stash sans l'appliquer
git stash drop stash@{0}

Récupérer des erreurs — le reflog comme filet de sécurité

bash
# Annuler le dernier commit (garder les modifs en staging)
git reset --soft HEAD~1

# Annuler le dernier commit (garder les modifs non-stagées)
git reset --mixed HEAD~1   # c'est le défaut

# Annuler le dernier commit ET effacer les modifs (ATTENTION)
git reset --hard HEAD~1

# Récupérer un fichier supprimé par erreur
git checkout HEAD -- fichier_supprime.txt

# Le reflog — journal de TOUT ce que Git a fait
git reflog
# abc1234 HEAD@{0}: reset: moving to HEAD~1
# def5678 HEAD@{1}: commit: feat: add volumes config  ← le commit "perdu"
# ghi9012 HEAD@{2}: commit: feat: add Dockerfile

# Récupérer un commit "perdu" après un reset --hard
git checkout def5678      # se placer sur le commit perdu
git checkout -b recovery  # créer une branche pour le sauver
💡
Le reflog sauve des vies

Git ne supprime jamais vraiment un commit immédiatement — il le garde pendant 90 jours par défaut. Le reflog liste tout l'historique de tes actions HEAD, même après un reset --hard. En production, avant de paniquer, lance toujours git reflog en premier.

// 06

Hooks Git

Les hooks sont des scripts exécutés automatiquement par Git avant ou après certaines actions. C'est ta première introduction à l'automatisation dans un workflow DevOps — les pipelines CI/CD reposent sur le même principe.

git commit → pre-commit hook → commit-msg hook → commit créé → post-commit hook
git push   → pre-push hook → push → post-receive hook (côté serveur)
bash
# Les hooks se trouvent dans .git/hooks/
ls .git/hooks/
# pre-commit.sample  commit-msg.sample  pre-push.sample ...
# Supprimer le .sample pour activer un hook

# ── Hook pre-commit : vérifier avant chaque commit ─────────
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/bash
set -euo pipefail

echo "── Vérifications pre-commit ──"

# Bloquer le commit si un fichier .env est stagé
if git diff --cached --name-only | grep -q "\.env$"; then
    echo "ERREUR : tentative de commit d'un fichier .env !"
    echo "Ajoutez .env à votre .gitignore"
    exit 1
fi

# Bloquer si des mots-clés de debug sont présents
if git diff --cached | grep -E "^\+" | grep -qE "(TODO: REMOVE|FIXME: PROD|console\.log\(|pdb\.set_trace)"; then
    echo "AVERTISSEMENT : code de debug détecté dans le diff"
    echo "Vérifiez avant de committer"
    exit 1
fi

# Vérifier la syntaxe des scripts shell modifiés
for file in $(git diff --cached --name-only | grep "\.sh$"); do
    bash -n "$file" || { echo "Syntaxe incorrecte : $file"; exit 1; }
done

echo "── Vérifications OK ──"
EOF
chmod +x .git/hooks/pre-commit
bash
# ── Hook commit-msg : valider le format du message ─────────
cat > .git/hooks/commit-msg << 'EOF'
#!/bin/bash
MSG_FILE="$1"
MSG=$(cat "$MSG_FILE")

# Conventional commits : type(scope): description
PATTERN="^(feat|fix|docs|chore|refactor|test|style|perf|ci|build|revert)(\(.+\))?: .{1,72}$"

if ! echo "$MSG" | grep -qE "$PATTERN"; then
    echo "ERREUR : format de commit invalide"
    echo "Format attendu : type(scope): description"
    echo "Types valides  : feat fix docs chore refactor test style perf ci build revert"
    echo "Exemple        : feat(docker): add Dockerfile for backend"
    exit 1
fi
EOF
chmod +x .git/hooks/commit-msg
ℹ️
Hooks partagés avec l'équipe

Les hooks dans .git/hooks/ ne sont pas versionnés. Pour partager des hooks avec ton équipe, crée un dossier .githooks/ à la racine du projet, versionne-le, et configure Git : git config core.hooksPath .githooks. On le fera dans la Phase 2 avec GitLab CI.

// 07

SSH & multi-remote

Travailler avec GitHub et GitLab simultanément nécessite une configuration SSH soignée. Le fichier ~/.ssh/config permet de router automatiquement la bonne clé vers le bon service.

bash
# Générer une clé par service (ED25519 — plus moderne que RSA)
ssh-keygen -t ed25519 -C "titi@capgemini.com" -f ~/.ssh/id_ed25519_github
ssh-keygen -t ed25519 -C "titi@capgemini.com" -f ~/.ssh/id_ed25519_gitlab

# Afficher la clé publique à copier dans GitHub / GitLab
cat ~/.ssh/id_ed25519_github.pub
cat ~/.ssh/id_ed25519_gitlab.pub

# Configurer le routage automatique
cat > ~/.ssh/config << 'EOF'
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_github
    IdentitiesOnly yes

Host gitlab.com
    HostName gitlab.com
    User git
    IdentityFile ~/.ssh/id_ed25519_gitlab
    IdentitiesOnly yes
EOF
chmod 600 ~/.ssh/config

# Tester les connexions
ssh -T git@github.com    # Hi Titi! You've successfully authenticated...
ssh -T git@gitlab.com    # Welcome to GitLab, @titi!

Travailler avec plusieurs remotes

bash
# Voir les remotes configurés
git remote -v

# Ajouter un second remote (ex: miroir sur GitLab)
git remote add gitlab git@gitlab.com:titi/devcorp.git

# Pousser vers les deux
git push origin main    # GitHub
git push gitlab main    # GitLab

# Ou configurer un push vers plusieurs remotes en même temps
git remote set-url --add --push origin git@gitlab.com:titi/devcorp.git

.gitignore & gestion des secrets

Un .gitignore mal configuré est à l'origine de nombreuses fuites de secrets sur GitHub. Une fois un secret poussé publiquement, il faut considérer qu'il est compromis — même si tu supprimes le fichier dans un commit suivant, il reste dans l'historique.

bash — .gitignore projet DevOps
# ── Secrets — JAMAIS dans Git ──────────────────────────────
.env
.env.local
.env.*.local
*.key
*.pem
*.p12
*_rsa
*_ed25519
secrets/
vault-token
.vault-token

# ── Dépendances (régénérables) ─────────────────────────────
node_modules/
vendor/
__pycache__/
*.pyc
.venv/
venv/

# ── Outils locaux ──────────────────────────────────────────
.vscode/
.idea/
*.swp
.DS_Store
Thumbs.db

# ── Artefacts de build ─────────────────────────────────────
*.log
dist/
build/
target/
*.tar.gz
*.zip
bash
# Supprimer un fichier déjà versionné par erreur
git rm --cached fichier_secret.env   # retire du suivi Git sans supprimer le fichier
git commit -m "chore: remove accidentally tracked secret file"

# Vérifier ce qui sera commité avant de commiter
git diff --cached --name-only

# gitignore global (pour tous tes projets)
git config --global core.excludesfile ~/.gitignore_global
echo ".DS_Store" >> ~/.gitignore_global
echo "*.swp" >> ~/.gitignore_global
