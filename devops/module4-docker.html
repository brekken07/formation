C'est quoi Docker ?

Le problème qu'il résout et les concepts clés

CONCEPT ▼
🎯

Objectif : comprendre pourquoi Docker existe et la différence entre une image et un conteneur — les deux concepts fondamentaux sur lesquels tout repose.

Le problème avant Docker

Tu as sûrement déjà entendu : "Ça marche sur ma machine !". C'est le problème classique :

Développeur A  → Python 3.11, Ubuntu 22   ✓ fonctionne
Développeur B  → Python 3.9, Windows 11   ⚠️ erreur bizarre
Serveur prod   → Python 3.8, CentOS 7     💥 plante complètement

Docker résout ça en embarquant l'application + tout son environnement dans une boîte hermétique — le conteneur. Cette boîte tourne de façon identique partout.

Image vs Conteneur — la distinction clé
Image Docker                   = la recette, le moule (lecture seule)
        ↓ docker run
Conteneur Docker              = l'instance en cours d'exécution (vivant)

Analogie : l'image est comme un fichier .iso — tu peux l'utiliser pour démarrer autant de machines virtuelles (conteneurs) que tu veux. Chaque conteneur est indépendant.

Docker vs Machine Virtuelle
Machine Virtuelle  = OS complet (plusieurs Go, démarre en minutes)
Conteneur Docker   = partage le noyau Linux (quelques Mo, démarre en secondes)
💡

Docker ne virtualise pas le matériel — il isole juste les processus. C'est pour ça qu'un conteneur démarre en moins d'une seconde là où une VM met plusieurs minutes.

Installer Docker
bash — Ubuntu/Debian
# Installation en une commande (script officiel Docker)
curl -fsSL https://get.docker.com | sh

# Ajouter ton utilisateur au groupe docker (évite de taper sudo)
sudo usermod -aG docker $USER

# Déconnecte-toi et reconnecte-toi, puis vérifie
docker --version
docker run hello-world
✅

Si docker run hello-world affiche un message "Hello from Docker!", l'installation est réussie.

💻

Sur Windows/macOS : installe Docker Desktop depuis docker.com — il inclut une interface graphique et tout le nécessaire.

Écrire un Dockerfile

La recette pour construire ton image

PRATIQUE ▼
🎯

Objectif : écrire ton premier Dockerfile pour conteneuriser hello-pipeline. Comprendre chaque instruction — pas juste copier.

C'est quoi un Dockerfile ?

Un Dockerfile est un fichier texte qui décrit comment construire une image Docker. Chaque ligne est une instruction. Docker les exécute dans l'ordre pour créer l'image couche par couche.

Créer le Dockerfile

À la racine de hello-pipeline, crée un fichier nommé exactement Dockerfile (sans extension) :

Dockerfile — hello-pipeline/ (fichier complet)
# FROM : image de base — on part d'une image Python officielle
# "slim" = version allégée (moins de poids, idéale pour la prod)
FROM python:3.11-slim

# WORKDIR : définit le dossier de travail dans le conteneur
# Toutes les commandes suivantes s'exécutent depuis ce dossier
WORKDIR /app

# COPY : copie les fichiers depuis ta machine vers le conteneur
# On copie requirements.txt EN PREMIER (optimisation du cache)
COPY requirements.txt .

# RUN : exécute une commande lors de la construction de l'image
# --no-cache-dir évite de stocker le cache pip dans l'image
RUN pip install --no-cache-dir -r requirements.txt

# COPY : copie maintenant tout le reste du code
COPY . .

# CMD : commande lancée quand on démarre un conteneur
# C'est le point d'entrée de ton application
CMD ["python", "app.py"]
🧱

Pourquoi copier requirements.txt avant le reste du code ? Docker met en cache chaque couche. Si tu modifies app.py, Docker réutilise le cache jusqu'à la couche COPY requirements.txt et ne réinstalle pas les dépendances. Ça accélère considérablement les builds.

Créer le .dockerignore

Comme .gitignore pour Git, .dockerignore liste les fichiers à ne pas copier dans l'image :

.dockerignore — fichier complet
__pycache__/
*.pyc
.pytest_cache/
.venv/
.git/
.github/
*.md
Structure du projet maintenant
hello-pipeline/
  ├── Dockerfile       ← nouveau
  ├── .dockerignore   ← nouveau
  ├── app.py
  ├── conftest.py
  ├── requirements.txt
  ├── .gitignore
  ├── .github/workflows/
  └── tests/test_app.py

Construire et lancer ton conteneur

docker build, docker run — les commandes essentielles

PRATIQUE ▼
🎯

Objectif : construire ton image depuis le Dockerfile et lancer ton premier conteneur. À la fin de cette leçon, ton application Python tourne dans Docker.

Construire l'image
bash — dans hello-pipeline/
# docker build : construit l'image depuis le Dockerfile
# -t : donne un nom (tag) à l'image — ici "hello-pipeline"
# .  : le contexte de build (dossier courant)
docker build -t hello-pipeline .

# Vérifie que l'image apparaît dans la liste
docker images

Tu verras Docker télécharger l'image de base python:3.11-slim puis exécuter chaque instruction du Dockerfile. C'est normal que ça prenne un peu de temps la première fois.

Lancer un conteneur
bash
# docker run : crée et démarre un conteneur depuis l'image
docker run hello-pipeline

# Tu dois voir :
# Hello, DevOps! Pipeline is alive 🚀
# 2 + 3 = 5
Les commandes Docker du quotidien
bash — gestion des conteneurs et images
# Lister les conteneurs en cours d'exécution
docker ps

# Lister tous les conteneurs (y compris arrêtés)
docker ps -a

# Lister les images disponibles
docker images

# Supprimer un conteneur arrêté
docker rm NOM_OU_ID

# Supprimer une image
docker rmi hello-pipeline

# Supprimer tous les conteneurs arrêtés et images inutilisées
docker system prune
Lancer les tests dans le conteneur

On peut aussi exécuter une commande différente au démarrage du conteneur, en écrasant le CMD du Dockerfile :

bash
# Lance pytest au lieu de app.py
docker run hello-pipeline pytest tests/ -v
✅

Si les 5 tests passent dans le conteneur, ton image est valide. L'application fonctionne de façon identique dans Docker et sur ta machine.

🧠 Question
Quelle est la différence entre docker build et docker run ? Que produit chacune de ces commandes ?
✅ Réponse

docker build lit le Dockerfile et produit une image — c'est la recette figée, stockée localement.

docker run prend une image existante et crée une instance en cours d'exécution : le conteneur. Un même docker build peut donner naissance à plusieurs conteneurs via plusieurs docker run.

Docker dans le pipeline CI

Construire et tester l'image automatiquement à chaque push

CI/CD ▼
🎯

Objectif : mettre à jour le workflow CI pour qu'il construise l'image Docker et lance les tests à l'intérieur du conteneur. Plus de dépendance Python locale — tout se passe dans Docker.

Pourquoi tester dans Docker en CI ?

Jusqu'ici, la CI installait Python et les dépendances directement sur la machine virtuelle GitHub. Avec Docker, on teste dans l'environnement exact qui partira en production — c'est plus fiable.

Avant :  VM GitHub → install Python → install deps → pytest
Après :  VM GitHub → docker build → docker run pytest
Mettre à jour ci.yml

Remplace le contenu de .github/workflows/ci.yml par ceci :

yaml — .github/workflows/ci.yml (fichier complet)
name: CI — Docker Build & Test

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
    build-and-test:
        runs-on: ubuntu-latest

        steps:
            - name: Checkout code
                uses: actions/checkout@v4

            # Construit l'image Docker depuis le Dockerfile
            - name: Build Docker image
                run: docker build -t hello-pipeline .

            # Lance le linter dans le conteneur
            - name: Lint with flake8
                run: |
                    docker run hello-pipeline flake8 . --count --select=E9,F63,F7,F82 --show-source --statistics
                    docker run hello-pipeline flake8 . --count --max-line-length=120 --statistics

            # Lance les tests dans le conteneur
            - name: Run tests
                run: docker run hello-pipeline pytest tests/ -v
🧠 Question
Tu viens de modifier ci.yml et le Dockerfile. Quelles commandes Git utilises-tu pour commiter ces deux fichiers ensemble et les pousser sur GitHub ?
✅ Réponse
bash
git add Dockerfile .dockerignore .github/workflows/ci.yml
git commit -m "feat: add Dockerfile and update CI to use Docker"
git push

On peut aussi faire git add . pour tout stager d'un coup, mais lister les fichiers explicitement est une meilleure pratique — ça évite d'embarquer accidentellement des fichiers non souhaités.

👀

Dans l'onglet Actions, tu verras maintenant une étape "Build Docker image" avant les tests. C'est le signe que Docker est bien intégré dans ton pipeline.

Publier l'image sur GitHub Container Registry

Stocker et versionner ton image Docker dans GitHub

CI/CD ▼
🎯

Objectif : créer un second workflow CD qui construit l'image et la publie automatiquement sur GitHub Container Registry (GHCR) à chaque merge sur main. N'importe qui pourra ensuite la télécharger avec docker pull.

C'est quoi un registry ?

Un registry est un serveur de stockage d'images Docker — comme GitHub pour le code, mais pour les images. GitHub en propose un gratuit intégré à ton compte : ghcr.io.

Ton pipeline :
git push → CI : build + test → CD : docker push ghcr.io

N'importe où ensuite :
docker pull ghcr.io/brekken07/hello-pipeline:latest
Créer le workflow de publication

Crée .github/workflows/docker-publish.yml :

yaml — .github/workflows/docker-publish.yml (fichier complet)
name: CD — Publish Docker Image

on:
  push:
    branches: [ main ]

# Permissions pour écrire dans le registry GitHub
permissions:
    contents: read
    packages: write

jobs:
    publish:
        runs-on: ubuntu-latest

        steps:
            - name: Checkout code
                uses: actions/checkout@v4

            # Connexion au registry GitHub (utilise un token automatique)
            - name: Login to GitHub Container Registry
                uses: docker/login-action@v3
                with:
                    registry: ghcr.io
                    username: ${{ github.actor }}
                    password: ${{ secrets.GITHUB_TOKEN }}

            # Construit ET pousse l'image vers ghcr.io
            - name: Build and push image
                uses: docker/build-push-action@v5
                with:
                    context: .
                    push: true
                    # Format : ghcr.io/username/nom-repo:tag
                    tags: ghcr.io/${{ github.repository_owner }}/hello-pipeline:latest
🔑

secrets.GITHUB_TOKEN est un token créé automatiquement par GitHub pour chaque workflow — tu n'as rien à configurer. C'est différent des secrets que tu créerais manuellement.

🧠 Question
Après avoir pushé ce nouveau workflow, comment vérifies-tu que l'image a bien été publiée sur GitHub Container Registry ?
✅ Réponse

Dans ton profil GitHub, clique sur l'onglet Packages — tu dois voir hello-pipeline apparaître. En cliquant dessus, tu verras les versions disponibles et la commande docker pull correspondante.

Tu peux aussi vérifier dans l'onglet Actions que le workflow CD — Publish Docker Image est passé au vert.

Quiz de clôture
🧠 Question 1 / 3
Dans un Dockerfile, pourquoi copie-t-on requirements.txt en premier, avant de copier le reste du code ?
✅ Réponse

Pour exploiter le cache de Docker. Chaque instruction crée une couche mise en cache. Si on copie requirements.txt en premier et qu'on ne le modifie pas, Docker réutilise le cache jusqu'à cette couche et ne réinstalle pas les dépendances à chaque build. Ça fait passer un build de 2 minutes à quelques secondes.

🧠 Question 2 / 3
Tu veux ouvrir un terminal interactif à l'intérieur d'un conteneur hello-pipeline pour déboguer. Quelle commande utilises-tu ?
✅ Réponse
bash
docker run -it hello-pipeline bash

-i garde l'entrée standard ouverte, -t alloue un pseudo-terminal. Ensemble ils permettent une session interactive. Tu te retrouves dans le shell du conteneur, dans le dossier /app.

🧠 Question 3 / 3
Dans le pipeline, le workflow CI et le workflow CD se déclenchent tous les deux sur push: branches: [main]. Quelle est la différence de rôle entre les deux ?
✅ Réponse

CI (build-and-test) : vérifie que le code est correct — il build l'image et lance les tests. Son rôle est de valider. Il tourne aussi sur les Pull Requests.

CD (docker-publish) : publie l'image validée vers le registry. Son rôle est de livrer. Il ne tourne que sur main — uniquement après qu'un code validé y a été mergé.
