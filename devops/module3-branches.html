Ce qu'est vraiment une branche

Pas un dossier — un pointeur. Comprendre le mécanisme interne.

CONCEPT ▼
🎯

Objectif : comprendre ce qu'est une branche en interne — pas juste comment l'utiliser, mais pourquoi elle se comporte comme elle le fait. Ce module répond à toutes les questions que tu t'es posées.

Une branche = un pointeur, pas un dossier

C'est la confusion la plus fréquente. Quand tu crées une branche, Git ne copie aucun fichier et ne crée aucun dossier. Il crée juste un petit fichier texte qui contient l'identifiant d'un commit.

Situation initiale — 3 commits sur main : A ── B ── C ← main ← HEAD

Quand tu fais git checkout -b feature/login, Git crée un nouveau pointeur qui pointe sur le même commit C :

A ── B ── C ← main ↑ feature/login ← HEAD

Ton dossier de travail n'a pas changé d'un bit. La seule chose qui a changé : HEAD pointe maintenant sur feature/login au lieu de main.

HEAD — c'est quoi ?

HEAD est un pointeur spécial qui indique "où tu es en ce moment". C'est comme un curseur dans ton historique Git. Il pointe toujours sur la branche active, qui elle-même pointe sur le dernier commit.

HEAD → feature/login → commit C HEAD est indirect : il pointe sur une branche, pas directement sur un commit
💡

C'est pour ça que git log affiche (HEAD -> feature/login) — ça te dit exactement où tu es : sur la branche feature/login, au commit le plus récent.

Ce qui se passe quand tu commites sur une branche

Tu fais un nouveau commit D sur feature/login :

A ── B ── C ← main ╲ D ← feature/login ← HEAD

main est resté sur C. feature/login a avancé sur D. Les deux branches ont maintenant des histoires différentes — c'est exactement le but.

Vérifie en pratique
bash — dans hello-pipeline
# Visualise graphiquement toutes tes branches et commits
git log --oneline --graph --all

# Exemple de ce que tu verras :
# * a1b2c3d (HEAD -> main, origin/main) dernier commit
# * 9f8e7d6 commit précédent
# * 5c4b3a2 encore avant
👁️

Sur GitHub, tu peux visualiser les branches dans ton repo → Insights → Network. C'est le graphe le plus parlant pour comprendre comment les branches divergent et convergent.

🧠 Question
Tu crées une branche feature/test depuis main sans faire aucun commit dessus. Combien de fichiers sont différents entre les deux branches à ce moment-là ?
✅ Réponse

Zéro fichier différent. Les deux branches pointent sur le même commit — elles sont identiques. Une branche nouvellement créée est juste une copie du pointeur, pas des fichiers. La divergence ne commence qu'au premier commit fait sur l'une des deux branches.

2

origin, remote, local — démystifiés

Pourquoi il y a deux versions de chaque branche

CONCEPT ▼
🎯

Objectif : comprendre la relation entre ton repo local et GitHub, et pourquoi origin/main et main sont deux choses différentes.

Local vs Remote

Quand tu travailles avec Git, tu as en réalité deux copies du repo :

Ta machine (local) : main → commit C ← ce que tu vois dans ton dossier GitHub (remote) : origin/main → commit C ← la copie sur les serveurs GitHub

Tant que tu ne pushes pas, les deux peuvent diverger. C'est pour ça que git status peut afficher "Your branch is ahead of 'origin/main' by 2 commits" — tu as 2 commits locaux que GitHub ne connaît pas encore.

origin — juste un alias

origin est le surnom par défaut donné au serveur distant lors d'un git clone. Ce n'est ni une branche, ni un dossier — c'est juste un nom qui cache une URL.

bash — voir ce que origin veut dire
# Affiche tous les remotes et leurs URLs
git remote -v

# Résultat :
# origin  git@github.com:brekken07/hello-pipeline.git (fetch)
# origin  git@github.com:brekken07/hello-pipeline.git (push)

# Tu pourrais renommer origin en "github" si tu voulais
# git remote rename origin github
# Mais tout le monde utilise "origin" par convention
Le -u dans git push -u origin main

La première fois que tu pushes une nouvelle branche, tu dois dire à Git "cette branche locale correspond à cette branche distante". C'est le rôle de -u (ou --set-upstream) :

Sans -u (première fois) : git push origin feature/login → crée la branche sur GitHub mais Git ne mémorise pas le lien Avec -u (première fois) : git push -u origin feature/login → crée la branche ET mémorise que local/feature/login ↔ origin/feature/login → les prochains git push / git pull fonctionnent sans arguments
💡

Après un git push -u, tu peux faire juste git push et git pull sans préciser la destination — Git sait où aller.

Visualiser local ET remote
bash
# Voir toutes les branches locales ET distantes
git branch -a

# Résultat (exemple) :
#   feature/login          ← locale
# * main                   ← locale active (astérisque)
#   remotes/origin/main    ← distante (GitHub)
#   remotes/origin/feature/login  ← distante (GitHub)
🧠 Question
Tu fais 3 commits en local sur main sans pousser. Que t'affiche git status ? Et quelle commande utilises-tu pour envoyer ces commits sur GitHub ?
✅ Réponse

git status affiche : "Your branch is ahead of 'origin/main' by 3 commits" — ton main local est en avance de 3 commits sur le main de GitHub.

bash
git push
# ou explicitement :
git push origin main
3

Merge vs Rebase

Deux façons de réunir des branches — quand utiliser laquelle

PRATIQUE ▼
🎯

Objectif : comprendre la différence entre merge et rebase visuellement, et savoir lequel utiliser selon la situation. C'est une question qu'on pose souvent en entretien DevOps.

La situation de départ

Tu as travaillé sur feature/login pendant que main avançait de son côté :

A ── B ── C ── D ← main ╲ E ── F ← feature/login ← HEAD

Tu veux intégrer le travail de feature/login dans main. Deux approches possibles :

Option 1 — git merge

Crée un nouveau commit de fusion (merge commit) qui réunit les deux histoires :

A ── B ── C ── D ──────── M ← main ╲ ╱ E ────────────── F ── M = merge commit (conserve tout l'historique, montre quand les branches ont divergé)
bash
# Se placer sur main et merger feature/login
git checkout main
git merge feature/login
Option 2 — git rebase

Rejoue les commits de feature/login par-dessus le dernier commit de main. L'historique devient linéaire :

Avant rebase : A ── B ── C ── D ← main ╲ E ── F ← feature/login Après rebase : A ── B ── C ── D ── E' ── F' ← feature/login E' et F' sont des copies de E et F, repositionnées après D
bash
# Sur feature/login, on se rebase sur main
git checkout feature/login
git rebase main

# Puis on merge (fast-forward, pas de merge commit)
git checkout main
git merge feature/login
Quand utiliser lequel ?
git merge → Pull Requests sur GitHub, préserve l'historique réel git rebase → Mettre à jour ta branche avant de la merger, historique propre Règle d'or : ne jamais rebase une branche déjà pushée et partagée (ça réécrit l'historique et perturbe les autres)
⚠️

rebase réécrit l'historique — il crée de nouveaux commits (E', F'). Ne l'utilise jamais sur une branche que d'autres personnes ont déjà récupérée. Sur ta branche locale perso, c'est parfait.

🧠 Question
Quelle est la différence principale entre un merge et un rebase dans le résultat visible dans git log ?
✅ Réponse

Merge : crée un commit de fusion supplémentaire, l'historique montre clairement les deux branches qui se rejoignent — git log --graph affiche une forme en losange.

Rebase : l'historique est linéaire, comme si la branche avait toujours été développée après main. Plus lisible, mais l'historique réel est perdu.

Le GitHub Flow — workflow professionnel

La méthode utilisée dans les vraies équipes

PRO ▼
🎯

Objectif : mettre en pratique le workflow complet utilisé en entreprise — de la création de la branche jusqu'au merge sur main, en passant par la Pull Request et la CI.

Le GitHub Flow en 5 étapes
1. main est toujours stable et déployable ↓ 2. Crée une branche pour chaque nouvelle fonctionnalité ou correction ↓ 3. Commite et pousse régulièrement sur ta branche ↓ 4. Ouvre une Pull Request → la CI valide automatiquement ↓ 5. Merge sur main → déploiement automatique (CD)
Conventions de nommage des branches

En équipe, tout le monde nomme ses branches selon des préfixes — ça permet de savoir au premier coup d'œil à quoi sert une branche :

— conventions standard —
feature/nom-fonctionnalite   # nouvelle fonctionnalité
fix/nom-bug                  # correction de bug
hotfix/correction-urgente    # correction urgente en prod
docs/mise-a-jour-readme      # documentation
chore/mise-a-jour-deps       # maintenance, dépendances
refactor/renommer-fonctions  # refactoring sans nouvelle feature
Exercice — workflow complet sur hello-pipeline

Mets en pratique le GitHub Flow complet :

bash — workflow complet
# 1. S'assurer d'être à jour sur main
git checkout main
git pull

# 2. Créer une branche avec une convention de nommage
git checkout -b feature/add-divide-function

# 3. Ajouter une fonction divide() dans app.py
#    def divide(a: int, b: int) -> float:
#        """Divise a par b."""
#        if b == 0:
#            raise ValueError("Division par zéro impossible")
#        return a / b

# 4. Ajouter le test dans test_app.py
#    from app import greet, add, multiply, subtract, divide
#
#    def test_divide():
#        assert divide(10, 2) == 5.0
#        assert divide(9, 3) == 3.0
#
#    def test_divide_by_zero():
#        with pytest.raises(ValueError):
#            divide(10, 0)

# 5. Commiter
git add app.py tests/test_app.py
git commit -m "feat: add divide function with zero check"

# 6. Pousser la branche sur GitHub
git push -u origin feature/add-divide-function

# 7. Sur GitHub : ouvrir une Pull Request
#    → la CI se déclenche automatiquement
#    → attendre le feu vert
#    → merger

# 8. Nettoyer après le merge
git checkout main
git pull
git branch -d feature/add-divide-function
git fetch --prune
🧠 Question
Pourquoi fait-on git pull sur main avant de créer une nouvelle branche ? Que risque-t-il de se passer si on ne le fait pas ?
✅ Réponse

Si on crée la branche sans avoir fait git pull, elle va partir d'un commit ancien — pas du dernier état de main. Quand on voudra merger, il y aura potentiellement des conflits avec tous les commits intervenus entre-temps sur main.

Règle : toujours git pull sur main avant de créer une branche.

Situations qui font peur — démystifiées

Branches divergentes, detached HEAD, reset — sans stress

RESCUE ▼
🎯

Objectif : savoir diagnostiquer et résoudre les situations Git qui inquiètent les débutants — dont certaines que tu as déjà vécues dans ce parcours.

Situation 1 — Branches divergentes (tu l'as vécu !)
🎬 Situation

Git refuse de pusher : "Updates were rejected because the remote contains work that you do not have locally"

Ce qui s'est passé : origin/main : A ── B ── C (modifié sur GitHub) local/main : A ── B ── D (modifié en local) Git ne sait pas lequel est "le bon" → il refuse
bash — solution
# Récupère les commits distants et les rejoue sous les tiens
git pull --rebase origin main
# → si conflit : résoudre, git add, git rebase --continue

git push
Situation 2 — Detached HEAD
🎬 Situation

Git affiche : "HEAD detached at a1b2c3d" — ça fait peur mais c'est bénin.

Ça arrive quand tu fais git checkout directement sur un commit au lieu d'une branche. HEAD ne pointe plus sur une branche — il pointe directement sur un commit. Tout commit que tu ferais serait "orphelin" (sans branche pour le référencer).

bash — solution
# Revenir sur une branche existante
git checkout main

# Ou créer une branche depuis cet état (pour garder le travail fait)
git checkout -b ma-branche-de-secours
Situation 3 — Annuler le dernier commit
🎬 Situation

Tu as commité quelque chose qu'il ne fallait pas. Le commit n'est pas encore pushé.

bash
# Annule le commit mais GARDE les modifications dans les fichiers
git reset --soft HEAD~1

# Annule le commit ET remet les fichiers comme avant
git reset --hard HEAD~1

# HEAD~1 = "le commit juste avant HEAD"
# HEAD~2 = "2 commits avant HEAD", etc.
⚠️

--hard supprime définitivement les modifications — utilise-le seulement si tu es sûr de ne pas vouloir les garder. --soft est toujours plus sûr.

Situation 4 — Se tromper de branche
🎬 Situation

Tu as fait des commits sur main alors que tu aurais dû les faire sur une branche.

bash
# 1. Crée la branche qui aurait dû recevoir les commits
git checkout -b feature/oups

# 2. Reviens sur main et annule les commits (soft = garde les fichiers)
git checkout main
git reset --soft HEAD~2   # si tu avais fait 2 commits par erreur

# 3. main est propre, tes modifs sont dans feature/oups
git status  # tes fichiers sont encore là, stagés
🧠 Question
Tu viens de faire git reset --soft HEAD~1. Qu'est-ce qui se passe exactement ? Tes fichiers sont-ils perdus ?
✅ Réponse

Non, aucun fichier n'est perdu. --soft annule uniquement le commit dans l'historique Git, mais remet les modifications dans la zone de staging (comme si tu venais de faire git add sans git commit). Tu peux recommiter avec un meilleur message ou ajouter d'autres fichiers.

SYNTHÈSE ▼

Mémo — toutes les commandes branches
bash — référence complète
── VISUALISER ───────────────────────────────────
git branch                  # branches locales
git branch -a               # locales + distantes
git branch -v               # avec dernier commit
git log --oneline --graph --all  # graphe ASCII

── CRÉER / NAVIGUER ──────────────────────────────
git checkout -b nom         # créer + basculer
git checkout nom            # basculer sur existante
git switch -c nom           # créer + basculer (moderne)
git switch nom              # basculer (moderne)

── POUSSER / SYNCHRONISER ────────────────────────
git push -u origin nom      # première fois
git push                    # fois suivantes
git pull                    # récupérer + merger
git pull --rebase           # récupérer + rebaser
git fetch --prune           # nettoyer les refs distantes

── MERGER / REBASER ──────────────────────────────
git merge nom               # merger dans branche active
git rebase main             # rebaser sur main
git rebase --continue       # après résolution conflit
git rebase --abort          # annuler le rebase
git merge --abort           # annuler le merge

── SUPPRIMER ─────────────────────────────────────
git branch -d nom           # supprimer locale (mergée)
git branch -D nom           # forcer la suppression
git push origin --delete nom # supprimer sur GitHub

── ANNULER ───────────────────────────────────────
git reset --soft HEAD~1     # annuler commit, garder fichiers
git reset --hard HEAD~1     # annuler commit + modifications
git restore --staged fichier # déstaguer un fichier
Quiz de clôture
🧠 Question 1 / 3
Un collègue te dit : "J'ai pushé sur main directement au lieu de passer par une PR, comment on fait pour que ça n'arrive plus ?" Quelle fonctionnalité GitHub utilises-tu ?
✅ Réponse

Les Branch Protection Rules dans GitHub : Settings → Branches → Add rule → main. En cochant "Require a pull request before merging", personne ne peut plus pusher directement sur main — même les admins. C'est ce qu'on a fait à l'étape 7 du Module 1.

🧠 Question 2 / 3
Explique avec tes propres mots pourquoi origin n'est pas une branche. Qu'est-ce que c'est vraiment ?
✅ Réponse

origin est un alias (surnom) qui pointe vers l'URL du repo distant — dans notre cas GitHub. Ce n'est pas une branche, c'est le nom du serveur. On peut le vérifier avec git remote -v. Le nom origin est juste une convention — on pourrait l'appeler autrement, mais tout le monde utilise origin.

🧠 Question 3 / 3
Tu dois choisir entre merge et rebase pour intégrer ta branche feature/dashboard dans main via une Pull Request sur GitHub. Lequel utilises-tu et pourquoi ?
✅ Réponse

Pour une Pull Request sur GitHub, on utilise merge. C'est ce que fait le bouton "Merge pull request" — il crée un merge commit qui documente quand et comment les branches ont été réunies.

Le rebase est utile localement pour mettre à jour sa branche avant d'ouvrir la PR (git rebase main depuis sa branche), mais on ne rebase jamais une branche déjà pushée et partagée.
