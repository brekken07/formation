Le cycle Git au quotidien

Les 5 commandes que tu utiliseras chaque jour

MÉMO ▼
🎯

Objectif : ancrer le cycle de travail Git de base. Comprendre ce qui se passe à chaque étape — pas juste copier des commandes.

Le cycle de base

À chaque session de travail, tu répètes toujours le même cycle :

Fichiers modifiés
  ↓ git add .              ← mise en zone de préparation (staging)
  ↓ git commit -m "..."     ← photo de l'état actuel
  ↓ git push               ← envoi vers GitHub
Les commandes d'état — tes yeux

Avant de faire quoi que ce soit, tu regardes toujours où tu en es :

bash
# Quelle branche ? Quels fichiers modifiés ?
git status

# Historique des commits (q pour quitter)
git log --oneline

# Version compacte avec les branches
git log --oneline --graph --all
Mémo — les 5 commandes du quotidien
CommandeCe qu'elle fait
git statusAffiche l'état de tes fichiers (modifiés, stagés, non suivis)
git add .Prépare tous les fichiers modifiés pour le prochain commit
git add fichier.pyPrépare un seul fichier spécifique
git commit -m "msg"Crée le commit avec un message descriptif
git pushEnvoie les commits locaux vers GitHub
Exercice — manipule sur hello-pipeline

Ouvre ton projet hello-pipeline et exécute ces commandes dans l'ordre :

bash — dans hello-pipeline/
# 1. Regarde l'état actuel
git status

# 2. Regarde l'historique de tes commits
git log --oneline

# 3. Regarde l'historique avec les branches
git log --oneline --graph --all
👀

Tu dois voir tous les commits que tu as faits dans le Module 1. Le dernier doit être sur main avec HEAD pointant dessus. Prends le temps de lire ce que ça affiche — tu vas le voir souvent.

🧠 Question
Quelle commande utilises-tu pour voir l'état de tes fichiers avant de faire un commit ? Et que signifie un fichier affiché en rouge vs en vert ?
✅ Réponse

git status affiche l'état. Rouge = fichier modifié mais pas encore préparé (pas encore dans le staging). Vert = fichier préparé avec git add, prêt à être commité.

2

Corriger une erreur de commit

Tu as commité trop vite — voilà comment t'en sortir

RESCUE ▼
🎯

Objectif : savoir corriger les erreurs les plus fréquentes après un commit. Ça arrive à tout le monde — même aux seniors.

Cas 1 — Mauvais message de commit
🎬 Situation

Tu viens de faire git commit -m "truc" et tu réalises que le message est nul. Le commit n'est pas encore pushé.

bash — modifier le dernier message
# --amend modifie le dernier commit (message + contenu)
git commit --amend -m "feat: nouveau message correct"

# Vérifie que le message a bien changé
git log --oneline
⚠️

--amend ne fonctionne que si le commit n'a pas encore été pushé sur GitHub. Si tu l'as déjà pushé, ne l'amende pas — fais un nouveau commit correctif.

Cas 2 — Fichier oublié dans le commit
🎬 Situation

Tu as commité mais tu as oublié d'inclure un fichier. Le commit n'est pas encore pushé.

bash
# Prépare le fichier oublié
git add fichier_oublie.py

# Ajoute-le au dernier commit sans changer le message
git commit --amend --no-edit
Cas 3 — Annuler le staging d'un fichier
🎬 Situation

Tu as fait git add . mais tu réalises qu'un fichier ne devrait pas être dans ce commit.

bash
# Retire un fichier du staging (sans supprimer tes modifications)
git restore --staged fichier.py

# Vérifie : le fichier repasse en rouge dans git status
git status
Exercice — pratique les 3 cas
bash — sur hello-pipeline
# 1. Ajoute une ligne de commentaire dans app.py
#    (par exemple : # Module 2 - exercice Git)

# 2. Fais un commit avec un mauvais message
git add app.py
git commit -m "oups"

# 3. Corrige le message
git commit --amend -m "docs: add comment in app.py"

# 4. Vérifie le résultat
git log --oneline
🧠 Question
Tu as fait git add . et tu veux retirer tests/test_app.py du staging sans perdre tes modifications. Quelle commande utilises-tu ?
✅ Réponse
bash
git restore --staged tests/test_app.py

Tes modifications dans le fichier sont conservées, il est juste retiré de la zone de staging. Il repassera en rouge dans git status.

3

Gérer les branches

Créer, naviguer, nettoyer — le workflow quotidien

PRATIQUE ▼
🎯

Objectif : être à l'aise avec les branches — les créer, naviguer entre elles, voir leur état, et les supprimer quand elles sont mergées.

Visualiser les branches
bash
# Lister toutes les branches locales (astérisque = branche active)
git branch

# Lister aussi les branches distantes (sur GitHub)
git branch -a

# Voir le dernier commit de chaque branche
git branch -v
Créer et naviguer
CommandeCe qu'elle fait
git checkout -b nomCrée une branche ET bascule dessus (le plus courant)
git checkout nomBascule sur une branche existante
git switch nomMême chose que checkout (syntaxe moderne)
git switch -c nomCrée + bascule (syntaxe moderne de checkout -b)
💡

checkout et switch font la même chose. Tu verras les deux en entreprise — retiens les deux.

Supprimer une branche après merge
bash
# Supprimer une branche locale mergée
git branch -d feature/ma-branche

# Supprimer la branche distante sur GitHub
git push origin --delete feature/ma-branche

# Nettoyer les références aux branches distantes supprimées
git fetch --prune
🧹

Bonne habitude : supprime tes branches après chaque merge. Un repo avec 50 branches mortes devient vite illisible.

Exercice — créer, travailler, nettoyer
bash — sur hello-pipeline
# 1. Crée une branche d'exercice
git checkout -b feature/git-practice

# 2. Ajoute un fichier README.md simple
#    Contenu : # Hello Pipeline
#              Projet d'apprentissage CI/CD

# 3. Commite
git add README.md
git commit -m "docs: add README"

# 4. Reviens sur main
git checkout main

# 5. Merge la branche localement (sans PR cette fois)
git merge feature/git-practice

# 6. Supprime la branche locale devenue inutile
git branch -d feature/git-practice

# 7. Pousse tout sur GitHub
git push
🧠 Question
Tu es sur la branche feature/test et tu veux revenir sur main. Quelle commande utilises-tu ? Et comment vérifies-tu ensuite que tu es bien sur main ?
✅ Réponse
bash
git checkout main
# ou
git switch main

# Vérification
git branch  # l'astérisque doit être sur main
# ou
git status  # affiche "On branch main"
4

Résoudre un conflit de merge

La situation qui fait peur — démystifiée

RESCUE ▼
🎯

Objectif : comprendre pourquoi un conflit apparaît et savoir le résoudre calmement. C'est la chose qui fait le plus peur aux débutants — mais c'est en fait très simple une fois qu'on a compris le principe.

Pourquoi un conflit ?

Un conflit se produit quand deux branches ont modifié la même ligne du même fichier. Git ne sait pas laquelle garder — il te demande de choisir.

main       ●——● modifie ligne 5 de app.py
                ↘
feature/x  ● modifie aussi ligne 5 de app.py
                   ↗
               ⚡ CONFLIT — Git ne peut pas décider
À quoi ressemble un conflit dans un fichier

Quand Git détecte un conflit, il modifie le fichier concerné pour te montrer les deux versions :

— exemple de conflit dans app.py —
<<<<<<< HEAD
    return f"Hello, {name}! Version main 🚀"
=======
    return f"Bonjour, {name}! Version feature 🎉"
>>>>>>> feature/x

<<<<<<< HEAD = ce qui est sur ta branche actuelle
====== = séparateur
>>>>>>> feature/x = ce qui vient de la branche à merger

Comment résoudre — 4 étapes
bash
# 1. Git te dit quels fichiers sont en conflit
git status
#    → "both modified: app.py"

# 2. Ouvre le fichier, supprime les marqueurs <<<<, ====, >>>>
#    et garde uniquement la version que tu veux conserver

# 3. Marque le conflit comme résolu
git add app.py

# 4. Finalise le merge
git commit -m "fix: resolve merge conflict in app.py"
💡

VS Code colore automatiquement les conflits et affiche des boutons cliquables : "Accept Current Change", "Accept Incoming Change", "Accept Both". En pratique, c'est plus confortable que d'éditer à la main.

Exercice — provoquer et résoudre un conflit
bash — simulation complète
# 1. Sur main, modifie la ligne de retour de greet() dans app.py
#    return f"Hello, {name}! Version A 🚀"
git add app.py && git commit -m "feat: version A"

# 2. Crée une branche et modifie la MÊME ligne différemment
git checkout -b feature/conflict-test
#    return f"Hello, {name}! Version B 🎉"
git add app.py && git commit -m "feat: version B"

# 3. Reviens sur main et tente de merger
git checkout main
git merge feature/conflict-test
#    → CONFLICT ! Git t'arrête

# 4. Ouvre app.py, choisis la version que tu veux, supprime les marqueurs

# 5. Résous le conflit
git add app.py
git commit -m "fix: resolve conflict, keep version A"

# 6. Nettoie
git branch -d feature/conflict-test
🧠 Question
Après avoir résolu manuellement un conflit dans un fichier (tu as supprimé les marqueurs et gardé la bonne version), quelle est la séquence de commandes pour finaliser le merge ?
✅ Réponse
bash
# Marquer le fichier comme résolu
git add app.py

# Finaliser le merge avec un commit
git commit -m "fix: resolve merge conflict"

La clé : git add ne sert pas seulement à préparer un commit ordinaire — il sert aussi à dire à Git "j'ai résolu ce conflit, tu peux continuer".

5

Récupérer et synchroniser

fetch, pull, stash — travailler en équipe sans casser

PRO ▼
🎯

Objectif : savoir synchroniser ton travail local avec GitHub, et gérer la situation où tu as des modifications en cours mais tu dois changer de contexte.

fetch vs pull — quelle différence ?
CommandeCe qu'elle fait
git fetchTélécharge les changements distants sans les appliquer. Tu peux les inspecter avant.
git pullTélécharge ET applique les changements. Équivaut à fetch + merge.
git pull --rebaseTélécharge et rejoue tes commits par-dessus. Historique plus propre.
💡

En équipe, prends l'habitude de faire git fetch avant de travailler, pour voir si des collègues ont pushé des changements. Ensuite tu décides si tu veux merger ou rebaser.

git stash — mettre son travail de côté
🎬 Situation fréquente

Tu es en train de modifier des fichiers, mais un collègue te demande de regarder un bug urgent sur une autre branche. Tu ne veux pas commiter un travail inachevé.

bash
# Mets ton travail en cours de côté (dans une "remise")
git stash

# Maintenant tu peux changer de branche
git checkout main

# ... tu fais ce que tu as à faire ...

# Reviens sur ta branche et récupère ton travail
git checkout ma-branche
git stash pop   # récupère et supprime le stash

# Lister les stashs en attente
git stash list
Exercice — synchronisation complète
bash
# Simule le stash : modifie app.py sans commiter
# (ajoute juste un commentaire)

# 1. Vérifie l'état
git status  # → fichier modifié

# 2. Mets en stash
git stash

# 3. Vérifie l'état
git status  # → rien à commiter (le stash a tout rangé)

# 4. Liste les stashs
git stash list  # → stash@{0}: WIP on main: ...

# 5. Récupère ton travail
git stash pop

# 6. Vérifie que ta modification est revenue
git status
🧠 Question
Quelle est la différence entre git fetch et git pull ? Dans quel cas préfères-tu utiliser fetch plutôt que pull ?
✅ Réponse

git fetch récupère les changements distants sans les appliquer — tu peux inspecter ce qui a changé avant de décider. git pull récupère ET applique directement.

On préfère fetch quand on veut voir ce qu'ont fait les collègues avant de merger, pour éviter les mauvaises surprises.

Quiz de clôture — 3 questions
🧠 Question 1 / 3
Tu reprends après le week-end. Avant de faire quoi que ce soit, quelles sont les 2 commandes que tu lances en premier pour savoir où tu en es ?
✅ Réponse
bash
# Voir sur quelle branche tu es et l'état des fichiers
git status

# Voir l'historique et où en sont les branches
git log --oneline --graph --all

C'est le réflexe à avoir systématiquement : status + log avant de toucher quoi que ce soit.

🧠 Question 2 / 3
Tu travailles sur feature/mon-truc et tu reçois un message urgent : "il y a un bug en prod sur main, regarde ça tout de suite !". Tu as des modifications non commitées. Que fais-tu ?
✅ Réponse
bash
# 1. Mettre les modifications de côté
git stash

# 2. Aller sur main pour corriger le bug
git checkout main
git pull  # s'assurer d'avoir la dernière version

# ... corriger le bug, commiter, pusher ...

# 3. Revenir sur sa branche et récupérer son travail
git checkout feature/mon-truc
git stash pop
🧠 Question 3 / 3
Tu viens de merger ta PR sur GitHub. Quelles commandes lances-tu localement pour que ton dossier soit à jour, et pour nettoyer la branche qui n'est plus utile ?
✅ Réponse
bash
# 1. Revenir sur main et récupérer le merge
git checkout main
git pull

# 2. Supprimer la branche locale
git branch -d feature/ma-branche

# 3. Nettoyer les références distantes
git fetch --prune

GitHub propose souvent de supprimer la branche distante directement après le merge via un bouton. Dans ce cas fetch --prune suffit pour nettoyer les références locales.
