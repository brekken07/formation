1

Installer Git et créer ton compte GitHub

Les fondations — sans ça, rien n'est possible

GIT ▼
🎯

Ce que tu vas faire : installer Git sur ta machine, le configurer avec ton nom et email, puis créer un compte GitHub gratuit. À la fin de cette étape, tu seras prêt à travailler.

1 — Installer Git

Ouvre un terminal et vérifie d'abord si Git est déjà installé :

bash — terminal
git --version

Si une version s'affiche (ex: git version 2.43.0), Git est déjà installé, passe à l'étape suivante. Sinon :

bash — Ubuntu / Debian
sudo apt update && sudo apt install git -y
💻

Sur Windows : télécharge "Git for Windows" sur git-scm.com et installe-le avec les options par défaut.
Sur macOS : tape brew install git dans le terminal.

2 — Configurer ton identité Git

Git a besoin de savoir qui tu es pour "signer" chaque modification que tu feras. C'est obligatoire avant de pouvoir créer ton premier commit :

bash — remplace avec tes vraies infos
git config --global user.name "Ton Prénom Nom"
git config --global user.email "ton.email@exemple.com"
git config --global init.defaultBranch main
💡

Utilise exactement le même email que celui que tu vas utiliser pour créer ton compte GitHub. C'est important pour que tes contributions soient bien associées à ton profil.

3 — Créer ton compte GitHub
  • Va sur github.com et crée un compte gratuit
  • Vérifie ton adresse email (GitHub envoie un mail de confirmation)
  • Connecte-toi à ton compte — tu es prêt !
4 — Générer une clé SSH

La clé SSH permet à Git (sur ta machine) de se connecter à GitHub sans te demander un mot de passe à chaque commande. C'est un peu comme une carte d'identité numérique :

bash — génération de la clé
# Génère une paire de clés (publique + privée)
ssh-keygen -t ed25519 -C "ton.email@exemple.com"

# Appuie 3 fois sur Entrée pour valider les options par défaut
# (emplacement du fichier et pas de mot de passe sur la clé)

# Affiche ta clé PUBLIQUE — c'est celle-ci qu'on va donner à GitHub
cat ~/.ssh/id_ed25519.pub

Copie entièrement la ligne qui s'affiche (elle commence par ssh-ed25519), puis dans GitHub : Settings → SSH and GPG keys → New SSH key → colle ta clé et valide.

🔐

Ne confonds pas les deux fichiers créés : id_ed25519 est ta clé privée (ne jamais la partager), id_ed25519.pub est ta clé publique (celle qu'on donne à GitHub).

2

Créer le projet Python et le mettre sur GitHub

Structure des fichiers, premier commit, premier push

GIT ▼
🎯

Ce que tu vas faire : créer une mini-application Python avec ses tests, initialiser un dépôt Git local, puis pousser tout ça sur GitHub. Tu feras ton tout premier commit !

1 — Créer la structure du projet

Dans ton terminal, crée un dossier pour le projet et navigue dedans :

bash
# Crée le dossier principal et le sous-dossier tests
mkdir hello-pipeline
cd hello-pipeline
mkdir tests

Voici la structure complète que tu vas créer dans cette étape :

hello-pipeline/
  ├── app.py          ← l'application principale
  ├── conftest.py     ← fichier vide, aide pytest à trouver app.py
  ├── requirements.txt ← liste des dépendances Python
  ├── .gitignore      ← fichiers que Git doit ignorer
  └── tests/
      └── test_app.py   ← les tests automatiques
2 — Créer app.py

C'est l'application elle-même. Crée le fichier app.py à la racine du projet avec ce contenu exact :

python — app.py (fichier complet)
def greet(name: str) -> str:
    """Retourne un message de bienvenue."""
    if not name:
        raise ValueError("Le nom ne peut pas être vide")
    return f"Hello, {name}! Pipeline is alive 🚀"


def add(a: int, b: int) -> int:
    """Additionne deux nombres."""
    return a + b


def multiply(a: int, b: int) -> int:
    """Multiplie deux nombres."""
    return a * b


if __name__ == "__main__":
    print(greet("DevOps"))
    print(f"2 + 3 = {add(2, 3)}")
📌

Note les 2 lignes vides entre chaque fonction — c'est obligatoire en Python (règle PEP8). Le linter que tu installeras plus tard vérifiera ça automatiquement.

3 — Créer tests/test_app.py

Ce fichier contient les tests automatiques. Chaque fonction test_* vérifie qu'une partie de ton application fonctionne correctement :

python — tests/test_app.py (fichier complet)
import pytest
from app import greet, add, multiply  # on importe les 3 fonctions


def test_greet_normal():
    assert greet("Titi") == "Hello, Titi! Pipeline is alive 🚀"


def test_greet_empty_raises():
    with pytest.raises(ValueError):
        greet("")


def test_add():
    assert add(2, 3) == 5
    assert add(-1, 1) == 0


def test_multiply():
    assert multiply(3, 4) == 12
    assert multiply(0, 99) == 0
4 — Créer conftest.py, requirements.txt et .gitignore

conftest.py — crée ce fichier à la racine, laisse-le complètement vide. Sa seule présence indique à pytest où est la racine du projet, ce qui lui permet de trouver app.py quand il lance les tests.

bash — crée un fichier vide
touch conftest.py

requirements.txt — liste les outils Python nécessaires au projet :

txt — requirements.txt (fichier complet)
pytest==8.1.1
flake8==7.0.0

.gitignore — liste les fichiers/dossiers que Git ne doit pas versionner :

txt — .gitignore (fichier complet)
__pycache__/
*.pyc
.pytest_cache/
.venv/
5 — Initialiser Git et faire le premier commit

Un commit c'est comme une photo de l'état de tes fichiers à un instant T. Git conserve l'historique de toutes ces photos. Voici les commandes dans l'ordre :

bash — dans le dossier hello-pipeline
# 1. Initialise un dépôt Git dans ce dossier
git init

# 2. Affiche l'état de tes fichiers (tous en rouge = pas encore suivis)
git status

# 3. Ajoute tous les fichiers dans la zone de préparation ("staging")
git add .

# 4. Vérifie que tout est bien prêt (tout en vert maintenant)
git status

# 5. Crée le commit avec un message descriptif
git commit -m "feat: initial project structure"
📸

Le message après -m décrit ce que contient le commit. La convention est d'utiliser un préfixe : feat: pour une nouvelle fonctionnalité, fix: pour une correction, ci: pour les pipelines.

6 — Créer le repo GitHub et pousser
  • Sur GitHub, clique sur le bouton vert New (ou le + en haut à droite → New repository)
  • Nom du repository : hello-pipeline
  • Visibilité : Public
  • Ne coche rien dans "Initialize this repository" — tu as déjà du code
  • Clique Create repository

GitHub t'affiche alors les commandes à lancer. Copie et exécute celles-ci (remplace TON_USERNAME par ton vrai nom d'utilisateur GitHub) :

bash
# Lie ton repo local au repo distant GitHub
git remote add origin git@github.com:TON_USERNAME/hello-pipeline.git

# Pousse ton code sur GitHub (-u mémorise la destination pour les prochains push)
git push -u origin main
✅

Rafraîchis la page de ton repo sur GitHub — tu dois voir tes 5 fichiers apparaître. Félicitations, c'est ton premier push !

Premier workflow CI — tests automatiques

GitHub Actions lance les tests à chaque push

CI ▼
🎯

Ce que tu vas faire : créer ton premier fichier de workflow GitHub Actions. À chaque fois que tu feras un git push, GitHub lancera automatiquement les tests Python sur une machine virtuelle dans le cloud.

Comment fonctionne GitHub Actions ?

GitHub lit les fichiers .yml placés dans un dossier précis de ton repo (.github/workflows/). Ces fichiers décrivent des workflows : une suite d'étapes automatiques à exécuter.

📤
git push
→
⚙️
Actions déclenché
→
🐍
Setup Python
→
📦
Install deps
→
🧪
pytest
→
✅
Résultat
1 — Créer le dossier workflows

Ce dossier doit s'appeler exactement .github/workflows (avec le point devant). C'est l'emplacement standard que GitHub reconnaît :

bash — dans hello-pipeline/
mkdir -p .github/workflows

Voici où ce nouveau dossier se situe dans ton projet :

hello-pipeline/
  ├── app.py
  ├── conftest.py
  ├── requirements.txt
  ├── .gitignore
  ├── .github/                ← nouveau
  │   └── workflows/          ← nouveau
  │       └── ci.yml        ← à créer juste après
  └── tests/
      └── test_app.py
2 — Créer le fichier ci.yml

Crée le fichier .github/workflows/ci.yml avec ce contenu exact. Attention au YAML : l'indentation doit être exactement comme ci-dessous (utilise des espaces, jamais des tabulations) :

yaml — .github/workflows/ci.yml (fichier complet)
name: CI — Tests Python

# Ce workflow se déclenche à chaque push sur main
# et à chaque fois qu'une Pull Request cible main
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
    # "test" est le nom du job — il sera visible dans l'onglet Actions
    test:
        # GitHub crée une machine virtuelle Ubuntu temporaire
        runs-on: ubuntu-latest

        steps:
            # Étape 1 : télécharge le code de ton repo sur la VM
            - name: Checkout code
                uses: actions/checkout@v4

            # Étape 2 : installe Python 3.11 sur la VM
            - name: Setup Python
                uses: actions/setup-python@v5
                with:
                    python-version: '3.11'

            # Étape 3 : installe pytest et flake8 (listés dans requirements.txt)
            - name: Install dependencies
                run: pip install -r requirements.txt

            # Étape 4 : lance les tests avec pytest
            - name: Run tests
                run: pytest tests/ -v
⚠️

En YAML, l'indentation compte énormément. Chaque niveau d'indentation représente une hiérarchie. N'utilise que des espaces (jamais la touche Tab). Si quelque chose ne s'aligne pas, le pipeline plantera avec une erreur de parsing.

3 — Pousser le workflow sur GitHub
bash
git add .github/
git commit -m "ci: add automated test workflow"
git push
👀

Sur GitHub, clique sur l'onglet Actions de ton repo. Tu verras le workflow tourner en temps réel avec un cercle orange. Clique dessus pour voir les logs de chaque étape. Quand tout est vert : bravo !

Ajouter le contrôle qualité — Linting

Flake8 analyse ton code avant chaque merge

QUALITÉ ▼
🎯

Ce que tu vas faire : ajouter une étape de linting dans le pipeline CI. Flake8 va analyser ton code Python à chaque push pour détecter les erreurs de style et les bugs potentiels, avant même d'exécuter les tests.

C'est quoi le linting ?

Un linter est un outil qui lit ton code sans l'exécuter, et signale tout ce qui ne respecte pas les standards. En Python, le standard s'appelle PEP8. Par exemple :

  • 2 lignes vides entre chaque fonction (E302)
  • Pas de variables déclarées mais jamais utilisées (F841)
  • Pas d'imports inutiles (F401)

En CI, si flake8 trouve un problème, le pipeline s'arrête et bloque le merge. C'est ton filet de sécurité qualité.

1 — Mettre à jour ci.yml

Tu vas modifier le fichier .github/workflows/ci.yml existant. Voici le fichier complet tel qu'il doit être après modification — remplace tout le contenu :

yaml — .github/workflows/ci.yml (fichier complet)
name: CI — Tests Python

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
    test:
        runs-on: ubuntu-latest

        steps:
            - name: Checkout code
                uses: actions/checkout@v4

            - name: Setup Python
                uses: actions/setup-python@v5
                with:
                    python-version: '3.11'

            - name: Install dependencies
                run: pip install -r requirements.txt

            # NOUVEAU : analyse du style et des erreurs Python
            # Le "|" permet d'écrire plusieurs commandes shell sur plusieurs lignes
            # IMPORTANT : pas de commentaires à l'intérieur du bloc run: |
            - name: Lint with flake8
                run: |
                    flake8 . --count --select=E9,F63,F7,F82 --show-source --statistics
                    flake8 . --count --max-line-length=120 --statistics

            - name: Run tests
                run: pytest tests/ -v
⚠️

À l'intérieur d'un bloc run: |, ne mets jamais de commentaires #. GitHub les interprète comme des commandes shell à exécuter et le pipeline plante. Les commentaires doivent toujours être en dehors du bloc run.

2 — Tester volontairement une erreur
🧪 ERREUR INTENTIONNELLE — objectif pédagogique

Pour comprendre ce que fait le linter, on va volontairement introduire une erreur dans app.py. Ajoute cette ligne n'importe où dans le fichier (par exemple juste avant le if __name__) :

python — ligne à ajouter temporairement dans app.py
variable_inutile = "je ne sers à rien"  # F841 : variable jamais utilisée

Pousse ce changement et observe le pipeline passer en rouge dans l'onglet Actions. Flake8 va signaler l'erreur F841. Ensuite, supprime cette ligne, repousse, et regarde le pipeline repasser au vert.

🧠

Ce cycle rouge → correction → vert est le quotidien d'un DevOps. Le pipeline est ton filet de sécurité : il détecte les problèmes avant qu'ils n'arrivent en production.

Branches et Pull Requests

Le workflow Git professionnel — on commence à être autonome

PRO ▼
🎯

Ce que tu vas faire : travailler sur une branche séparée, créer une Pull Request sur GitHub, et voir la CI valider automatiquement ton code avant la fusion sur main. À partir d'ici, on commence à te poser des questions pour consolider tes acquis.

Pourquoi les branches ?

En équipe, on ne développe jamais directement sur main. Chaque nouvelle fonctionnalité vit sur sa propre branche. Ainsi main reste toujours stable, et le code est validé avant d'y être intégré.

1 — Créer une branche et basculer dessus
bash
# Crée la branche ET bascule dessus en une seule commande
git checkout -b feature/add-subtract-function

# Vérifie que tu es bien sur la nouvelle branche (astérisque = branche active)
git branch
2 — Ajouter une nouvelle fonction

Tu vas ajouter une fonction subtract dans app.py. Voici le fichier complet :

python — app.py (fichier complet)
def greet(name: str) -> str:

        raise ValueError("Le nom ne peut pas être vide")

    """Multiplie deux nombres."""

if __name__ == "__main__":

Déploiement automatique — CD vers GitHub Pages

Publier un rapport de tests à chaque merge sur main

CD ▼
🎯

Ce que tu vas faire : créer un second workflow qui génère un rapport HTML des tests et le déploie automatiquement sur GitHub Pages à chaque push sur main. C'est ton premier vrai déploiement continu !

1 — Activer GitHub Pages
  • Dans ton repo GitHub : Settings → Pages (dans le menu gauche)
  • Sous "Build and deployment" → Source : sélectionne GitHub Actions
  • Sauvegarde
2 — Créer le workflow de déploiement

Crée un nouveau fichier .github/workflows/deploy.yml (ne modifie pas ci.yml) :

yaml — .github/workflows/deploy.yml (fichier complet)
name: CD — Deploy Test Report

# Se déclenche uniquement sur les pushs directs sur main
# (donc après un merge de Pull Request)
on:
  push:
    branches: [ main ]

# Permissions nécessaires pour déployer sur GitHub Pages
permissions:
    contents: read
    pages: write
    id-token: write

jobs:
    deploy:
        runs-on: ubuntu-latest
        environment:
            name: github-pages
            url: ${{ steps.deployment.outputs.page_url }}

        steps:
            - name: Checkout code
                uses: actions/checkout@v4

            - name: Setup Python
                uses: actions/setup-python@v5
                with:
                    python-version: '3.11'

            # Installe pytest + le plugin qui génère un rapport HTML
            - name: Install dependencies
                run: pip install -r requirements.txt pytest-html

            # Lance les tests ET génère le rapport dans le dossier public/
            - name: Run tests and generate HTML report
                run: |
                    mkdir -p public
                    pytest tests/ -v --html=public/index.html --self-contained-html

            # Prépare le dossier public/ comme artifact Pages
            - name: Upload Pages artifact
                uses: actions/upload-pages-artifact@v3
                with:
                    path: ./public

            # Déploie sur GitHub Pages
            - name: Deploy to GitHub Pages
                id: deployment
                uses: actions/deploy-pages@v4
🧠 Question — pratique
Tu viens de créer le fichier deploy.yml. Quelles commandes Git utilises-tu pour l'ajouter au dépôt avec un commit approprié, puis le pousser sur GitHub ?
✅ Réponse
bash
git add .github/workflows/deploy.yml
git commit -m "cd: deploy test report to GitHub Pages"
git push
🌐

Après le déploiement (environ 1-2 minutes), ton rapport sera accessible à l'URL : https://TON_USERNAME.github.io/hello-pipeline/

🧠

Tu viens de réaliser un vrai CD : chaque push sur main déclenche automatiquement un build, des tests, et une publication. C'est exactement ce qui se passe dans les entreprises.

Protéger main — Branch Protection Rules

Impossible de merger sans CI verte — comme en prod

PRO ▼
🎯

Ce que tu vas faire : configurer des règles sur main pour qu'il soit techniquement impossible de merger du code sans CI verte et sans Pull Request. C'est la dernière brique de ton pipeline.

Configurer les règles de protection
  • GitHub → ton repo → Settings → Branches
  • Clique Add branch protection rule
  • Branch name pattern : main
  • Coche Require a pull request before merging
  • Coche Require status checks to pass before merging et sélectionne le job test
  • Coche Require branches to be up to date before merging
  • Clique Create pour sauvegarder
🔒

Désormais, personne ne peut pousser directement sur main sans passer par une PR avec la CI verte. Même toi en tant qu'admin ! C'est la pratique standard dans toutes les équipes professionnelles.

🧠 Question de synthèse
Explique dans tes propres mots la différence entre CI et CD, et donne un exemple concret de chacun tiré de ce TP.
✅ Réponse

CI (Intégration Continue) : automatise la vérification du code à chaque push. Dans ce TP : GitHub Actions lance flake8 et pytest à chaque fois qu'on pousse du code ou qu'on ouvre une Pull Request.


CD (Déploiement Continu) : automatise la livraison du code validé vers un environnement. Dans ce TP : après chaque merge sur main, le workflow deploy.yml génère un rapport HTML et le publie automatiquement sur GitHub Pages sans aucune intervention humaine.
