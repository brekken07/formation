Pourquoi Kubernetes ?

Le problème qu'il résout — depuis Docker jusqu'à K8s

CONCEPT ▼
🎯

Objectif : comprendre pourquoi Docker seul ne suffit pas en production, et quel problème précis Kubernetes résout — avant d'écrire la moindre commande.

Le problème avec Docker en production

Docker résout le problème de l'environnement. Mais en production, d'autres questions se posent :

Avec Docker seul : — Que se passe-t-il si le conteneur plante ? → il reste mort — Comment gérer 10 serveurs avec 50 conteneurs ? → à la main... — Comment passer de 1 à 10 instances sous charge ? → manuellement — Comment déployer sans interruption de service ? → compliqué — Comment répartir le trafic entre les conteneurs ? → outil externe

Kubernetes répond à toutes ces questions automatiquement. C'est un orchestrateur de conteneurs — il décide où, quand et comment lancer tes conteneurs sur un ensemble de machines.

L'analogie — Docker vs Kubernetes
Docker = un cuisiner qui prépare un plat K8s = le chef qui gère toute la brigade, les commandes, les remplacements en cas d'absence, et la salle
Les concepts clés — vue d'ensemble
Cluster K8s = ensemble de machines (nodes) gérées par K8s ├── Node = une machine (VM ou physique) dans le cluster │ └── Pod = la plus petite unité K8s — 1 ou plusieurs conteneurs ├── Deployment = gère le nombre de Pods et les mises à jour └── Service = expose les Pods sur le réseau (point d'entrée stable)
💡

On parle souvent de K8s (raccourci de Kubernetes — 8 lettres entre K et s). C'est le standard de facto pour orchestrer des conteneurs en entreprise. AWS, Azure, GCP proposent tous leur version managée (EKS, AKS, GKE).

🧠 Question
Tu as une application Docker qui tourne en production sur un seul serveur. Le conteneur plante à 3h du matin. Avec Docker seul, que se passe-t-il ? Qu'apporte Kubernetes dans ce cas ?
✅ Réponse

Avec Docker seul, le conteneur reste mort jusqu'à ce que quelqu'un le redémarre manuellement — l'application est hors ligne.

Kubernetes surveille en permanence l'état des Pods. Si un Pod plante, il en relance automatiquement un nouveau en quelques secondes, sans intervention humaine. C'est ce qu'on appelle le self-healing.

Installer Minikube — cluster local

Un cluster K8s sur ta machine pour apprendre

PRATIQUE ▼
🎯

Objectif : installer Minikube et kubectl sur ta machine, démarrer un cluster local, et vérifier qu'il fonctionne. À la fin de cette leçon tu as un vrai cluster K8s qui tourne.

C'est quoi Minikube ?

Minikube crée un cluster Kubernetes en local — une seule machine qui joue le rôle de tout le cluster. C'est l'outil standard pour apprendre K8s sans avoir accès à un cloud.

kubectl est la ligne de commande pour parler au cluster — comme git est la CLI de Git.

1 — Installer kubectl
bash — Ubuntu/Debian
# Télécharger kubectl
curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"

# Rendre exécutable et déplacer
chmod +x kubectl
sudo mv kubectl /usr/local/bin/

# Vérifier
kubectl version --client
2 — Installer Minikube
bash — Ubuntu/Debian
# Télécharger et installer Minikube
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube

# Vérifier
minikube version
3 — Démarrer le cluster
bash
# Démarrer Minikube avec Docker comme driver
minikube start --driver=docker

# Vérifier que le cluster est actif
kubectl cluster-info

# Voir les nodes du cluster (il n'y en a qu'un — minikube)
kubectl get nodes
✅

Si kubectl get nodes affiche un node avec le statut Ready, ton cluster est opérationnel. La première fois, Minikube télécharge une image — ça peut prendre quelques minutes.

Les commandes Minikube essentielles
bash — référence
minikube start          # démarrer le cluster
minikube stop           # arrêter (sans tout supprimer)
minikube delete         # supprimer complètement le cluster
minikube status         # état du cluster
minikube dashboard      # interface web (optionnel)
minikube ip             # IP du cluster local
🧠 Question
Quelle est la différence entre minikube stop et minikube delete ? Lequel utilises-tu en fin de journée si tu veux reprendre le lendemain ?
✅ Réponse

minikube stop arrête le cluster mais conserve tout son état — tes déploiements, services et configurations sont sauvegardés. Le lendemain, minikube start reprend exactement là où tu t'étais arrêté.

minikube delete supprime complètement le cluster et tout ce qu'il contient. À n'utiliser que quand tu veux repartir de zéro.

En fin de journée : minikube stop.

Pods et kubectl — premiers pas

Créer, inspecter et supprimer des Pods

PRATIQUE ▼
🎯

Objectif : comprendre ce qu'est un Pod, créer ton premier Pod avec kubectl, et apprendre les commandes de base pour l'inspecter et le déboguer.

Le Pod — unité de base de K8s

Un Pod est la plus petite unité déployable dans Kubernetes. Il contient un ou plusieurs conteneurs qui partagent le même réseau et le même stockage. En pratique, la plupart des Pods contiennent un seul conteneur.

Pod hello-pipeline-pod └── conteneur : hello-pipeline:latest IP : 10.244.0.5 (IP interne au cluster) Port : aucun exposé pour l'instant
⚠️

Un Pod est éphémère — si il plante ou est supprimé, il n'est pas recréé automatiquement. C'est pour ça qu'on utilise des Deployments en pratique (leçon 4).

Créer un Pod en ligne de commande
bash
# Charger l'image hello-pipeline dans Minikube
# (Minikube a son propre Docker, différent du tien)
minikube image load hello-pipeline:latest

# Créer un Pod depuis l'image
kubectl run hello-pod \
  --image=hello-pipeline:latest \
  --image-pull-policy=Never \
  --restart=Never

# --image-pull-policy=Never = utiliser l'image locale, ne pas chercher sur internet
# --restart=Never = Pod simple (pas de redémarrage automatique)
Les commandes kubectl essentielles
bash — inspecter les Pods
# Lister tous les Pods
kubectl get pods

# Infos détaillées sur un Pod (events, état, IP...)
kubectl describe pod hello-pod

# Voir les logs du Pod
kubectl logs hello-pod

# Ouvrir un terminal dans le Pod
kubectl exec -it hello-pod -- bash

# Supprimer le Pod
kubectl delete pod hello-pod
Exercice — créer et explorer
  • Charger l'image dans Minikube avec minikube image load
  • Créer le Pod hello-pod
  • Vérifier son statut avec kubectl get pods
  • Lire ses logs avec kubectl logs hello-pod
  • L'inspecter avec kubectl describe pod hello-pod
  • Le supprimer avec kubectl delete pod hello-pod
🧠 Question
Tu supprimes un Pod avec kubectl delete pod hello-pod. Que se passe-t-il ? Est-ce que le Pod est recréé automatiquement ?
✅ Réponse

Non — le Pod est définitivement supprimé et n'est pas recréé. C'est parce qu'on l'a créé directement avec kubectl run --restart=Never, sans Deployment pour le surveiller.

C'est précisément pour cette raison qu'en production on n'utilise jamais des Pods "nus" — on passe toujours par un Deployment qui surveille les Pods et les recrée automatiquement si nécessaire.

4

Deployments — la vraie façon de déployer

Fichiers YAML, replicas, self-healing

YAML ▼
🎯

Objectif : créer un Deployment en écrivant un fichier YAML K8s, comprendre comment il gère les Pods, et observer le self-healing en action.

Pourquoi un Deployment ?

Un Deployment dit à K8s : "Je veux toujours 3 copies de ce Pod en fonctionnement". K8s surveille en permanence et recrée automatiquement les Pods qui tombent.

Deployment hello-deployment (replicas: 3) ├── Pod hello-deployment-abc12 Running ├── Pod hello-deployment-def34 Running └── Pod hello-deployment-ghi56 Running ← si celui-ci plante... K8s en crée un nouveau automatiquement
Créer le fichier deployment.yml

Crée un fichier k8s/deployment.yml dans hello-pipeline :

yaml — k8s/deployment.yml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello-deployment
spec:
  replicas: 2           # 2 Pods en permanence
  selector:
    matchLabels:
      app: hello-pipeline
  template:
    metadata:
      labels:
        app: hello-pipeline  # label pour que le Service trouve ces Pods
    spec:
      containers:
        - name: hello-pipeline
          image: hello-pipeline:latest
          imagePullPolicy: Never  # image locale Minikube
          env:
            - name: APP_ENV
              value: production
Appliquer le Deployment
bash
# Appliquer le fichier YAML au cluster
kubectl apply -f k8s/deployment.yml

# Voir les Pods créés
kubectl get pods

# Voir le Deployment
kubectl get deployments

# Infos détaillées
kubectl describe deployment hello-deployment
Observer le self-healing
bash — simulation de panne
# Note le nom d'un de tes Pods
kubectl get pods

# Supprime-le brutalement
kubectl delete pod hello-deployment-XXXXX

# Observe immédiatement — K8s recrée un nouveau Pod
kubectl get pods

# Tu verras l'ancien Terminating et un nouveau en Creating
✨

C'est le self-healing en action — l'une des fonctionnalités les plus puissantes de Kubernetes. Le Deployment maintient en permanence le nombre de replicas demandé.

🧠 Question
Dans le fichier YAML, à quoi sert le champ selector.matchLabels ? Que se passerait-il si les labels du selector et du template ne correspondaient pas ?
✅ Réponse

selector.matchLabels dit au Deployment "surveille les Pods qui ont ce label". Le Deployment compte les Pods correspondants et en crée ou supprime pour atteindre le nombre de replicas.

Si les labels ne correspondent pas, le Deployment ne "voit" pas les Pods qu'il a créés — il pense qu'il n'y en a aucun et en crée indéfiniment. K8s refuserait d'ailleurs d'appliquer ce fichier avec une erreur de validation.

Services — exposer les Pods

Comment accéder à une application dans K8s

YAML ▼
🎯

Objectif : comprendre pourquoi les Pods seuls ne sont pas accessibles, créer un Service pour exposer ton Deployment, et y accéder depuis ta machine.

Le problème — les Pods ont des IPs éphémères

Chaque Pod a une IP interne au cluster — mais cette IP change à chaque recréation du Pod. On ne peut pas s'y connecter directement depuis l'extérieur.

Un Service est une adresse stable qui pointe vers les Pods correspondants, peu importe leur IP actuelle.

Utilisateur ↓ Service hello-service (IP stable: 10.96.0.1) ├── Pod 10.244.0.3 (IP change à chaque recréation) └── Pod 10.244.0.4 (K8s répartit le trafic entre les Pods)
Les types de Services
— les 3 types principaux —
ClusterIP   # accessible uniquement DANS le cluster (défaut)
NodePort    # expose un port sur chaque node — accessible de l'extérieur
LoadBalancer # crée un load balancer cloud (AWS, GCP, Azure)

Pour Minikube en local, on utilise NodePort pour accéder depuis ta machine.

Créer le fichier service.yml
yaml — k8s/service.yml
apiVersion: v1
kind: Service
metadata:
  name: hello-service
spec:
  type: NodePort
  selector:
    app: hello-pipeline   # même label que dans le Deployment
  ports:
    - port: 80             # port du Service dans le cluster
      targetPort: 8080    # port du conteneur
      nodePort: 30080     # port exposé sur le node (30000-32767)
bash
# Appliquer le Service
kubectl apply -f k8s/service.yml

# Voir les Services
kubectl get services

# Accéder au service depuis ta machine (Minikube)
minikube service hello-service --url

# Ou ouvrir directement dans le navigateur
minikube service hello-service
💡

Sur Minikube, minikube service nom-service --url te donne l'URL exacte pour accéder au service depuis ton navigateur. En production cloud, le LoadBalancer te donnerait directement une IP publique.

🧠 Question
Comment le Service sait-il vers quels Pods rediriger le trafic ? Quel mécanisme K8s utilise-t-il ?
✅ Réponse

Le Service utilise les labels. Dans son fichier YAML, selector.app: hello-pipeline dit au Service de rediriger vers tous les Pods qui ont le label app: hello-pipeline.

C'est le même système de labels que dans le Deployment — c'est pour ça que les deux doivent utiliser exactement le même label. K8s met à jour dynamiquement la liste des Pods cibles au fur et à mesure qu'ils sont créés ou supprimés.

Quiz de clôture
🧠 Question 1 / 3
Quelle est la différence entre un Pod et un Deployment ? Pourquoi utilise-t-on toujours un Deployment en production plutôt que des Pods directs ?
✅ Réponse

Un Pod est une instance unique et éphémère — s'il plante, il reste mort.

Un Deployment est un contrôleur qui surveille ses Pods et maintient en permanence le nombre demandé (replicas). Si un Pod plante, le Deployment en recrée un automatiquement.

En production on utilise toujours un Deployment pour le self-healing, la gestion des mises à jour progressives (rolling updates), et le scaling facile.

🧠 Question 2 / 3
Tu veux passer ton application de 2 à 5 replicas sans modifier le fichier YAML. Quelle commande kubectl utilises-tu ?
✅ Réponse
bash
kubectl scale deployment hello-deployment --replicas=5

K8s crée immédiatement 3 nouveaux Pods pour atteindre 5. C'est le scaling horizontal en action — ajouter des instances plutôt qu'augmenter les ressources d'une seule machine.

🧠 Question 3 / 3
Explique le rôle des labels dans Kubernetes. Comment le Deployment et le Service s'en servent-ils pour fonctionner ensemble ?
✅ Réponse

Les labels sont des paires clé/valeur attachées aux objets K8s (Pods, Services, etc.). Ils permettent de sélectionner des groupes d'objets.

Le Deployment utilise selector.matchLabels pour savoir quels Pods il doit surveiller et gérer.

Le Service utilise selector pour savoir vers quels Pods rediriger le trafic entrant.

Les deux utilisent le même label (app: hello-pipeline) — c'est ce qui les lie ensemble sans référence directe. C'est un système de découverte par labels, très flexible.
