Les layers et le cache Docker

Comprendre pourquoi un build est rapide ou lent

LAYERS ▼
🎯

Objectif : comprendre comment Docker construit les images couche par couche, et pourquoi l'ordre des instructions dans le Dockerfile change radicalement la vitesse des builds.

Chaque instruction = une couche

Docker construit une image comme une pile de couches (layers). Chaque instruction du Dockerfile crée une nouvelle couche immuable. Voici ce que ça donne pour ton hello-pipeline :

1 FROM python:3.11-slim — image de base CACHED ~150 Mo
2 WORKDIR /app — définit le dossier de travail CACHED 0 Mo
3 COPY requirements.txt . — copie les dépendances CACHED ~1 Ko
4 RUN pip install -r requirements.txt CACHED ~50 Mo
5 COPY . . — copie tout le code REBUILT ~10 Ko
6 CMD ["python", "app.py"] — commande de démarrage REBUILT 0 Mo

Quand tu modifies app.py, Docker invalide le cache à partir de la couche 5 seulement — les couches 1 à 4 sont réutilisées. C'est pour ça que le build est rapide après la première fois.

⚠️

Si tu avais mis COPY . . avant pip install, chaque modification de code invaliderait le cache du pip install — et tu réinstallerais toutes les dépendances à chaque build. C'est l'erreur la plus fréquente des débutants.

Voir les layers en pratique
bash — dans hello-pipeline/
# Voir l'historique des layers d'une image
docker history hello-pipeline

# Version lisible avec les tailles
docker history hello-pipeline --no-trunc

# Construire et observer les messages CACHED vs REBUILT
docker build -t hello-pipeline .
# Modifie app.py, puis rebuild — observe ce qui est CACHED
docker build -t hello-pipeline .
🧠 Question
Tu modifies uniquement requirements.txt pour ajouter une nouvelle dépendance. À partir de quelle couche le cache est-il invalidé ?
✅ Réponse

À partir de la couche 3 (COPY requirements.txt .). Docker détecte que le fichier a changé, invalide cette couche et toutes celles qui suivent — y compris le pip install qui sera entièrement réexécuté. C'est normal et c'est le comportement attendu : les nouvelles dépendances doivent être installées.

Déboguer dans un conteneur

Entrer dans un conteneur, lire les logs, inspecter

DEBUG ▼
🎯

Objectif : savoir diagnostiquer un problème dans un conteneur — entrer dedans, lire les logs, inspecter sa configuration. C'est la compétence numéro 1 du quotidien DevOps.

Entrer dans un conteneur en cours d'exécution
bash
# Lancer un conteneur en mode interactif avec un terminal
docker run -it hello-pipeline bash

# Tu es maintenant DANS le conteneur — dans /app
# Tu peux explorer les fichiers :
ls -la
cat app.py
python app.py

# Quitter le conteneur
exit
Exécuter une commande dans un conteneur déjà lancé
bash
# Lancer un conteneur en arrière-plan (détaché)
docker run -d --name mon-app hello-pipeline sleep infinity

# Exécuter une commande dans le conteneur actif
docker exec -it mon-app bash

# Ou juste une commande sans ouvrir de shell
docker exec mon-app python app.py

# Arrêter et supprimer le conteneur
docker stop mon-app
docker rm mon-app
Lire les logs d'un conteneur
bash
# Voir les logs d'un conteneur (même arrêté)
docker logs mon-app

# Suivre les logs en temps réel (comme tail -f)
docker logs -f mon-app

# Les 20 dernières lignes seulement
docker logs --tail 20 mon-app
Inspecter un conteneur ou une image
bash
# Toutes les infos d'un conteneur (JSON)
docker inspect mon-app

# Voir les variables d'environnement d'un conteneur
docker inspect mon-app | grep -A 10 '"Env"'

# Voir les stats en temps réel (CPU, mémoire)
docker stats mon-app

🧠 Question
Quelle est la différence entre docker run -it image bash et docker exec -it conteneur bash ?
✅ Réponse

docker run crée un nouveau conteneur depuis une image et y ouvre un terminal. À chaque run, tu as un conteneur vierge.

docker exec ouvre un terminal dans un conteneur déjà en cours d'exécution. C'est utilisé pour déboguer un conteneur qui tourne en production sans l'arrêter.

Variables d'environnement dans Docker

Configurer un conteneur sans modifier son image

PRATIQUE ▼
🎯

Objectif : comprendre et utiliser les variables d'environnement pour configurer un conteneur différemment selon l'environnement (dev, staging, prod) sans changer l'image.

Pourquoi les variables d'environnement ?

L'image Docker doit être identique entre dev, staging et prod. Ce qui change selon l'environnement — URL de base de données, clés API, mode debug — passe par des variables d'environnement injectées au démarrage du conteneur.

Passer des variables au démarrage
bash
# Passer une variable avec -e
docker run -e APP_ENV=production hello-pipeline env

# Passer plusieurs variables
docker run -e APP_ENV=production -e DEBUG=false hello-pipeline env

# Passer un fichier de variables (.env)
docker run --env-file .env hello-pipeline env
Utiliser les variables dans app.py

Mets à jour app.py pour lire une variable d'environnement :

python — app.py mis à jour
import os


def greet(name: str) -> str:
    """Retourne un message de bienvenue."""
    if not name:
        raise ValueError("Le nom ne peut pas être vide")
    env = os.getenv("APP_ENV", "development")
    return f"Hello, {name}! Pipeline is alive 🚀 [{env}]"


def add(a: int, b: int) -> int:
    """Additionne deux nombres."""
    return a + b


def multiply(a: int, b: int) -> int:
    """Multiplie deux nombres."""
    return a * b


def subtract(a: int, b: int) -> int:
    """Soustrait b de a."""
    return a - b


if __name__ == "__main__":
    print(greet("DevOps"))
    print(f"2 + 3 = {add(2, 3)}")
⚠️

Après avoir modifié app.py, le test test_greet_normal va échouer car le message a changé. Mets à jour le test pour qu'il accepte n'importe quelle valeur de APP_ENV — ou passe la variable dans la commande pytest.

Tester avec différents environnements
bash
# Rebuild l'image avec le nouveau app.py
docker build -t hello-pipeline .

# En mode development (défaut)
docker run hello-pipeline python app.py
# → Hello, DevOps! Pipeline is alive 🚀 [development]

# En mode production
docker run -e APP_ENV=production hello-pipeline python app.py
# → Hello, DevOps! Pipeline is alive 🚀 [production]
🧠 Question
Pourquoi est-il important de ne jamais mettre des secrets (mots de passe, clés API) directement dans le Dockerfile avec une instruction ENV MOT_DE_PASSE=secret ?
✅ Réponse

Parce que les layers d'une image sont permanents et visibles. Même si tu supprimes la variable dans une couche ultérieure, elle reste accessible dans l'historique de l'image via docker history. Tout le monde qui a accès à l'image peut lire le secret.

La bonne pratique : injecter les secrets uniquement au démarrage du conteneur via -e ou --env-file, ou via les secrets GitHub Actions / Kubernetes secrets.

Introduction à Docker Compose

Orchestrer plusieurs conteneurs avec un seul fichier

PRATIQUE ▼
🎯

Objectif : comprendre Docker Compose et créer un fichier compose.yml pour lancer ton application en une seule commande, avec ses variables d'environnement configurées.

Pourquoi Docker Compose ?

Quand une application a plusieurs composants (app + base de données + cache), lancer chaque conteneur à la main avec docker run devient vite compliqué. Docker Compose permet de tout décrire dans un fichier YAML et de tout démarrer en une commande.

💡

Même pour une application simple comme hello-pipeline, Compose est utile pour figer les options de lancement (-e, ports, volumes) et ne plus avoir à les retaper à chaque fois.

Créer compose.yml

Crée le fichier compose.yml à la racine de hello-pipeline :

yaml — compose.yml
# Version minimale — un seul service
services:
  app:
    build: .
    environment:
      - APP_ENV=development
    command: python app.py
Les commandes Compose essentielles
bash — dans hello-pipeline/
# Construire et démarrer tous les services
docker compose up

# En arrière-plan (détaché)
docker compose up -d

# Voir les logs
docker compose logs

# Arrêter les services
docker compose down

# Rebuild forcé de l'image avant de démarrer
docker compose up --build

# Lancer une commande dans le service app
docker compose run app pytest tests/ -v
Ajouter un service de test séparé

Une bonne pratique : avoir un service dédié aux tests dans Compose :

yaml — compose.yml étendu
services:
  app:
    build: .
    environment:
      - APP_ENV=development
    command: python app.py

  test:
    build: .
    environment:
      - APP_ENV=test
    command: pytest tests/ -v
bash
# Lancer uniquement les tests
docker compose run test

# Lancer uniquement l'app
docker compose run app
🧠 Question
Tu as modifié app.py et tu veux relancer les tests avec Compose en t'assurant que l'image est bien reconstruite. Quelle commande utilises-tu ?
✅ Réponse
bash
docker compose run --build test

Le flag --build force la reconstruction de l'image avant de lancer le conteneur. Sans ce flag, Compose utilise l'image existante même si le code a changé.

Bonnes pratiques Dockerfile

Images légères, sécurisées, maintenables

BEST PRACTICES ▼
🎯

Objectif : appliquer les bonnes pratiques qui font la différence entre une image "qui marche" et une image prête pour la production — légère, sécurisée, maintenable.

1 — Utiliser une image de base slim ou alpine
— comparaison des tailles —
python:3.11          ~ 900 Mo  (pleine — tout inclus)
python:3.11-slim     ~ 150 Mo  (sans les outils de build)
python:3.11-alpine   ~  50 Mo  (minimaliste — Linux Alpine)
💡

slim est le meilleur compromis pour Python — alpine peut causer des problèmes de compatibilité avec certaines librairies C. Pour commencer, reste sur slim.

2 — Ne pas lancer en root

Par défaut, Docker tourne en root dans le conteneur — une faille de sécurité. Crée un utilisateur dédié :

Dockerfile — version sécurisée
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

# Crée un utilisateur non-root et bascule dessus
RUN useradd -m appuser
USER appuser

CMD ["python", "app.py"]
3 — Regrouper les RUN pour réduire les layers
— mauvaise pratique vs bonne pratique —
# ❌ 3 layers séparés inutilement
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get clean

# ✅ 1 seul layer — plus léger
RUN apt-get update \
    && apt-get install -y curl \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
4 — Vérifier la taille de ton image
bash
# Voir la taille de toutes les images
docker images

# Analyse détaillée avec dive (outil externe)
# Installation : sudo apt install dive
dive hello-pipeline
🧠 Question
Dans ton Dockerfile actuel, quelle instruction empêche les fichiers inutiles d'être copiés dans l'image ? Comment fonctionne-t-elle ?
✅ Réponse

Le fichier .dockerignore à la racine du projet. Il fonctionne exactement comme .gitignore mais pour Docker : il liste les fichiers et dossiers à exclure lors du COPY . .. Sans lui, Docker copierait .git/, __pycache__/, .pytest_cache/ etc. dans l'image, l'alourdissant inutilement.

Récap & quiz de clôture

Mémo commandes + 3 questions de synthèse

SYNTHÈSE ▼
🎯

Objectif : avoir une référence complète sur Docker et valider les acquis des modules 4 et 5 avec 3 questions de synthèse.

Mémo — toutes les commandes Docker
bash — référence complète
── IMAGES ────────────────────────────────────────
docker build -t nom .         # construire une image
docker images                 # lister les images
docker rmi nom                # supprimer une image
docker history nom            # voir les layers
docker pull image:tag         # télécharger une image
docker push image:tag         # pousser vers un registry

── CONTENEURS ────────────────────────────────────
docker run nom                # créer et lancer
docker run -it nom bash       # terminal interactif
docker run -d nom             # en arrière-plan
docker run -e VAR=val nom     # avec variable d'env
docker run --name mon-app nom # avec un nom
docker ps                     # conteneurs actifs
docker ps -a                  # tous les conteneurs
docker stop nom               # arrêter
docker rm nom                 # supprimer
docker logs nom               # voir les logs
docker logs -f nom            # logs en temps réel
docker exec -it nom bash      # entrer dans un actif
docker inspect nom            # toutes les infos (JSON)
docker stats nom              # CPU/mémoire en temps réel

── COMPOSE ───────────────────────────────────────
docker compose up             # démarrer tous les services
docker compose up -d          # en arrière-plan
docker compose up --build     # rebuild + démarrer
docker compose down           # arrêter et supprimer
docker compose logs           # voir les logs
docker compose run service cmd # lancer une commande

── NETTOYAGE ─────────────────────────────────────
docker system prune           # nettoyer tout l'inutilisé
docker system prune -a        # tout supprimer (images comprises)

🧠 Question 1 / 3
Tu veux vérifier qu'un conteneur en cours d'exécution a bien reçu la variable APP_ENV=production. Sans l'arrêter, quelle commande utilises-tu ?

bash

🧠 Question 2 / 3
Un collègue te dit que son build Docker prend 3 minutes à chaque fois qu'il modifie une ligne de code. Sans voir son Dockerfile, quelle est la cause probable et comment le corriger ?
✅ Réponse

Il a probablement mis COPY . . avant le pip install. Du coup, chaque modification de code invalide le cache du pip install et réinstalle toutes les dépendances depuis zéro.

La correction : toujours copier requirements.txt en premier, installer les dépendances, puis copier le reste du code.

🧠 Question 3 / 3
Quelle est la prochaine étape logique après ce module ? Qu'est-ce qu'on va pouvoir faire avec les workflows avancés GitHub Actions qu'on ne pouvait pas faire avant ?
✅ Réponse

Maintenant qu'on maîtrise Docker, les workflows avancés vont permettre de :

— Utiliser des secrets GitHub pour stocker des tokens de registry de façon sécurisée
— Créer des jobs multiples (build, test, publish) qui s'enchaînent ou tournent en parallèle
— Ajouter des conditions (déployer seulement sur certaines branches)
— Utiliser des environments (staging / production) avec des approbations manuelles

En résumé : assembler tout ce qu'on a appris en un pipeline professionnel complet.
