Workflow CLI
01 Le workflow fondamental Terraform ▼
À retenir
Le workflow officiel HashiCorp est en trois étapes : Write → Plan → Apply. En CLI, cela se traduit par init → plan → apply. 
terraform init     # télécharge providers + modules + init backend
terraform plan     # dry-run : montre ce qui va changer
terraform apply    # applique les changements
terraform destroy  # détruit toutes les ressources du state
Piège
terraform get télécharge uniquement les modules. terraform init fait tout (providers + modules + backend). À l'exam, la bonne réponse est presque toujours init.
02 terraform plan — que fait-il exactement ? ▼
À retenir
terraform plan est un dry-run. Il compare la configuration souhaitée avec l'état actuel (state file + infra réelle) et affiche le diff. Il ne modifie rien. 
terraform plan              # aperçu
terraform plan -out=tfplan  # sauvegarder le plan
terraform apply tfplan      # appliquer exactement ce plan
Symboles du plan
+ créer  ·  - détruire  ·  ~ modifier  ·  +/- remplacer  ·  <= data source lue
03 terraform fmt et terraform validate ▼
CommandeRôleAccès réseau ?
terraform fmtFormate les fichiers .tf selon le style canonique HashiCorpNon
terraform validateVérifie la syntaxe et la cohérence interne des fichiers .tfNon
terraform planCalcule le diff avec l'infra réelleOui
Piège
terraform validate n'accède pas au provider ni au state. Il vérifie uniquement la syntaxe HCL. Pour détecter les erreurs d'API (ex. type d'instance invalide), il faut un plan.
Ordre recommandé
fmt → validate → plan → apply
04 -auto-approve et -refresh=false ▼ 
terraform apply -auto-approve      # skip la confirmation interactive
terraform plan  -refresh=false     # n'interroge pas l'infra réelle
terraform apply -replace=aws_instance.web  # force recréation (ex-taint)
terraform apply -refresh-only      # met à jour le state sans toucher l'infra
Piège
TF_AUTO_APPROVE n'existe pas. Le flag est -auto-approve sur la commande. Pour désactiver les prompts en CI/CD, utiliser ce flag ou TF_INPUT=0.
Variables d'env clés
TF_VAR_name · TF_INPUT · TF_LOG · TF_LOG_PATH · TF_IN_AUTOMATION
05 terraform taint est déprécié → -replace ▼
À retenir
Depuis Terraform 0.15.2, terraform taint est déprécié. Le remplacement moderne est le flag -replace sur apply. 
terraform taint aws_instance.web           # déprécié
terraform apply -replace=aws_instance.web  # moderne
Piège certif
Les questions de certif peuvent encore mentionner taint. Savoir que les deux existent, mais que -replace est la réponse moderne attendue.

HCL — Blocs & syntaxe
06 Les 5 blocs fondamentaux HCL ▼
BlocRôle
terraform {}Meta-config : version, required_providers, backend
provider {}Configure le plugin cloud (credentials, région…)
resource {}Crée et gère une ressource
data {}Lit une ressource existante (read-only)

module {}Appelle un module enfant
Piège
data ne crée rien. resource gère le cycle de vie complet. Ne pas les confondre à l'exam.
07 Référencer un attribut de ressource ▼
À retenir
La syntaxe est : TYPE.NOM.ATTRIBUT. Le mot-clé resource n'est jamais inclus dans la référence. 
# Déclaration :
resource "aws_instance" "web" { ... }

# Référence :
aws_instance.web.id          # ✅ correct
resource.aws_instance.web.id # ❌ incorrect
Autres références
var.nom · local.nom · data.type.nom.attr · module.nom.output
08 count vs for_each — quelle différence ? ▼
countfor_each
Type d'entréeIntegerMap ou Set of strings
IdentifiantIndex numérique [0], [1]…Clé nommée ["prod"]…
Référence internecount.indexeach.key / each.value
Risque suppressionDécale les index !Stable par clé
Piège
Avec count, supprimer un élément au milieu d'une liste décale tous les index suivants → Terraform peut vouloir détruire et recréer des ressources. for_each est plus sûr pour les collections nommées. 
# count :
count = 3  # référence : aws_instance.web[0]

# for_each avec map :
for_each = var.instances  # référence : aws_instance.web["prod"]
09 L'opérateur ternaire — seule syntaxe conditionnelle HCL ▼
À retenir
HCL n'a pas de if/else. La seule expression conditionnelle est le ternaire : condition ? val_si_vrai : val_si_faux. 
# Ternaire :
instance_type = var.env == "prod" ? "t3.large" : "t3.micro"

# Ressource optionnelle :
count = var.create_instance ? 1 : 0

# For expression avec filtre :
[for s in var.list : upper(s) if length(s) > 3]
Piège
if/else, switch, when n'existent pas en HCL. Seul le ternaire ? : est valide.
10 locals vs variable — quelle différence ? ▼
variable {}locals {}
Surcharge externeOui (CLI, env, tfvars)Non

Calcul dynamiqueNon (valeur fixe)Oui
Référencevar.nomlocal.nom
Règle simple
variable = entrée externe configurable. locals = calcul interne non surchargeable.
11 lifecycle — les 4 arguments ▼
ArgumentEffet
create_before_destroyCrée la nouvelle ressource AVANT de détruire l'ancienne
prevent_destroyTerraform échoue si le plan détruit cette ressource
ignore_changesIgnore les diffs sur les attributs listés
replace_triggered_byForce le remplacement si les ressources listées changent
Piège
prevent_destroy = true ne bloque pas la commande terraform destroy en silence — il fait échouer le plan avec une erreur. Pour détruire la ressource, il faut d'abord retirer le bloc lifecycle.
12 depends_on — quand l'utiliser ? ▼
À retenir
Terraform infère automatiquement les dépendances depuis les références. depends_on sert uniquement quand la dépendance est implicite (non visible dans la config). 
# Exemple : ressource qui dépend d'une policy attachée
# mais dont l'ARN n'est pas dans sa config
resource "aws_instance" "app" {
  depends_on = [aws_iam_role_policy_attachment.app]
}
Mémo
Préférer les références implicites (plus claires). depends_on = dernier recours pour dépendances invisibles.

15 sensitive = true — ce que ça fait et ce que ça ne fait pas ▼
À retenir
sensitive = true cache la valeur dans les outputs CLI (plan, apply). La valeur est toujours présente en clair dans le state file.
Piège
sensitive = true ne chiffre pas la valeur dans le state. Pour protéger les secrets dans le state, il faut chiffrer le backend (ex: S3 + SSE-KMS).
Mémo
sensitive = masqué en CLI ≠ chiffré. Le state file reste la source de vérité, et peut contenir des mots de passe en clair.

State & Backends
17 Rôle du state file ▼
À retenir
Le state file (terraform.tfstate) est le mappage entre les ressources déclarées en HCL et leur identifiant réel dans le cloud. Sans lui, Terraform ne sait pas ce qu'il gère.
Localisation défaut./terraform.tfstate (backend local)
FormatJSON (ne jamais éditer à la main)
Contenu sensible ?Oui — peut contenir des mots de passe en clair
À commiter dans Git ?Non — utiliser un remote backend
18 Commandes terraform state — lesquelles font quoi ? ▼
CommandeEffet
state listListe toutes les adresses dans le state
state show addrDétails d'une ressource dans le state
state mv src dstRenomme une adresse dans le state
state rm addrSupprime du state (sans détruire l'infra !)
state pullExporte le state brut (JSON)
state pushPousse un state local vers le backend
Piège
terraform state rm retire la ressource du state sans toucher à l'infra réelle. La ressource continue d'exister dans le cloud — Terraform ne la gère plus.
19 State locking — pourquoi et comment ▼
À retenir
Le state locking empêche deux opérations Terraform concurrentes d'écrire dans le state simultanément (risque de corruption).
Backend AWSS3 (stockage) + DynamoDB (verrou)
Backend AzureAzure Blob Storage (verrou natif via lease)
HCP TerraformVerrou intégré
# Libérer un verrou bloqué :
terraform force-unlock LOCK_ID
Piège
La table DynamoDB AWS doit avoir une clé primaire nommée exactement LockID (type String). Un nom différent empêche le locking de fonctionner.
20 Backend S3 + DynamoDB — configuration type ▼ 
terraform {
  backend "s3" {
    bucket         = "my-tf-state"
    key            = "prod/terraform.tfstate"
    region         = "eu-west-1"
    dynamodb_table = "terraform-locks"  # LockID
    encrypt        = true               # SSE-S3
  }
}
Mémo
S3 = stockage du state. DynamoDB = verrou. encrypt = true active SSE-S3. Pour SSE-KMS, ajouter kms_key_id.
21 terraform import — avant et après Terraform 1.5 ▼
Avant 1.51.5+
Écrire la configManuel (obligatoire avant import)Auto-généré avec -generate-config-out
Commandeterraform import addr idBloc import {} + terraform plan
# Avant 1.5 :
terraform import aws_instance.web i-1234567890abcdef0

# 1.5+ avec génération auto :
terraform plan -generate-config-out=generated.tf
Piège
terraform import ne génère pas la configuration HCL (avant 1.5). Il ajoute uniquement la ressource au state. La config doit être écrite manuellement, sinon le plan suivant montrera des diffs.
22 Le bloc moved {} — refactoriser sans détruire ▼
À retenir
Le bloc moved {} (Terraform 1.1+) permet de renommer ou déplacer une ressource dans le state sans la détruire. C'est l'alternative déclarative à terraform state mv. 
moved {
  from = aws_instance.old_name
  to   = aws_instance.new_name
}
Mémo
Après avoir appliqué le bloc moved, retirer le bloc de la config. Il n'est utile que le temps de la migration.

25 version argument — pour quel type de source ? ▼
À retenir
L'argument version dans un bloc module ne fonctionne qu'avec les sources de type Terraform Registry (public ou privé). Il est ignoré pour les chemins locaux et les sources Git.
Sourceversion argument
Registry (hashicorp/consul/aws)Supporté
Chemin local (./modules/vpc)Non supporté
Git (git::https://…)Non → utiliser ?ref=
Opérateur ~>
version = "~> 3.0" autorise 3.x mais pas 4.0. Opérateur idiomatique Terraform.
26 TF_VAR_ — propagation aux modules enfants ? ▼
Piège majeur
Les variables d'environnement TF_VAR_ et les fichiers terraform.tfvars sont lus uniquement par le module racine. Les modules enfants reçoivent leurs valeurs exclusivement via les arguments du bloc module {}. 
# ✅ Bonne façon de passer une valeur à un module enfant :
module "vpc" {
  source   = "./modules/vpc"
  cidr     = var.vpc_cidr   # ← passé via argument
}
27 Backend dans un module enfant — interdit ▼
Piège
Le bloc backend dans un bloc terraform {} est uniquement valide dans le module racine. Placer un backend dans un module enfant provoque une erreur.
Mémo
Module racine = là où on exécute terraform init/plan/apply. Il peut y avoir un seul backend par configuration. Tous les modules partagent le même state.

29 compact() et toset() — effets sur les listes ▼ 
# compact() supprime les chaînes vides et null :
compact(["a", "", "b", null, "c"])
→ ["a", "b", "c"]

# toset() supprime les doublons (ordre non garanti) :
toset(["a", "b", "a", "c"])
→ {"a", "b", "c"}
Piège
toset() ne retourne pas une liste — c'est un set. L'ordre est perdu. Pour passer à for_each, c'est parfait. Pour indexer avec [0], ce n'est pas adapté.

path.cwdRépertoire de travail courant
# Usage typique dans un module :

Mémo
Toujours utiliser path.module dans un module pour référencer des fichiers locaux. path.root pointe vers le module racine, pas le module courant.

31 .terraform.lock.hcl — rôle et gestion ▼
À retenir
Le fichier .terraform.lock.hcl fixe les versions exactes des providers et leurs checksums. Il doit être commité dans Git pour garantir des builds reproductibles.
Fichier/DossierGit ?
.terraform.lock.hclOui — épingle les versions
.terraform/Non — binaires providers
*.tfstateNon — secrets possibles
*.tfvarsParfois — si pas de secrets
terraform init -upgrade  # mise à jour providers + lock file
32 Contraintes de versions — opérateurs ▼
OpérateurSignification
= 1.5.0Exactement 1.5.0
>= 1.51.5 ou plus (sans plafond)
~> 1.5>= 1.5.0 et < 2.0.0
~> 1.5.0>= 1.5.0 et < 1.6.0
>= 1.5, < 2.0Plage explicite
Mémo — pessimistic constraint operator
~> est l'opérateur idiomatique Terraform. ~> 1.5 autorise les mises à jour mineures (1.6, 1.7…) mais pas la version majeure suivante (2.0).
33 Provider alias — plusieurs configurations du même provider ▼ 
# Deux régions AWS dans la même config :
provider "aws" { region = "eu-west-1" }

provider "aws" {
  alias  = "us"
  region = "us-east-1"
}

# Référencer l'alias dans une ressource :
resource "aws_s3_bucket" "us_bucket" {
  provider = aws.us
}
Piège
Dans la ressource, le mot-clé est provider (méta-argument), pas alias. La valeur est aws.us (type.alias), pas une string.

Pièges certification HCTA-003
34 terraform apply échoue à mi-chemin — état du state ? ▼
À retenir
Si terraform apply échoue à mi-parcours, le state reflète ce qui a été créé avec succès. Terraform n'a pas de rollback automatique. L'apply n'est pas transactionnel.
Piège
Le state n'est pas vidé, ni rollbacké. Il contient les ressources partiellement créées. Lancer terraform plan après un échec montre ce qui reste à créer.
35 terraform refresh (déprécié) → -refresh-only ▼
CommandeComportement
terraform refresh (déprécié)Met à jour le state sans modifier l'infra
terraform apply -refresh-onlyModerne : idem, avec confirmation
terraform plan -refresh=falseSaute la réconciliation state/infra
Piège
terraform refresh est déprécié depuis Terraform 0.15.4. La réponse moderne à l'exam est terraform apply -refresh-only.
36 Configuration drift — comment Terraform le détecte ▼
À retenir
Le drift = l'infra réelle diverge du state (changement manuel hors Terraform). terraform plan le détecte en comparant l'infra réelle avec le state. Le plan montre ce qu'il faut corriger.
Mémo
Détecter : terraform plan. Corriger : terraform apply. Mettre à jour le state sans corriger : terraform apply -refresh-only.
37 terraform_remote_state — lire les outputs d'une autre config ▼
À retenir
La data source terraform_remote_state permet à une configuration B de lire les outputs du state d'une configuration A. Seules les outputs sont accessibles — pas tous les attributs. 
data "terraform_remote_state" "vpc" {
  backend = "s3"
  config = {
    bucket = "my-state"
    key    = "vpc/terraform.tfstate"
    region = "eu-west-1"
  }
}
# Usage :
vpc_id = data.terraform_remote_state.vpc.outputs.vpc_id
38 Provisioners — pourquoi les éviter ▼
À retenir
HashiCorp considère les provisioners comme un dernier recours. Ils brisent l'idempotence et créent des dépendances externes. Préférer user_data, cloud-init ou des outils de configuration management.
ProvisionerRôle
local-execExécute une commande sur la machine qui lance Terraform
remote-execExécute une commande sur la ressource distante (via SSH/WinRM)
fileCopie un fichier ou dossier vers la ressource distante
Piège certif
Un provisioner on_failure = continue fait ignorer l'erreur du provisioner. La ressource reste dans le state mais est taintée automatiquement.
