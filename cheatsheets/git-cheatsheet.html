Git Cheatsheet

Toutes les commandes essentielles · avec explications et exemples

🌱
Initialisation & Configuration
git init · git clone · git config
git init
Crée un nouveau dépôt Git dans le dossier courant. Un répertoire caché .git/ est créé — c'est lui qui contient tout l'historique.
$ git init
Initialized empty Git repository in /projet/.git/

# Ou directement dans un nouveau dossier :
$ git init mon-projet
git clone
Copie un dépôt distant en local, avec tout l'historique. Crée automatiquement le dossier et configure le remote origin.
$ git clone https://github.com/user/repo.git

# Cloner dans un dossier spécifique :
$ git clone https://…/repo.git mon-dossier

# Clone superficiel (uniquement le dernier commit) :
$ git clone --depth 1 https://…/repo.git
git config
Configure Git : identité, éditeur, alias… Le flag --global s'applique à tous vos projets.
# Identité obligatoire (une seule fois en global)
$ git config --global user.name "Prénom Nom"
$ git config --global user.email "toi@mail.com"

# Éditeur par défaut :
$ git config --global core.editor vim

# Voir toute la config :
$ git config --list

# Créer un alias :
$ git config --global alias.st "status"
Les niveaux de config : --system (machine) < --global (utilisateur) < --local (projet). Le plus proche l'emporte.

Snapshot — Staging & Commit
git status · git add · git commit · git diff · git rm · git mv
flux de travail
Les 3 zones de Git : Working Directory → Index (Staging) → Repository
ZONES GIT ┌─────────────────────┐ git add ┌───────────────┐ git commit ┌──────────────────┐ │ Working Directory │ ──────────▶ │ Index/Stage │ ──────────▶ │ Repository │ │ (fichiers locaux) │ │ (zone tampon)│ │ (historique) │ └─────────────────────┘ └───────────────┘ └──────────────────┘ ▲ │ └───────────────────────── git checkout / restore ─────────────────┘
git status
Affiche l'état du dépôt : fichiers modifiés, en attente de commit, non suivis. À lancer en permanence.
$ git status

# Version courte :
$ git status -s
M README.md ← modifié (non stagé)
M main.py ← modifié (stagé)
?? nouveau.txt ← non suivi
git add
Déplace des modifications dans l'index (zone de staging) pour préparer le prochain commit.
# Ajouter un fichier spécifique :
$ git add fichier.txt

# Tout ajouter (fichiers suivis + nouveaux) :
$ git add .

# Mode interactif (choisir les modifications) :
$ git add -p

# Ajouter tous les fichiers .py :
$ git add *.py
git add -p est très puissant : il permet de ne committer qu'une partie d'un fichier (hunk par hunk).
git commit
Enregistre le contenu du staging dans l'historique avec un message. Chaque commit = un snapshot immuable.
# Commit avec message inline :
$ git commit -m "feat: ajoute la page login"

# Stage + commit en une seule commande :
$ git commit -am "fix: corrige le bug d'authentification"

# Modifier le dernier commit (message ou contenu) :
$ git commit --amend -m "nouveau message"
Convention Conventional Commits : feat:, fix:, docs:, chore:, refactor:, test: — très utilisée en CI/CD.
git diff
Montre les différences ligne par ligne. Sans argument : compare working directory vs index. Avec --staged : index vs dernier commit.
# Diff non stagé :
$ git diff

# Diff stagé (prêt à committer) :
$ git diff --staged

# Diff entre deux branches :
$ git diff main..feature/login

# Diff d'un fichier spécifique :
$ git diff HEAD~1 fichier.py
git rm / mv
Supprime ou renomme des fichiers en informant Git. Évite les fichiers fantômes dans l'historique.
# Supprimer un fichier suivi :
$ git rm fichier.txt

# Retirer du suivi SANS supprimer le fichier :
$ git rm --cached secret.env

# Renommer / déplacer :
$ git mv ancien.py nouveau.py
git rm --cached est utile pour retirer un fichier accidentellement commité (ex: .env), puis l'ajouter au .gitignore.
.gitignore
Fichier listant les patterns de fichiers que Git doit ignorer (ne jamais suivre). À placer à la racine du projet.
# Exemple de .gitignore :
.env
*.log
node_modules/
__pycache__/
.DS_Store

# Voir quels fichiers sont ignorés :
$ git status --ignored

# Vérifier pourquoi un fichier est ignoré :
$ git check-ignore -v fichier.log

Branches & Fusion
git branch · git checkout · git switch · git merge · git rebase
git branch
Gère les branches : lister, créer, renommer, supprimer. Une branche = un pointeur léger vers un commit.
# Lister les branches locales :
$ git branch

# Lister toutes les branches (locales + distantes) :
$ git branch -a

# Créer une branche :
$ git branch feature/login

# Supprimer une branche (fusionnée) :
$ git branch -d feature/login

# Forcer la suppression :
$ git branch -D feature/login

# Renommer la branche courante :
$ git branch -m nouveau-nom
git switch
Bascule sur une branche. git switch est la nouvelle syntaxe (Git 2.23+) plus claire que git checkout.
# Basculer sur une branche existante :
$ git switch main

# Créer ET basculer en une commande :
$ git switch -c feature/paiement

# Équivalent ancienne syntaxe :
$ git checkout -b feature/paiement

# Revenir à la branche précédente :
$ git switch -
git merge
Fusionne une branche dans la branche courante. Préserver l'historique ou l'aplatir avec --squash.
# Fusionner feature dans main :
$ git switch main
$ git merge feature/login

# Forcer un commit de merge (pas fast-forward) :
$ git merge --no-ff feature/login

# Squash : écraser tous les commits en 1 :
$ git merge --squash feature/login

# Annuler un merge en cours :
$ git merge --abort
Fast-forward = pas de commit de merge si la branche n'a pas divergé. --no-ff crée toujours un commit de merge (traçabilité).
git rebase
Rejoue les commits d'une branche sur une autre base. Produit un historique linéaire et propre. Ne jamais rebaser une branche partagée.
# Rebaser feature sur main :
$ git switch feature/login
$ git rebase main

# Rebase interactif (nettoyer 3 derniers commits) :
$ git rebase -i HEAD~3

# Annuler un rebase en cours :
$ git rebase --abort

# Continuer après résolution de conflit :
$ git rebase --continue
En rebase interactif : pick (garder), squash (fusionner), reword (renommer), drop (supprimer), edit (modifier).
conflits
Lors d'un merge ou rebase, Git signale les conflits. Il faut les résoudre manuellement puis continuer.
# 1. Voir les fichiers en conflit :
$ git status

# 2. Le fichier contient des marqueurs :
<<<<<<< HEAD
version actuelle (branche courante)
=======
version entrante (branche mergée)
>>>>>>> feature/login

# 3. Après édition manuelle :
$ git add fichier-conflit.py
$ git merge --continue

Remote — Dépôts distants
git remote · git fetch · git pull · git push
git remote
Gère les connexions vers des dépôts distants. origin est le nom conventionnel du dépôt source.
# Lister les remotes :
$ git remote -v

# Ajouter un remote :
$ git remote add origin https://github.com/user/repo.git

# Changer l'URL d'un remote :
$ git remote set-url origin git@github.com:user/repo.git

# Supprimer un remote :
$ git remote remove origin
git fetch
Télécharge les changements du remote sans les fusionner. Sûr : ne touche pas au working directory.
# Récupérer tout depuis origin :
$ git fetch

# Récupérer depuis un remote spécifique :
$ git fetch upstream

# Voir ce qui a changé après fetch :
$ git log HEAD..origin/main --oneline
git pull
Équivalent de git fetch + git merge. Récupère et intègre les changements du remote dans la branche courante.
# Pull depuis origin/branche courante :
$ git pull

# Pull avec rebase (historique linéaire) :
$ git pull --rebase

# Pull depuis une branche spécifique :
$ git pull origin main
Préférer git pull --rebase pour garder un historique linéaire. Configurer globalement : git config --global pull.rebase true
git push
Envoie les commits locaux vers le dépôt distant. -u définit le tracking entre branche locale et distante.
# Premier push d'une branche :
$ git push -u origin feature/login

# Push classique (après -u configuré) :
$ git push

# Pousser tous les tags :
$ git push --tags

# Supprimer une branche distante :
$ git push origin --delete feature/login

# Force push (⚠ dangereux sur branches partagées) :
$ git push --force-with-lease
Toujours préférer --force-with-lease à --force : il vérifie que personne d'autre n'a pushé entre-temps.

Historique & Exploration
git log · git show · git blame · git bisect · git grep
git log
Affiche l'historique des commits. Extrêmement personnalisable avec des formats et filtres.
# Affichage compact (une ligne par commit) :
$ git log --oneline

# Avec graphe des branches :
$ git log --oneline --graph --all

# Filtrer par auteur :
$ git log --author="Titi"

# Commits depuis une date :
$ git log --since="2 weeks ago"

# Commits touchant un fichier :
$ git log --follow -p fichier.py

# Format personnalisé :
$ git log --pretty=format:"%h %an %s"
git show
Affiche le détail d'un commit : métadonnées + diff complet des changements.
# Détail du dernier commit :
$ git show

# Détail d'un commit spécifique :
$ git show a3f9c21

# Voir un fichier à une version donnée :
$ git show HEAD~2:src/main.py
git blame
Affiche l'auteur et le commit de chaque ligne d'un fichier. Utile pour comprendre l'historique d'une modification.
$ git blame fichier.py

# Afficher les lignes 10 à 20 seulement :
$ git blame -L 10,20 fichier.py
git bisect
Trouve le commit qui a introduit un bug par recherche dichotomique. Très efficace sur un grand historique.
# 1. Démarrer la recherche :
$ git bisect start

# 2. Marquer un commit "mauvais" :
$ git bisect bad # HEAD est buggué

# 3. Marquer un commit "bon" :
$ git bisect good v1.2.0

# Git se positionne au milieu, tester, puis :
$ git bisect good # ou bad

# 4. Terminer :
$ git bisect reset
git grep
Recherche un motif dans les fichiers suivis par Git. Plus rapide que grep classique sur les dépôts.
# Chercher dans les fichiers courants :
$ git grep "def connect"

# Avec numéros de ligne :
$ git grep -n "TODO"

# Chercher dans un commit spécifique :
$ git grep "motif" HEAD~5

Annuler & Corriger
git restore · git reset · git revert · git clean
quelle commande choisir ?
Trois façons d'annuler — le choix dépend de ce qu'on veut annuler et de si les commits ont été pushés.
GUIDE DE CHOIX Situation Commande recommandée ──────────────────────────────────────────────────────────────────────── Fichier modifié, pas encore stagé git restore fichier Fichier stagé (git add fait) git restore --staged fichier Dernier commit (pas pushé) git reset HEAD~1 ou git commit --amend Plusieurs commits (pas pushés) git reset HEAD~N Commit déjà pushé / partagé git revert <sha> (crée un commit d'annulation) Fichiers non suivis à supprimer git clean -fd
git restore
Annule des modifications dans le working directory ou retire des fichiers du staging. N'affecte pas l'historique.
# Annuler les modifs non stagées d'un fichier :
$ git restore fichier.py

# Retirer un fichier du staging (unstage) :
$ git restore --staged fichier.py

# Restaurer un fichier depuis un commit :
$ git restore --source HEAD~2 fichier.py
git reset
Déplace le pointeur HEAD en arrière. Trois modes : soft (garde staging), mixed (garde fichiers), hard (tout efface).
# Annuler le dernier commit, garder les modifs stagées :
$ git reset --soft HEAD~1

# Annuler le commit ET le staging (défaut) :
$ git reset HEAD~1

# ⚠ Tout effacer (commit + modifs) :
$ git reset --hard HEAD~1

# Revenir à l'état du remote :
$ git reset --hard origin/main
--hard est irréversible (sans reflog). Ne jamais reset des commits pushés sur une branche partagée.
git revert
Crée un nouveau commit qui annule les effets d'un commit passé. Méthode sûre pour annuler sur des branches partagées.
# Annuler le dernier commit en créant un revert :
$ git revert HEAD

# Annuler un commit spécifique :
$ git revert a3f9c21

# Sans ouvrir l'éditeur :
$ git revert --no-edit HEAD
git clean
Supprime les fichiers non suivis du working directory. Utiliser -n pour prévisualiser avant d'effacer.
# Voir ce qui serait supprimé (dry run) :
$ git clean -n

# Supprimer fichiers non suivis :
$ git clean -f

# Inclure les dossiers non suivis :
$ git clean -fd
git reflog
Le filet de sécurité de Git. Affiche tout ce que HEAD a pointé — même les commits "perdus" après un reset hard.
# Voir l'historique complet des déplacements de HEAD :
$ git reflog
a3f9c21 HEAD@{0}: commit: feat: page login
b72d1e8 HEAD@{1}: reset: moving to HEAD~1

# Récupérer un commit "perdu" :
$ git checkout b72d1e8
# ou créer une branche depuis ce point
$ git branch sauvetage b72d1e8
Le reflog est local et conservé 90 jours par défaut. C'est votre dernier recours en cas de suppression accidentelle.

Stash — Mise en attente
git stash · git stash pop · git stash apply
git stash
Met de côté les modifications en cours (sans commit) pour changer de contexte, puis les restaure. Comme une pile de sauvegardes temporaires.
# Mettre de côté toutes les modifs :
$ git stash

# Avec un message descriptif :
$ git stash push -m "WIP: refacto du module auth"

# Inclure les fichiers non suivis :
$ git stash -u

# Lister tous les stashs :
$ git stash list
stash@{0}: WIP: refacto du module auth
stash@{1}: On main: fix rapide

# Restaurer le dernier stash (et le supprimer) :
$ git stash pop

# Appliquer sans supprimer :
$ git stash apply stash@{1}

# Voir le contenu d'un stash :
$ git stash show -p

# Supprimer un stash :
$ git stash drop stash@{0}

# Vider tous les stashs :
$ git stash clear
Cas typique : tu travailles sur feature/A, un bug urgent sur main arrive → git stash → git switch main → corriger → git switch feature/A → git stash pop

Tags — Versioning
git tag · git push --tags
git tag
Pose une étiquette permanente sur un commit. Utilisé pour marquer les versions de release (v1.0.0, v2.1.3…).
# Lister les tags :
$ git tag

# Tag léger (simple pointeur) :
$ git tag v1.0.0

# Tag annoté (avec message, recommandé) :
$ git tag -a v1.0.0 -m "Release v1.0.0"

# Taguer un commit passé :
$ git tag -a v0.9.0 a3f9c21

# Pousser un tag :
$ git push origin v1.0.0

# Pousser tous les tags :
$ git push --tags

# Supprimer un tag local :
$ git tag -d v1.0.0

# Supprimer un tag distant :
$ git push origin --delete v1.0.0

Avancé
cherry-pick · worktree · submodule · shortlog
git cherry-pick
Applique un commit spécifique d'une autre branche. Utile pour backporter un correctif sans merger toute la branche.
# Appliquer un commit sur la branche courante :
$ git cherry-pick a3f9c21

# Plusieurs commits :
$ git cherry-pick a3f9c21 b72d1e8

# Sans committer (juste stager) :
$ git cherry-pick -n a3f9c21
git worktree
Vérifie plusieurs branches du même dépôt dans des dossiers séparés simultanément. Évite les stash/switch constants.
# Créer un worktree pour une branche :
$ git worktree add ../hotfix hotfix/urgent

# Lister les worktrees :
$ git worktree list

# Supprimer un worktree :
$ git worktree remove ../hotfix
git submodule
Intègre un dépôt Git externe comme sous-dossier. Utile pour les dépendances versionnées ou les bibliothèques partagées.
# Ajouter un submodule :
$ git submodule add https://github.com/user/lib.git libs/util

# Initialiser après un clone :
$ git submodule init
$ git submodule update

# Clone avec sous-modules d'emblée :
$ git clone --recurse-submodules https://…/repo.git
divers utiles
Commandes pratiques pour les statistiques, la documentation et le versioning.
# Résumé des commits par auteur :
$ git shortlog -sn

# Décrire le commit courant :
$ git describe --tags
v1.0.0-3-ga3f9c21

# Compter les commits d'une branche :
$ git rev-list --count HEAD

# Voir le SHA d'un commit/branche :
$ git rev-parse HEAD
$ git rev-parse main
aliases recommandés
Alias à ajouter dans ~/.gitconfig pour booster la productivité au quotidien.
# Dans ~/.gitconfig section [alias] :

$ git config --global alias.st "status -s"
$ git config --global alias.co "checkout"
$ git config --global alias.br "branch -a"
$ git config --global alias.lg "log --oneline --graph --all --decorate"
$ git config --global alias.undo "reset --soft HEAD~1"
$ git config --global alias.wip "!git add -A && git commit -m 'WIP'"

# Usage :
$ git lg # beau graphe de l'historique
$ git undo # annuler le dernier commit sans perdre les fichiers
$ git wip # sauvegarder rapidement un WIP
